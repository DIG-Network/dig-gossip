# CNC-005: Address Manager Timestamp Update on Message Receipt

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 6.4 item 7](../../../resources/SPEC.md)

## Summary

On every message received from an outbound peer, the address manager MUST update the peer's timestamp. This ensures the address manager's "last seen" data stays fresh, which influences peer selection (more recently seen peers are preferred) and prevents the address manager from evicting active peers. Ported from Chia's node_discovery.py:139-154.

## Specification

### Timestamp Update on Message Receipt

```rust
use std::time::{SystemTime, UNIX_EPOCH};

/// Called by the per-connection message loop whenever a message
/// is received from an outbound peer.
/// Chia reference: node_discovery.py:139-154.
///
/// This keeps the address manager's timestamp fresh for active peers,
/// ensuring they are not evicted and are preferred in peer selection.
pub async fn update_peer_timestamp_on_message(
    address_manager: &Arc<RwLock<AddressManager>>,
    peer_info: &PeerInfo,
    is_outbound: bool,
) {
    // Only update timestamps for outbound peers.
    // Inbound peers are not tracked in the same way.
    if !is_outbound {
        return;
    }

    let now = SystemTime::now()
        .duration_since(UNIX_EPOCH)
        .unwrap()
        .as_secs();

    let mut am = address_manager.write().await;
    am.update_timestamp(&peer_info.get_key(), now);

    tracing::trace!(
        peer = %peer_info,
        timestamp = now,
        "updated address manager timestamp for outbound peer"
    );
}
```

### Integration with Per-Connection Loop

```rust
/// Per-connection message loop (runs as a spawned task per peer).
async fn run_peer_loop(
    peer_id: PeerId,
    peer: Peer,
    peer_rx: mpsc::Receiver<Message>,
    inbound_tx: mpsc::Sender<(PeerId, Message)>,
    outbound_rx: mpsc::Receiver<Message>,
    address_manager: Arc<RwLock<AddressManager>>,
    peer_info: PeerInfo,
    is_outbound: bool,
    shutdown: watch::Receiver<bool>,
) {
    loop {
        tokio::select! {
            // Receive message from peer
            Some(msg) = peer_rx.recv() => {
                // Update timestamp on every received message (CNC-005)
                update_peer_timestamp_on_message(
                    &address_manager, &peer_info, is_outbound
                ).await;

                // Forward to application layer
                if inbound_tx.send((peer_id, msg)).await.is_err() {
                    break; // Inbound channel closed
                }
            }

            // ... outbound sends, shutdown check ...
        }
    }
}
```

### AddressManager::update_timestamp

```rust
impl AddressManager {
    /// Update the last-seen timestamp for an address entry.
    /// Called on every message received from an outbound peer.
    /// Chia: address_manager.py connected() updates nTime.
    pub fn update_timestamp(&mut self, key: &str, timestamp: u64) {
        if let Some(entry) = self.lookup_mut(key) {
            entry.last_success = timestamp;
        }
    }
}
```

## Acceptance Criteria

- [ ] On every message received from an outbound peer, `address_manager.update_timestamp()` is called
- [ ] Timestamp is set to the current Unix time in seconds
- [ ] Only outbound peers trigger timestamp updates (inbound peers are skipped)
- [ ] The update is performed asynchronously (does not block message processing beyond the write lock)
- [ ] The address manager entry's `last_success` field is updated
- [ ] If the peer is not found in the address manager, the update is silently skipped
- [ ] Trace-level logging records each timestamp update

## Implementation Notes

- Primary files: `src/connection/peer_loop.rs`, `src/discovery/address_manager.rs`
- Dependencies: `tokio::sync::RwLock`, `std::time::SystemTime`
- Edge cases:
  - A burst of messages from a single peer will repeatedly acquire the write lock; this is acceptable because `update_timestamp` is fast (single HashMap lookup + field write)
  - If the address manager write lock is held by another operation (e.g., `add_to_new_table`), the update will briefly block; this is rare and acceptable
  - Clock jumps (NTP corrections) may cause timestamps to go backward; `last_success` should always be set to the provided value regardless
  - The peer may have been removed from the address manager between message receipt and timestamp update; the `lookup_mut` handles this by returning None
