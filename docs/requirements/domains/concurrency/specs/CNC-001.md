# CNC-001: GossipService and GossipHandle Thread Safety

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 3.3, Section 9.1](../../../resources/SPEC.md)

## Summary

GossipService and GossipHandle MUST be Send + Sync to allow safe sharing across async tasks and threads. GossipHandle MUST be Clone with an inner Arc, enabling cheap distribution of handles to multiple consumers. This is essential for the multi-task architecture where the handle is shared between the application layer, discovery task, relay task, and per-connection tasks.

## Specification

### GossipHandle Thread Safety

```rust
use std::sync::Arc;
use tokio::sync::{mpsc, RwLock};

/// Handle to a running gossip service. Cheaply cloneable (inner Arc).
/// SPEC Section 3.3: "Cheaply cloneable (inner Arc)"
#[derive(Clone)]
pub struct GossipHandle {
    inner: Arc<GossipHandleInner>,
}

struct GossipHandleInner {
    /// Shared connection map (CNC-003: RwLock for read-heavy access).
    connections: RwLock<HashMap<PeerId, PeerConnection>>,
    /// Per-connection outbound senders (CNC-003: lock-free mpsc).
    senders: RwLock<HashMap<PeerId, mpsc::Sender<Message>>>,
    /// Inbound message channel.
    inbound_rx: mpsc::Receiver<(PeerId, Message)>,
    /// Stats counters (CNC-003: AtomicU64).
    stats: Arc<GossipStatsAtomic>,
    /// Shutdown signal sender.
    shutdown_tx: tokio::sync::watch::Sender<bool>,
}

// GossipHandle is Send + Sync because:
// - Arc<T> is Send + Sync when T is Send + Sync
// - RwLock<T> is Send + Sync when T is Send
// - mpsc::Sender is Send + Sync
// - AtomicU64 is Send + Sync
```

### GossipService Thread Safety

```rust
/// The gossip service orchestrator.
/// Send + Sync to allow starting from any async context.
pub struct GossipService {
    config: GossipConfig,
    /// TLS connector (Send + Sync).
    tls_connector: TlsConnector,
    /// Address manager (behind Arc<RwLock<>>).
    address_manager: Arc<RwLock<AddressManager>>,
}
```

### Static Assertions

```rust
/// Compile-time verification of thread safety bounds.
#[cfg(test)]
mod assert_thread_safety {
    use super::*;

    fn assert_send<T: Send>() {}
    fn assert_sync<T: Sync>() {}
    fn assert_clone<T: Clone>() {}

    #[test]
    fn gossip_handle_is_send_sync_clone() {
        assert_send::<GossipHandle>();
        assert_sync::<GossipHandle>();
        assert_clone::<GossipHandle>();
    }

    #[test]

    }
}
```

## Acceptance Criteria

- [ ] `GossipService` implements `Send + Sync`
- [ ] `GossipHandle` implements `Send + Sync + Clone`
- [ ] `GossipHandle` uses an inner `Arc` for cheap cloning
- [ ] Cloning a `GossipHandle` does not clone the underlying state (shared via Arc)
- [ ] Static compile-time assertions verify `Send + Sync` bounds
- [ ] `GossipHandle` can be sent across tokio task boundaries
- [ ] Multiple clones of `GossipHandle` can be used concurrently from different tasks

## Implementation Notes

- Primary files: `src/service/gossip_service.rs`, `src/service/gossip_handle.rs`
- Dependencies: `std::sync::Arc`, `tokio::sync::RwLock`
- Edge cases:
  - All fields of `GossipHandleInner` must be Send + Sync; a single non-Send field breaks the entire struct
  - `mpsc::Receiver` is NOT Sync; it must be wrapped or accessed from a single task
  - The inbound receiver should remain in a single consumer task; only the sender side is shared
