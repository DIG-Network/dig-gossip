# PRI-002: PriorityOutbound Queue Structure

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 8.4](../../../resources/SPEC.md)

## Summary

Each peer connection maintains a `PriorityOutbound` structure containing three separate `VecDeque<Message>` queues -- one for each priority lane. When a message is enqueued for outbound delivery, it is routed to the appropriate queue based on its `MessagePriority` classification. This structure enables priority-ordered drain and per-lane backpressure without message reordering within a lane.

## Specification

### PriorityOutbound Struct

```rust
use std::collections::VecDeque;
use chia_protocol::Message;

/// Per-connection outbound message queue with priority lanes.
/// Each connection maintains its own PriorityOutbound instance.
pub struct PriorityOutbound {
    /// Consensus-critical messages. Drained first, always.
    critical: VecDeque<Message>,
    /// Normal protocol messages. Drained when critical is empty.
    normal: VecDeque<Message>,
    /// Bulk/background messages. Drained when both above are empty.
    bulk: VecDeque<Message>,
}

impl PriorityOutbound {
    /// Create a new empty priority outbound queue.
    pub fn new() -> Self {
        Self {
            critical: VecDeque::new(),
            normal: VecDeque::new(),
            bulk: VecDeque::new(),
        }
    }

    /// Enqueue a message into the appropriate priority lane.
    pub fn enqueue(&mut self, msg: Message, priority: MessagePriority) {
        match priority {
            MessagePriority::Critical => self.critical.push_back(msg),
            MessagePriority::Normal => self.normal.push_back(msg),
            MessagePriority::Bulk => self.bulk.push_back(msg),
        }
    }

    /// Total number of messages across all queues.
    pub fn total_len(&self) -> usize {
        self.critical.len() + self.normal.len() + self.bulk.len()
    }

    /// Number of messages in a specific priority lane.
    pub fn lane_len(&self, priority: MessagePriority) -> usize {
        match priority {
            MessagePriority::Critical => self.critical.len(),
            MessagePriority::Normal => self.normal.len(),
            MessagePriority::Bulk => self.bulk.len(),
        }
    }

    /// Returns true if all queues are empty.
    pub fn is_empty(&self) -> bool {
        self.critical.is_empty() && self.normal.is_empty() && self.bulk.is_empty()
    }
}
```

### Per-Connection Ownership

Each active peer connection MUST own exactly one `PriorityOutbound` instance. The connection manager creates a `PriorityOutbound::new()` when a connection is established and drops it when the connection is closed. Messages enqueued for a peer are always routed through that peer's `PriorityOutbound`.

### FIFO Within Lanes

Messages within the same priority lane MUST maintain FIFO order. `VecDeque` provides this guarantee: `push_back` for enqueue, `pop_front` for dequeue.

## Acceptance Criteria

- [ ] `PriorityOutbound` struct exists with fields `critical`, `normal`, `bulk` of type `VecDeque<Message>`
- [ ] `PriorityOutbound::new()` creates an instance with all three queues empty
- [ ] `enqueue()` with `MessagePriority::Critical` adds to the critical queue
- [ ] `enqueue()` with `MessagePriority::Normal` adds to the normal queue
- [ ] `enqueue()` with `MessagePriority::Bulk` adds to the bulk queue
- [ ] `total_len()` returns sum of all three queue lengths
- [ ] `lane_len(Critical)` returns critical queue length only
- [ ] `lane_len(Normal)` returns normal queue length only
- [ ] `lane_len(Bulk)` returns bulk queue length only
- [ ] `is_empty()` returns true only when all three queues are empty
- [ ] Messages within a lane are dequeued in FIFO order
- [ ] Each peer connection owns exactly one `PriorityOutbound` instance

## Implementation Notes

- Primary file: `src/gossip/priority.rs`
- Dependencies: `chia-protocol::Message`, `MessagePriority` (PRI-001)
- Edge cases:
  - `total_len()` must not overflow; with realistic queue sizes (hundreds, not billions) this is not a concern
  - The struct should not implement `Clone` to prevent accidental queue duplication between connections
  - Consider adding `with_capacity()` constructor for pre-allocating queue space if typical queue depths are known
