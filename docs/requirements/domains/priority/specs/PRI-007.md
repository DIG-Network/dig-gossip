# PRI-007: Bulk Drop Under Backpressure

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 8.5](../../../resources/SPEC.md)

## Summary

When the outbound queue depth reaches or exceeds the `bulk_drop_threshold` (default 50), all Bulk-priority messages are dropped silently and ERLAY set reconciliation is paused. This prevents low-priority background traffic from consuming outbound capacity that is needed for consensus-critical and normal protocol messages under load.

## Specification

### Bulk Drop Behavior

When `BackpressureLevel >= BulkDrop`, any message classified as `MessagePriority::Bulk` MUST be dropped before reaching the outbound queue:

```rust
impl PriorityOutbound {
    pub fn enqueue_with_backpressure(
        &mut self,
        msg: Message,
        priority: MessagePriority,
        bp_config: &BackpressureConfig,
        tx_dedup: &mut TxDedupFilter,
    ) -> EnqueueResult {
        let level = bp_config.level(self.total_len());

        // PRI-006: Transaction dedup (checked first, applies at >= TxDedup)
        if level >= BackpressureLevel::TxDedup {
            if let Some(tx_id) = extract_new_transaction_id(&msg) {
                if tx_dedup.should_suppress(&tx_id) {
                    return EnqueueResult::Suppressed;
                }
            }
        }

        // PRI-007: Bulk messages dropped at >= bulk_drop_threshold
        if level >= BackpressureLevel::BulkDrop && priority == MessagePriority::Bulk {
            return EnqueueResult::Dropped;
        }

        // ... PRI-008 checks ...

        self.enqueue(msg, priority);
        EnqueueResult::Enqueued
    }
}
```

### Dropped Message Types

The following message types are affected (all classified as `MessagePriority::Bulk` per PRI-001):

- `RequestBlocks`
- `RespondBlocks`
- `RequestPeers`
- `RespondPeers`
- `RequestMempoolTransactions`
- `RequestPeersIntroducer`
- `RespondPeersIntroducer`
- DIG `ValidatorAnnounce`

### ERLAY Reconciliation Pause

When `BackpressureLevel >= BulkDrop`, the ERLAY reconciliation scheduler MUST be paused for the affected connection:

```rust
/// Signal to the ERLAY reconciliation loop.
pub struct ErlayBackpressureSignal {
    paused: AtomicBool,
}

impl ErlayBackpressureSignal {
    /// Check if ERLAY reconciliation should be paused.
    pub fn is_paused(&self) -> bool {
        self.paused.load(Ordering::Relaxed)
    }

    /// Set pause state based on current backpressure level.
    pub fn update(&self, level: BackpressureLevel) {
        self.paused.store(
            level >= BackpressureLevel::BulkDrop,
            Ordering::Relaxed,
        );
    }
}
```

When paused:
- No new minisketch reconciliation rounds are initiated for this connection.
- In-flight reconciliation responses are still processed (they arrive as inbound messages, not outbound).
- Reconciliation resumes automatically when queue depth drops below `bulk_drop_threshold`.

### Silent Drop

Dropped bulk messages MUST NOT generate errors, warnings, or retries. They are silently discarded. The caller receives `EnqueueResult::Dropped` but no log is emitted at warn level or above. A `tracing::debug!` event MAY be emitted for diagnostics.

### Critical and Normal Unaffected

At `BackpressureLevel::BulkDrop`, Critical and Normal messages are enqueued normally (subject to PRI-006 tx dedup for Normal). Only Bulk-priority messages are dropped.

## Acceptance Criteria

- [ ] At queue depth >= 50 (default), Bulk messages return `EnqueueResult::Dropped`
- [ ] At queue depth < 50, Bulk messages are enqueued normally
- [ ] Critical messages are never dropped regardless of queue depth
- [ ] Normal messages are not dropped at BulkDrop level (only Bulk is)
- [ ] All Bulk message types (RequestBlocks, RespondBlocks, RequestPeers, RespondPeers, RequestMempoolTransactions, introducer messages, ValidatorAnnounce) are dropped
- [ ] ERLAY reconciliation is paused when `BackpressureLevel >= BulkDrop`
- [ ] ERLAY reconciliation resumes when queue depth drops below bulk_drop_threshold
- [ ] In-flight ERLAY responses are still processed (inbound, not outbound)
- [ ] Dropped messages do not generate errors or warnings
- [ ] `EnqueueResult::Dropped` is returned for dropped Bulk messages
- [ ] Tx dedup (PRI-006) is still active at BulkDrop level (cumulative)

## Implementation Notes

- Primary files: `src/gossip/backpressure.rs`, `src/gossip/priority.rs`
- Dependencies: `BackpressureConfig` (PRI-005), `PriorityOutbound` (PRI-002), `MessagePriority` (PRI-001)
- Edge cases:
  - If a `RequestPeers` is dropped, the peer will not receive a `RespondPeers`. This is acceptable because peer exchange is periodic and will be retried.
  - If `RespondBlocks` is dropped for a historical sync request, the requesting peer will timeout and retry. This is acceptable because historical sync is not time-critical.
  - Dropping `RequestMempoolTransactions` delays mempool sync but does not affect consensus.
  - The `ErlayBackpressureSignal` uses `AtomicBool` for lock-free signaling between the enqueue path and the ERLAY reconciliation task.
  - When queue depth oscillates around the threshold, ERLAY may be rapidly paused/unpaused. This is acceptable; reconciliation rounds are idempotent.
