# PRI-006: Transaction Deduplication Under Backpressure

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 8.5](../../../resources/SPEC.md)

## Summary

When the outbound queue depth reaches or exceeds the `tx_dedup_threshold` (default 25), duplicate `NewTransaction` announcements are suppressed. Only the first announcement for a given `tx_id` passes through; subsequent announcements for the same transaction are silently dropped. This reduces outbound queue growth during transaction bursts without losing any unique transaction data.

## Specification

### Deduplication Mechanism

The backpressure monitor MUST maintain a per-connection set of recently announced transaction IDs. When `BackpressureLevel >= TxDedup`:

```rust
use std::collections::HashSet;
use chia_protocol::Bytes32;

/// Per-connection transaction deduplication state.
pub struct TxDedupFilter {
    /// Set of tx_ids that have already been enqueued for this connection.
    /// Populated only when backpressure level >= TxDedup.
    seen_tx_ids: HashSet<Bytes32>,
}

impl TxDedupFilter {
    pub fn new() -> Self {
        Self {
            seen_tx_ids: HashSet::new(),
        }
    }

    /// Check if a NewTransaction should be suppressed.
    /// Returns true if the tx_id has already been seen (suppress it).
    /// Returns false if this is the first time (allow it).
    pub fn should_suppress(&mut self, tx_id: &Bytes32) -> bool {
        !self.seen_tx_ids.insert(*tx_id)
    }

    /// Clear the dedup filter. Called when backpressure drops below TxDedup.
    pub fn clear(&mut self) {
        self.seen_tx_ids.clear();
    }

    /// Number of tracked transaction IDs.
    pub fn len(&self) -> usize {
        self.seen_tx_ids.len()
    }
}
```

### Enqueue Decision Flow

When a `NewTransaction` message is enqueued:

```rust
impl PriorityOutbound {
    /// Enqueue a message with backpressure awareness.
    pub fn enqueue_with_backpressure(
        &mut self,
        msg: Message,
        priority: MessagePriority,
        bp_config: &BackpressureConfig,
        tx_dedup: &mut TxDedupFilter,
    ) -> EnqueueResult {
        let level = bp_config.level(self.total_len());

        // PRI-006: Transaction deduplication at >= tx_dedup_threshold
        if level >= BackpressureLevel::TxDedup {
            if let Some(tx_id) = extract_new_transaction_id(&msg) {
                if tx_dedup.should_suppress(&tx_id) {
                    return EnqueueResult::Suppressed;
                }
            }
        }

        // ... other backpressure checks (PRI-007, PRI-008) ...

        self.enqueue(msg, priority);
        EnqueueResult::Enqueued
    }
}

/// Result of attempting to enqueue a message.
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum EnqueueResult {
    /// Message was enqueued normally.
    Enqueued,
    /// Message was suppressed by tx deduplication.
    Suppressed,
    /// Message was dropped by bulk backpressure.
    Dropped,
    /// Message was delayed (will be batched).
    Delayed,
}
```

### Scope and Lifetime

1. **Per-connection:** Each connection maintains its own `TxDedupFilter`. A transaction announced to peer A is independent of peer B.
2. **Active only under backpressure:** The filter is only consulted when `BackpressureLevel >= TxDedup`. Below the threshold, all `NewTransaction` messages pass through.
3. **Cleared on recovery:** When queue depth drops below `tx_dedup_threshold`, the `TxDedupFilter` SHOULD be cleared to avoid permanent suppression of re-announced transactions.

### What Is Suppressed

Only `NewTransaction` messages are subject to dedup. Other Normal-priority messages (`RespondTransaction`, `NewUnfinishedBlock`, request messages) are never suppressed by this mechanism.

## Acceptance Criteria

- [ ] `TxDedupFilter` struct exists with `HashSet<Bytes32>` for tracking seen tx_ids
- [ ] `should_suppress()` returns false on first occurrence of a tx_id
- [ ] `should_suppress()` returns true on second occurrence of the same tx_id
- [ ] `clear()` resets the filter, allowing previously-seen tx_ids to pass again
- [ ] Dedup is only active when `BackpressureLevel >= TxDedup` (queue depth >= 25)
- [ ] Dedup is not active when queue depth < 25 (all NewTransaction messages pass)
- [ ] Only `NewTransaction` messages are subject to dedup, not other Normal messages
- [ ] Dedup is per-connection (separate filter instances per peer)
- [ ] `EnqueueResult::Suppressed` is returned for deduplicated messages
- [ ] Filter is cleared when backpressure drops below TxDedup level
- [ ] Non-NewTransaction messages at Normal priority are unaffected by dedup

## Implementation Notes

- Primary files: `src/gossip/backpressure.rs`, `src/gossip/priority.rs`
- Dependencies: `BackpressureConfig` (PRI-005), `PriorityOutbound` (PRI-002), `chia-protocol::Bytes32`
- Edge cases:
  - The `extract_new_transaction_id()` helper must parse the `Message` to extract the tx_id from `NewTransaction` payload. This requires deserializing the message data field.
  - If the `HashSet` grows very large during prolonged backpressure, consider switching to a bounded LRU set. For typical transaction bursts (thousands of txs), `HashSet` is sufficient.
  - The dedup filter should not prevent the same transaction from being sent to different peers (each peer has its own filter).
  - Transaction IDs that were suppressed are still "known" -- the peer will receive them when backpressure clears via normal gossip or ERLAY reconciliation.
