# PRI-003: Drain Order

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 8.4](../../../resources/SPEC.md)

## Summary

The drain algorithm for `PriorityOutbound` enforces strict priority ordering: all critical messages are sent before any normal messages, and all normal messages are sent before any bulk message. After sending one bulk message, the drain loop rechecks the critical queue before continuing. This ensures consensus-critical traffic experiences minimal latency while still making progress on lower-priority work.

## Specification

### Drain Algorithm

The `drain_next()` method on `PriorityOutbound` MUST implement the following logic:

```rust
impl PriorityOutbound {
    /// Drain the next message in priority order.
    ///
    /// Order: exhaust critical -> exhaust normal -> one bulk -> recheck critical.
    /// Returns None if all queues are empty.
    pub fn drain_next(&mut self) -> Option<Message> {
        // 1. Always try critical first
        if let Some(msg) = self.critical.pop_front() {
            return Some(msg);
        }

        // 2. Then normal
        if let Some(msg) = self.normal.pop_front() {
            return Some(msg);
        }

        // 3. Then one bulk message (caller will loop and recheck critical)
        self.bulk.pop_front()
    }
}
```

### Drain Loop Pattern

The caller MUST invoke `drain_next()` in a loop. Because `drain_next()` always checks critical first, the pattern naturally implements "exhaust critical, then normal, then one bulk, then check critical again":

```rust
// Caller drain loop (in the connection send task)
loop {
    match outbound.drain_next() {
        Some(msg) => {
            peer.send(msg).await?;
        }
        None => {
            // All queues empty; wait for new messages
            break;
        }
    }
}
```

### Priority Guarantees

1. **Critical preemption:** If a critical message arrives while normal or bulk messages are being drained, the next `drain_next()` call will return the critical message.
2. **Normal before bulk:** All queued normal messages are drained before any bulk message is considered.
3. **Bulk single-step:** Only one bulk message is drained per iteration before rechecking critical. This is inherent in the algorithm since `drain_next()` returns after one message and the caller loops.
4. **No reordering within lanes:** Messages within the same lane are always drained in FIFO order.

## Acceptance Criteria

- [ ] `drain_next()` returns critical messages before normal messages
- [ ] `drain_next()` returns normal messages before bulk messages
- [ ] `drain_next()` returns `None` when all queues are empty
- [ ] When critical has messages, `drain_next()` always returns from critical regardless of normal/bulk state
- [ ] After draining all critical and normal, `drain_next()` returns one bulk message
- [ ] Subsequent call to `drain_next()` after one bulk checks critical again
- [ ] Interleaving: enqueueing a critical message after draining some normal messages causes next `drain_next()` to return the critical message
- [ ] FIFO order preserved within each lane during drain

## Implementation Notes

- Primary file: `src/gossip/priority.rs`
- Dependencies: `PriorityOutbound` (PRI-002), `chia-protocol::Message`
- Edge cases:
  - If critical messages arrive faster than they can be sent, normal and bulk queues will grow unbounded -- this is by design and handled by backpressure (PRI-005 through PRI-008)
  - The drain loop runs in the connection's send task (a tokio task per connection)
  - `drain_next()` is synchronous; the async send happens in the caller
