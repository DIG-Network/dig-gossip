# PRI-001: MessagePriority Enum

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 8.4](../../../resources/SPEC.md)

## Summary

Each `ProtocolMessageType` (and DIG extension message type) is assigned to exactly one of three priority lanes. The `MessagePriority` enum encodes these lanes with explicit discriminants that also define their ordering: lower discriminant values represent higher priority. This classification drives all downstream queue routing, drain ordering, and backpressure decisions.

## Specification

### MessagePriority Enum

```rust
/// Message priority levels.
/// Lower discriminant = higher priority.
#[derive(Debug, Clone, Copy, PartialEq, Eq, PartialOrd, Ord, Hash)]
#[repr(u8)]
pub enum MessagePriority {
    /// Consensus-critical: NewPeak, blocks, attestations, checkpoints.
    /// Always sent first. Never dropped by backpressure.
    Critical = 0,
    /// Normal protocol: transactions, unfinished blocks, requests.
    /// Sent after critical. May be delayed under backpressure.
    Normal = 1,
    /// Bulk/background: mempool sync, peer exchange, historical block requests.
    /// Sent last. Dropped first under backpressure.
    Bulk = 2,
}
```

### Priority Assignment

The `MessagePriority::from_message_type()` function MUST classify every message type as follows:

| Priority | Message Types |
|----------|--------------|
| **Critical** | `NewPeak`, `RespondBlock`, `RespondUnfinishedBlock`, DIG `NewAttestation`, DIG `NewCheckpointProposal`, DIG `NewCheckpointSignature` |
| **Normal** | `NewTransaction`, `RespondTransaction`, `NewUnfinishedBlock`, `RequestBlock`, `RequestTransaction`, `RequestUnfinishedBlock`, DIG `RequestStatus`, DIG `RespondStatus` |
| **Bulk** | `RequestBlocks`, `RespondBlocks`, `RequestPeers`, `RespondPeers`, `RequestMempoolTransactions`, `RequestPeersIntroducer`, `RespondPeersIntroducer`, DIG `ValidatorAnnounce` |

### Classification Function

```rust
impl MessagePriority {
    /// Classify a protocol message type into its priority lane.
    pub fn from_message_type(msg_type: ProtocolMessageTypes) -> Self {
        match msg_type {
            // Critical: consensus-critical messages
            ProtocolMessageTypes::NewPeak
            | ProtocolMessageTypes::RespondBlock
            | ProtocolMessageTypes::RespondUnfinishedBlock => MessagePriority::Critical,

            // Normal: transaction and request messages
            ProtocolMessageTypes::NewTransaction
            | ProtocolMessageTypes::RespondTransaction
            | ProtocolMessageTypes::NewUnfinishedBlock
            | ProtocolMessageTypes::RequestBlock
            | ProtocolMessageTypes::RequestTransaction
            | ProtocolMessageTypes::RequestUnfinishedBlock => MessagePriority::Normal,

            // Bulk: background and batch messages
            ProtocolMessageTypes::RequestBlocks
            | ProtocolMessageTypes::RespondBlocks
            | ProtocolMessageTypes::RequestPeers
            | ProtocolMessageTypes::RespondPeers
            | ProtocolMessageTypes::RequestMempoolTransactions
            | ProtocolMessageTypes::RequestPeersIntroducer
            | ProtocolMessageTypes::RespondPeersIntroducer => MessagePriority::Bulk,

            // Default: unclassified message types fall to Normal
            _ => MessagePriority::Normal,
        }
    }

    /// Classify a DIG extension message type into its priority lane.
    pub fn from_dig_message_type(msg_type: DigMessageType) -> Self {
        match msg_type {
            DigMessageType::NewAttestation
            | DigMessageType::NewCheckpointProposal
            | DigMessageType::NewCheckpointSignature => MessagePriority::Critical,

            DigMessageType::RequestStatus
            | DigMessageType::RespondStatus => MessagePriority::Normal,

            DigMessageType::ValidatorAnnounce => MessagePriority::Bulk,

            _ => MessagePriority::Normal,
        }
    }
}
```

### Ordering Guarantee

Because `MessagePriority` derives `PartialOrd` and `Ord`, and uses `#[repr(u8)]` with `Critical = 0 < Normal = 1 < Bulk = 2`, the natural ordering places higher-priority variants first. This means `Critical < Normal < Bulk` in Rust's ordering, which is correct for min-heap or sorted drain patterns.

## Implementation Notes

- Primary file: `src/gossip/priority.rs`
- Dependencies: `chia-protocol::ProtocolMessageTypes`, `types::dig_messages::DigMessageType`
- Edge cases:
  - Unrecognized `ProtocolMessageTypes` variants (future Chia additions) default to `Normal` to avoid accidentally dropping consensus messages
  - DIG extension types use a separate classification method since they have their own enum
  - The `#[repr(u8)]` ensures the discriminant values are stable across compilations and can be used for serialization
