# PRI-004: Starvation Prevention

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 8.4](../../../resources/SPEC.md)

## Summary

Under sustained high-priority load, the strict drain order (PRI-003) could starve bulk messages indefinitely. To prevent this, the drain algorithm guarantees that at least 1 bulk message is sent for every `PRIORITY_STARVATION_RATIO` (10) critical/normal messages drained. This ensures bulk work (peer exchange, mempool sync, historical block requests) makes forward progress even during peak consensus activity.

## Specification

### Starvation Ratio Constant

```rust
/// Maximum number of critical + normal messages drained before
/// one bulk message must be sent (if any are queued).
pub const PRIORITY_STARVATION_RATIO: usize = 10;
```

### Enhanced Drain with Starvation Prevention

The `PriorityOutbound` MUST track a counter of consecutive critical/normal messages drained since the last bulk message. When this counter reaches `PRIORITY_STARVATION_RATIO` and the bulk queue is non-empty, the next `drain_next()` MUST return a bulk message.

```rust
pub struct PriorityOutbound {
    critical: VecDeque<Message>,
    normal: VecDeque<Message>,
    bulk: VecDeque<Message>,
    /// Counter: critical + normal messages drained since last bulk.
    high_priority_count: usize,
}

impl PriorityOutbound {
    /// Drain the next message with starvation prevention.
    pub fn drain_next(&mut self) -> Option<Message> {
        // Starvation check: if we've sent PRIORITY_STARVATION_RATIO
        // critical/normal messages, force one bulk message through.
        if self.high_priority_count >= PRIORITY_STARVATION_RATIO {
            if let Some(msg) = self.bulk.pop_front() {
                self.high_priority_count = 0;
                return Some(msg);
            }
            // No bulk messages queued; reset counter and continue normally.
            self.high_priority_count = 0;
        }

        // 1. Critical first
        if let Some(msg) = self.critical.pop_front() {
            self.high_priority_count += 1;
            return Some(msg);
        }

        // 2. Normal second
        if let Some(msg) = self.normal.pop_front() {
            self.high_priority_count += 1;
            return Some(msg);
        }

        // 3. Bulk last (resets counter)
        if let Some(msg) = self.bulk.pop_front() {
            self.high_priority_count = 0;
            return Some(msg);
        }

        None
    }
}
```

### Behavioral Rules

1. **Counter increments** for every critical or normal message drained.
2. **Counter resets to 0** when a bulk message is drained (whether forced or natural).
3. **Forced bulk:** When `high_priority_count >= PRIORITY_STARVATION_RATIO` and `bulk` is non-empty, the bulk message takes precedence over critical/normal.
4. **No-op when bulk is empty:** If the counter reaches the threshold but bulk is empty, the counter resets and draining proceeds normally. This avoids blocking critical/normal messages when there is no bulk work to do.
5. **Ratio applies per connection:** Each `PriorityOutbound` instance tracks its own counter independently.

## Acceptance Criteria

- [ ] `PRIORITY_STARVATION_RATIO` constant is defined with value 10
- [ ] `PriorityOutbound` tracks `high_priority_count` field
- [ ] Counter increments on each critical message drained
- [ ] Counter increments on each normal message drained
- [ ] Counter resets to 0 on each bulk message drained
- [ ] After 10 consecutive critical/normal drains, next drain returns a bulk message if available
- [ ] After 10 consecutive critical/normal drains with empty bulk queue, counter resets and critical/normal continue
- [ ] After forced bulk drain, counter is 0 and next drain returns critical if available
- [ ] Starvation prevention is per-connection (each PriorityOutbound has its own counter)
- [ ] With mixed traffic (e.g., 15 critical, 5 bulk), bulk messages are interleaved correctly

## Implementation Notes

- Primary file: `src/gossip/priority.rs`
- Dependencies: `PriorityOutbound` (PRI-002), `PRIORITY_STARVATION_RATIO` in `src/constants.rs`
- Edge cases:
  - If only critical messages are present with no bulk, the counter reaches the threshold, resets, and continues draining critical. No deadlock or stall occurs.
  - If bulk messages arrive between starvation checks, they will be picked up at the next threshold crossing or when critical/normal are exhausted.
  - The ratio of 10:1 means under maximum critical load, bulk gets ~9% of outbound bandwidth.
