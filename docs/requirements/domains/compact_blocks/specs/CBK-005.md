# CBK-005: Fallback to Full Block

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 8.2](../../../resources/SPEC.md)

## Summary

When compact block reconstruction results in more than `COMPACT_BLOCK_MAX_MISSING_TXS` (5) missing transactions, the receiver abandons the compact block approach and falls back to requesting the full block via the standard `RequestBlock`/`RespondBlock` protocol from `chia-protocol`. This threshold prevents excessive round-trips for missing transactions when the receiver's mempool has low overlap with the block contents, which would negate the bandwidth savings of compact block relay.

## Specification

### Fallback Constant

```rust
/// Max missing transactions before falling back to full block request.
/// If compact block reconstruction leaves more than this many transactions
/// unmatched, request the full block instead of individual missing txs.
pub const COMPACT_BLOCK_MAX_MISSING_TXS: usize = 5;
```

### Fallback Decision Logic

```rust
use chia_protocol::{RequestBlock, RespondBlock, Bytes32};

/// Determines the action to take after compact block reconstruction.
pub enum PostReconstructionAction {
    /// Block fully reconstructed -- no further action needed.
    Complete,
    /// Few missing transactions -- request them individually (CBK-004).
    RequestMissing {
        block_hash: Bytes32,
        missing_indices: Vec<u16>,
    },
    /// Too many missing transactions -- fall back to full block request.
    FallbackToFullBlock {
        block_hash: Bytes32,
    },
}

impl ReconstructionResult {
    /// Decide the next action based on how many transactions are missing.
    pub fn next_action(&self, block_hash: Bytes32) -> PostReconstructionAction {
        if self.missing_indices.is_empty() {
            PostReconstructionAction::Complete
        } else if self.missing_indices.len() <= COMPACT_BLOCK_MAX_MISSING_TXS {
            PostReconstructionAction::RequestMissing {
                block_hash,
                missing_indices: self.missing_indices.clone(),
            }
        } else {
            PostReconstructionAction::FallbackToFullBlock {
                block_hash,
            }
        }
    }
}
```

### Fallback Protocol Flow

```
Receiver (after reconstruction with >5 missing transactions):
  1. Determine missing_indices.len() > COMPACT_BLOCK_MAX_MISSING_TXS
  2. Discard the partial ReconstructionResult
  3. Send RequestBlock { header_hash } to the compact block sender
     (using chia-protocol's RequestBlock type)
  4. Receive RespondBlock { block } containing the full block
  5. Hand off to caller for block validation

Note: RequestBlock/RespondBlock are existing chia-protocol types,
not DIG-specific. This is the standard full-block request path.
```

### Threshold Justification

- At 5 or fewer missing transactions, requesting individual transactions is cheaper than retransmitting the full block (which may be 2MB+). Each `RequestBlockTransactions` response carries only the missing transaction data.
- At 6+ missing transactions, the round-trip overhead and partial retransmission approach diminishing returns. A single full block request is simpler and may be faster due to reduced round-trips.
- The threshold of 5 balances bandwidth savings against protocol complexity.

## Acceptance Criteria

- [ ] `COMPACT_BLOCK_MAX_MISSING_TXS` constant is defined as `5`
- [ ] When `missing_indices.len()` is 0, action is `Complete`
- [ ] When `missing_indices.len()` is 1-5, action is `RequestMissing` with the correct indices
- [ ] When `missing_indices.len()` is 6 or more, action is `FallbackToFullBlock`
- [ ] Fallback uses `chia-protocol::RequestBlock` (not a DIG-specific message type)
- [ ] Fallback uses `chia-protocol::RespondBlock` for the response
- [ ] Partial `ReconstructionResult` is discarded when falling back to full block
- [ ] The full block received via fallback is handed to the caller for validation

## Implementation Notes

- Primary file: `src/gossip/compact_block.rs` and `src/constants.rs`
- Dependencies: `chia-protocol` for `RequestBlock`/`RespondBlock`, `ReconstructionResult` from CBK-003
- Edge cases:
  - Exactly 5 missing transactions should use `RequestMissing` (the threshold is ">5", i.e., 6+ triggers fallback)
  - Exactly 6 missing transactions should trigger `FallbackToFullBlock`
  - If the full block request also fails (peer disconnected, timeout), the receiver should attempt to request from another peer that has the block
  - The compact block sender may have been an eager push peer in the Plumtree gossip tree -- the full block fallback should also be sent to this peer first
  - Statistics should track how often fallback occurs to monitor mempool synchronization quality across the network
