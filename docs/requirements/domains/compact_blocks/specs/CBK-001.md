## Summary

The `CompactBlock` struct is the core data type for compact block relay, inspired by Bitcoin BIP 152. Instead of transmitting full blocks (up to 2MB+), a `CompactBlock` contains the full block header, short transaction IDs (6 bytes each), prefilled transactions the sender predicts the receiver lacks, and the SipHash key used for short ID computation. Prefilled transactions MUST include the coinbase transaction and any transactions added within the last 2 seconds before block creation.

## Specification

### CompactBlock Struct

```rust
use serde::{Serialize, Deserialize};

/// Compact block representation for efficient relay.
/// Inspired by Bitcoin BIP 152.
pub struct CompactBlock {
    /// Full block header.
    pub header: BlockHeader,
    /// Short transaction IDs (6 bytes each, truncated SipHash).
    /// Receiver matches against mempool to reconstruct full block.
    pub short_tx_ids: Vec<ShortTxId>,
    /// Prefilled transactions the sender predicts the receiver
    /// doesn't have (e.g., coinbase, very recent transactions).
    pub prefilled_txs: Vec<PrefilledTransaction>,
    /// SipHash key derived from block header hash (for short ID computation).
    pub sip_hash_key: [u8; 16],
}

/// 6-byte truncated SipHash of transaction ID.
pub type ShortTxId = [u8; 6];

/// A transaction included in the compact block directly.
pub struct PrefilledTransaction {
    /// Index in the block's transaction list.
    pub index: u16,
    /// Full serialized transaction.
    pub tx: Vec<u8>,
}
```

### Prefilled Transaction Selection

When constructing a `CompactBlock` from a full block, the sender MUST include as prefilled transactions:

1. The **coinbase transaction** (always at index 0) -- receivers never have this in their mempool.
2. Any transactions that were **added to the block within the last 2 seconds** before block creation -- receivers are unlikely to have propagated these to their mempool yet.

```rust
impl CompactBlock {
    /// Construct a CompactBlock from a full block.
    /// `recent_threshold` is the Unix timestamp 2 seconds before block creation.
    pub fn from_full_block(
        header: BlockHeader,
        transactions: &[(u16, Vec<u8>, u64)], // (index, serialized_tx, added_timestamp)
        sip_hash_key: [u8; 16],
        recent_threshold: u64,
    ) -> Self {
        let mut prefilled_txs = Vec::new();
        let mut short_tx_ids = Vec::new();

        for (index, tx, added_at) in transactions {
            if *index == 0 || *added_at >= recent_threshold {
                // Coinbase or recent: prefill
                prefilled_txs.push(PrefilledTransaction {
                    index: *index,
                    tx: tx.clone(),
                });
            } else {
                // Compute short ID for mempool matching
                let short_id = compute_short_tx_id(&sip_hash_key, tx);
                short_tx_ids.push(short_id);
            }
        }

        CompactBlock {
            header,
            short_tx_ids,
            prefilled_txs,
            sip_hash_key,
        }
    }
}
```

## Acceptance Criteria

- [ ] `CompactBlock` struct has field `header` of type `BlockHeader`
- [ ] `CompactBlock` struct has field `short_tx_ids` of type `Vec<ShortTxId>`
- [ ] `CompactBlock` struct has field `prefilled_txs` of type `Vec<PrefilledTransaction>`
- [ ] `CompactBlock` struct has field `sip_hash_key` of type `[u8; 16]`
- [ ] `ShortTxId` is defined as `[u8; 6]`
- [ ] `PrefilledTransaction` struct has field `index` of type `u16`
- [ ] `PrefilledTransaction` struct has field `tx` of type `Vec<u8>`
- [ ] Coinbase transaction (index 0) is always included in `prefilled_txs`
- [ ] Transactions added within the last 2 seconds are included in `prefilled_txs`
- [ ] Non-prefilled transactions are represented as `ShortTxId` entries in `short_tx_ids`
- [ ] `CompactBlock` is constructible from a full block

## Implementation Notes

- Primary file: `src/types/compact_block.rs` (or `src/gossip/compact_block.rs` per SPEC Section 10.1)
- Dependencies: `BlockHeader` from the `dig-block` crate (the DIG L2 block model crate, NOT a Chia type), `siphasher` crate for short ID computation
- **BlockHeader origin:** `BlockHeader` is defined in the `dig-block` crate and imported as a dependency. It is NOT a Chia protocol type and is NOT defined within `dig-gossip`.
- Edge cases:
  - A block with only a coinbase transaction produces a `CompactBlock` with zero `short_tx_ids` and one `prefilled_txs` entry
  - The 2-second threshold for recent transactions is measured from the block creation timestamp, not wall clock time
  - `prefilled_txs` may overlap with `short_tx_ids` indices -- the implementation must ensure no transaction appears in both lists
  - Empty blocks (header only, no transactions) are valid and produce a `CompactBlock` with empty `short_tx_ids` and empty `prefilled_txs`
