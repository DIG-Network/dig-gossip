# CBK-006: Deterministic SipHash Key Derivation

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 8.2](../../../resources/SPEC.md)

## Summary

The SipHash key used for compact block short transaction ID computation MUST be derived deterministically from the block header hash. This ensures that the same block always produces the same `CompactBlock` representation, enabling consistent short ID computation between sender and receiver. Deterministic derivation from the header hash also prevents precomputed collision attacks: an attacker cannot predict the SipHash key before the block header is known, making it infeasible to craft transactions with colliding short IDs.

## Specification

### Key Derivation

```rust
/// Derive the 16-byte SipHash key deterministically from a block header hash.
///
/// The key is the first 16 bytes of the block header hash (which is
/// typically a 32-byte SHA256 or similar digest). Since the header hash
/// is unique per block and unpredictable before the block is created,
/// this provides a secure and deterministic key for SipHash computation.
pub fn derive_sip_hash_key(header_hash: &Bytes32) -> [u8; 16] {
    let mut key = [0u8; 16];
    key.copy_from_slice(&header_hash.as_ref()[0..16]);
    key
}
```

### Determinism Guarantee

```rust
impl CompactBlock {
    /// Create a CompactBlock from a full block.
    /// The sip_hash_key is derived deterministically from the header hash,
    /// ensuring the same block always produces the same CompactBlock.
    pub fn from_block(
        header: BlockHeader,
        transactions: &[(u16, Vec<u8>, u64)],
        recent_threshold: u64,
    ) -> Self {
        let header_hash = header.hash();
        let sip_hash_key = derive_sip_hash_key(&header_hash);

        // Construct CompactBlock using the derived key
        // (see CBK-001 for full construction logic)
        CompactBlock::from_full_block(header, transactions, sip_hash_key, recent_threshold)
    }
}
```

### Security Properties

1. **Unpredictability:** The SipHash key depends on the block header hash, which includes the previous block hash, timestamp, and other block-specific data. An attacker cannot predict the key before the block is produced.

2. **Collision resistance:** Since the key changes per block, an attacker cannot precompute transaction pairs that collide across multiple blocks. Each block's short IDs are independent.

3. **Determinism:** Given the same block header, the same SipHash key is derived, and therefore the same short transaction IDs are computed. Sender and receiver compute identical short IDs for the same transactions.

4. **Simplicity:** Using the first 16 bytes of the header hash avoids the need for a separate key derivation function (KDF). The header hash already provides sufficient entropy and uniformity.

### Verification of Determinism

```rust
/// Verify that two CompactBlocks created from the same full block
/// are byte-identical.
fn verify_determinism(block: &FullBlock) -> bool {
    let compact1 = CompactBlock::from_block(
        block.header.clone(),
        &block.transactions,
        block.recent_threshold,
    );
    let compact2 = CompactBlock::from_block(
        block.header.clone(),
        &block.transactions,
        block.recent_threshold,
    );

    compact1.sip_hash_key == compact2.sip_hash_key
        && compact1.short_tx_ids == compact2.short_tx_ids
        && compact1.prefilled_txs.len() == compact2.prefilled_txs.len()
}
```

## Acceptance Criteria

- [ ] `derive_sip_hash_key` accepts a `Bytes32` header hash and returns `[u8; 16]`
- [ ] The returned key is the first 16 bytes of the header hash
- [ ] Same header hash always produces the same 16-byte key
- [ ] Different header hashes produce different keys (with overwhelming probability)
- [ ] `CompactBlock::from_block` uses `derive_sip_hash_key` internally
- [ ] Same block (same header + same transactions) always produces the same CompactBlock
- [ ] The `sip_hash_key` field in the CompactBlock matches the derived key
- [ ] Short transaction IDs computed with the derived key are consistent between sender and receiver

## Implementation Notes

- Primary file: `src/gossip/compact_block.rs` or `src/types/compact_block.rs`
- Dependencies: `Bytes32` from `chia-protocol`
- Edge cases:
  - The header hash must be at least 16 bytes. `Bytes32` is 32 bytes, so this is always satisfied.
  - If the block header hash algorithm changes in a future protocol version, the key derivation remains valid as long as the hash output is >= 16 bytes.
  - The key derivation function is intentionally simple (first 16 bytes). A more complex KDF (e.g., HKDF) is unnecessary because the header hash already has the properties we need: uniform distribution, unpredictability, and uniqueness.
  - Nodes must agree on the key derivation method. Since it is deterministic and specified in the protocol, there is no negotiation needed.
  - The `sip_hash_key` is included in the `CompactBlock` struct for convenience and explicitness, even though the receiver could re-derive it from the header hash. This avoids requiring the receiver to know the key derivation algorithm independently.
