# CBK-002: ShortTxId Computation

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 8.2](../../../resources/SPEC.md)

## Summary

Short transaction IDs are 6-byte truncated SipHash values computed from the SipHash key (derived from the block header hash) and the full transaction ID. This compact representation allows receivers to match transactions against their mempool without transmitting full 32-byte transaction IDs. The `siphasher` crate provides the SipHash implementation. At 6 bytes, the collision probability is approximately 1 in 2^48 per transaction pair, which is acceptable for blocks containing up to thousands of transactions.

## Specification

### ShortTxId Type

```rust
/// 6-byte truncated SipHash of transaction ID.
/// SHORT_TX_ID_BYTES = 6
pub type ShortTxId = [u8; 6];

/// Short TX ID length in bytes.
pub const SHORT_TX_ID_BYTES: usize = 6;
```

### Short ID Computation

```rust
use siphasher::sip::SipHasher;
use std::hash::Hasher;

/// Compute a 6-byte short transaction ID using SipHash.
///
/// `sip_hash_key` is a 16-byte key derived from the block header hash.
/// `tx_id` is the full transaction identifier (typically 32 bytes).
///
/// Returns the first 6 bytes of the SipHash-2-4 output.
pub fn compute_short_tx_id(sip_hash_key: &[u8; 16], tx_id: &[u8]) -> ShortTxId {
    // Split the 16-byte key into two u64 values for SipHasher
    let k0 = u64::from_le_bytes(sip_hash_key[0..8].try_into().unwrap());
    let k1 = u64::from_le_bytes(sip_hash_key[8..16].try_into().unwrap());

    let mut hasher = SipHasher::new_with_keys(k0, k1);
    hasher.write(tx_id);
    let hash = hasher.finish();

    // Truncate to first 6 bytes (little-endian)
    let hash_bytes = hash.to_le_bytes();
    let mut short_id = [0u8; 6];
    short_id.copy_from_slice(&hash_bytes[0..6]);
    short_id
}
```

### Collision Analysis

- **Short ID size:** 6 bytes = 48 bits
- **Collision probability per pair:** ~1 / 2^48 = ~3.55 x 10^-15
- **For a block with 1000 transactions:** ~500,000 pairs, probability of any collision ~1.78 x 10^-9
- **For a block with 10,000 transactions:** ~50,000,000 pairs, probability of any collision ~1.78 x 10^-7
- **Acceptable:** Even at 10,000 transactions per block, collision probability is negligible. If a collision occurs, the missing transaction request protocol (CBK-004) handles it.

## Acceptance Criteria

- [ ] `ShortTxId` type is defined as `[u8; 6]`
- [ ] `SHORT_TX_ID_BYTES` constant is defined as `6`
- [ ] `compute_short_tx_id` accepts a 16-byte SipHash key and a transaction ID byte slice
- [ ] `compute_short_tx_id` returns a 6-byte array
- [ ] Implementation uses `siphasher` crate's `SipHasher`
- [ ] The SipHash key is split into two `u64` values (k0, k1) from the 16-byte key
- [ ] Output is the first 6 bytes of the SipHash-2-4 result in little-endian order
- [ ] Same key + same tx_id always produces the same ShortTxId (deterministic)
- [ ] Different tx_ids with the same key produce different ShortTxIds (with overwhelming probability)

## Implementation Notes

- Primary file: `src/types/compact_block.rs` or `src/gossip/compact_block.rs`
- Dependencies: `siphasher` crate (version 1.x), gated behind the `compact-blocks` feature flag
- Edge cases:
  - The SipHash key bytes are interpreted as little-endian u64 values -- this must be consistent between sender and receiver
  - An empty `tx_id` input is technically valid but should not occur in practice
  - The truncation to 6 bytes discards 2 bytes of the 8-byte SipHash output; this is intentional and matches BIP 152's approach
  - SipHash-2-4 (the default) is used, not SipHash-1-3 or other variants
