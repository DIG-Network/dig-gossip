# CBK-004: Missing Transaction Request

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 8.2](../../../resources/SPEC.md)

## Summary

When compact block reconstruction leaves some transactions unmatched (the receiver's mempool does not contain them), the receiver sends a `RequestBlockTransactions` message specifying the block hash and the indices of the missing transactions. The sender responds with `RespondBlockTransactions` containing the full serialized transactions for the requested indices. This completes the block reconstruction without requiring a full block retransmission.

## Specification

### Message Types

```rust
use serde::{Serialize, Deserialize};

/// Request for specific missing transactions from a compact block.
/// Sent by the receiver when mempool reconstruction leaves gaps.
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct RequestBlockTransactions {
    /// Hash of the block being reconstructed.
    pub block_hash: Bytes32,
    /// Indices of transactions missing from the receiver's mempool.
    /// These correspond to positions in the original block's transaction list.
    pub missing_indices: Vec<u16>,
}

/// Response containing the requested missing transactions.
/// Sent by the original compact block sender.
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct RespondBlockTransactions {
    /// The full serialized transactions at the requested indices.
    /// Order matches the order of indices in the request.
    pub transactions: Vec<Vec<u8>>,
}
```

### Protocol Flow

```
Receiver (after reconstruction with missing txs, count <= COMPACT_BLOCK_MAX_MISSING_TXS):
  1. Collect missing_indices from ReconstructionResult
  2. Send RequestBlockTransactions {
       block_hash: compact_block.header.hash(),
       missing_indices: missing_indices,
     } to the peer that sent the CompactBlock
  3. Await RespondBlockTransactions from that peer

Sender (receives RequestBlockTransactions):
  1. Look up the full block by block_hash
  2. Extract the serialized transactions at the requested indices
  3. Send RespondBlockTransactions { transactions } back to the requester

Receiver (receives RespondBlockTransactions):
  1. Insert received transactions into the ReconstructionResult at their
     corresponding missing indices
  2. Verify all transaction slots are now filled
  3. Full block is reconstructed: header + all transactions
  4. Hand off to caller for block validation
```

### Integration with Reconstruction

```rust
impl ReconstructionResult {
    /// Fill in missing transactions from a RespondBlockTransactions message.
    /// Returns Ok(()) if all gaps are filled, Err if counts don't match.
    pub fn fill_missing(
        &mut self,
        response: &RespondBlockTransactions,
    ) -> Result<(), CompactBlockError> {
        if response.transactions.len() != self.missing_indices.len() {
            return Err(CompactBlockError::MissingCountMismatch {
                expected: self.missing_indices.len(),
                received: response.transactions.len(),
            });
        }

        for (idx, tx) in self.missing_indices.iter().zip(response.transactions.iter()) {
            self.transactions[*idx as usize] = Some(tx.clone());
        }

        self.missing_indices.clear();
        Ok(())
    }

    /// Check if the block is fully reconstructed (no missing transactions).
    pub fn is_complete(&self) -> bool {
        self.missing_indices.is_empty()
            && self.transactions.iter().all(|t| t.is_some())
    }
}
```

## Acceptance Criteria

- [ ] `RequestBlockTransactions` struct has `block_hash` (Bytes32) and `missing_indices` (Vec<u16>)
- [ ] `RespondBlockTransactions` struct has `transactions` (Vec<Vec<u8>>)
- [ ] Request is sent only when `missing_indices` is non-empty and count <= COMPACT_BLOCK_MAX_MISSING_TXS (5)
- [ ] Request is sent to the same peer that sent the original CompactBlock
- [ ] Response transaction count matches the number of requested indices
- [ ] Received transactions are inserted at the correct indices in the reconstruction
- [ ] After filling, `is_complete()` returns true when all slots are filled
- [ ] Mismatched response count returns an error
- [ ] The full block (header + all transactions) is available after successful fill

## Implementation Notes

- Primary file: `src/gossip/compact_block.rs` and `src/types/compact_block.rs`
- Dependencies: `Bytes32` from `chia-protocol`, `ReconstructionResult` from CBK-003
- Edge cases:
  - The sender may no longer have the block (evicted from cache). In this case, no response is sent and the receiver should fall back to full block request (CBK-005 logic applies for timeout).
  - `missing_indices` must be sorted and deduplicated before sending the request.
  - If `RespondBlockTransactions` contains fewer transactions than requested, the reconstruction remains incomplete and the receiver should fall back to a full block request.
  - The response must maintain the same ordering as the request's `missing_indices` list for correct slot insertion.
  - This path is only used when the number of missing transactions is <= COMPACT_BLOCK_MAX_MISSING_TXS (5). For more missing transactions, CBK-005 applies.
