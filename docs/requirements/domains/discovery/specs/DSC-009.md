# DSC-009: Parallel Connection Establishment

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 6.4](../../../resources/SPEC.md)

## Summary

Parallel connection establishment batches up to `PARALLEL_CONNECT_BATCH_SIZE` (8) concurrent connection attempts using `FuturesUnordered`. This is an improvement over Chia's sequential one-at-a-time approach (which uses `asyncio.sleep()` between attempts) and provides N-times faster bootstrap when discovering and connecting to initial peers.

## Specification

### Constant

```rust
/// Maximum number of concurrent connection attempts in a single batch.
/// Improvement over Chia's sequential one-at-a-time approach.
pub const PARALLEL_CONNECT_BATCH_SIZE: usize = 8;
```

### Parallel Connection Logic

```rust
use futures::stream::{FuturesUnordered, StreamExt};

/// Attempt to connect to multiple peers concurrently.
/// Selects up to PARALLEL_CONNECT_BATCH_SIZE candidates from the address manager
/// and connects to them in parallel via FuturesUnordered.
///
/// Returns the list of successfully connected peers.
pub async fn connect_batch(
    address_manager: &AddressManager,
    handshake_config: &HandshakeConfig,
    existing_outbound: &HashSet<IpAddr>,
    group_filter: &GroupFilter,
) -> Vec<(Peer, PeerInfo)> {
    let mut futures = FuturesUnordered::new();
    let mut candidates = Vec::new();

    // Select up to PARALLEL_CONNECT_BATCH_SIZE candidates
    for _ in 0..PARALLEL_CONNECT_BATCH_SIZE {
        if let Some(candidate) = address_manager.select_peer(false).await {
            // Skip if already connected or fails diversity check
            if existing_outbound.contains(&candidate.peer_info.ip())
                || !group_filter.is_allowed(&candidate.peer_info)
            {
                continue;
            }
            candidates.push(candidate.clone());
            futures.push(try_connect_with_info(candidate, handshake_config));
        }
    }

    // Collect results as they complete
    let mut connected = Vec::new();
    while let Some(result) = futures.next().await {
        match result {
            Ok((peer, info)) => {
                address_manager.mark_good(&info).await;
                connected.push((peer, info));
            }
            Err((info, _err)) => {
                address_manager.attempt(&info, true).await;
            }
        }
    }

    connected
}

/// Connect to a single peer, returning the peer and its info on success.
async fn try_connect_with_info(
    candidate: ExtendedPeerInfo,
    config: &HandshakeConfig,
) -> Result<(Peer, PeerInfo), (PeerInfo, GossipError)> {
    match try_connect(&candidate.peer_info, config).await {
        Ok(peer) => Ok((peer, candidate.peer_info)),
        Err(e) => Err((candidate.peer_info, e)),
    }
}
```

### Integration with Discovery Loop

The parallel connection batch is invoked from the discovery loop (DSC-006) during Phase 2:

```rust
// Phase 2 of discovery loop (DSC-006):
let needed = config.target_outbound.saturating_sub(current_outbound_count);
if needed > 0 {
    let connected = connect_batch(
        &address_manager,
        &config.handshake,
        &existing_outbound,
        &group_filter,
    ).await;

    for (peer, info) in connected {
        // Register as outbound connection
        // Send RequestPeers (DSC-007)
        request_peers_from(&peer, &info, &address_manager).await.ok();
    }
}
```

## Acceptance Criteria

- [ ] PARALLEL_CONNECT_BATCH_SIZE is 8
- [ ] Up to 8 connection attempts run concurrently via `FuturesUnordered`
- [ ] Successfully connected peers are marked good via `mark_good()`
- [ ] Failed connection attempts are recorded via `attempt(count_failure=true)`
- [ ] Already-connected peers are skipped (no duplicate connections)
- [ ] Group filter (DSC-010, DSC-011) is applied before attempting connection
- [ ] Results are collected as they complete (not waiting for all to finish)
- [ ] Fewer than 8 candidates is handled gracefully (connect whatever is available)
- [ ] Zero candidates results in no connections attempted

## Implementation Notes

- Primary file: `src/discovery/node_discovery.rs`
- Dependencies: `futures` (`FuturesUnordered`, `StreamExt`), `tokio`, AddressManager (DSC-001), connection utilities
- Edge cases:
  - Address manager has fewer peers than PARALLEL_CONNECT_BATCH_SIZE: connect to however many are available
  - All connections fail: return empty vec, discovery loop handles retry
  - Some connections succeed, some fail: return partial results
  - Connection timeout should be per-attempt (e.g., 10 seconds), not shared across the batch
  - A peer selected by `select_peer()` may be selected again in the same batch if the manager returns it: deduplicate candidates by IP before spawning futures
  - FuturesUnordered is used (not JoinSet) to process results as they arrive rather than waiting for all
