# DSC-005: Introducer Registration

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 6.5](../../../resources/SPEC.md)

## Summary

Introducer registration is a DIG extension not present in Chia. It allows a node to register itself with an introducer server so that other nodes can discover it. The flow is: connect via WebSocket, perform handshake, send `register_peer` with the node's IP, port, and node type, receive `register_ack`, and close the connection.

## Specification

### Registration Flow

```rust
/// Register this node with an introducer server.
///
/// Flow:
///   1. Connect to introducer via WebSocket (TLS)
///   2. Perform Chia protocol handshake
///   3. Send RegisterPeer { ip, port, node_type }
///   4. Receive RegisterAck
///   5. Close connection
///
/// DIG extension: not present in Chia's introducer protocol.
pub async fn register_with_introducer(
    introducer_addr: &str,
    handshake_config: &HandshakeConfig,
    registration: &PeerRegistration,
    timeout: Duration,
) -> Result<RegisterAck, GossipError> {
    // 1. Connect via WebSocket with TLS
    let peer = connect_to_introducer(introducer_addr, handshake_config, timeout).await?;

    // 2. Send RegisterPeer
    let ack: RegisterAck = peer
        .request_infallible(RegisterPeer {
            ip: registration.ip.clone(),
            port: registration.port,
            node_type: registration.node_type,
        })
        .await
        .map_err(GossipError::from)?;

    // 3. Close connection
    drop(peer);

    Ok(ack)
}
```

### DIG Extension Types

```rust
/// Registration request sent to the introducer.
/// DIG extension -- not part of chia-protocol.
#[derive(Streamable, Serialize, Deserialize)]
pub struct RegisterPeer {
    /// This node's externally-reachable IP address.
    pub ip: String,
    /// This node's P2P listening port.
    pub port: u16,
    /// This node's type (typically NodeType::FullNode).
    pub node_type: NodeType,
}

/// Acknowledgement from the introducer that registration succeeded.
#[derive(Streamable, Serialize, Deserialize)]
pub struct RegisterAck {
    /// Whether the registration was accepted.
    pub success: bool,
}
```

### PeerRegistration Config

```rust
/// Configuration for registering with an introducer.
pub struct PeerRegistration {
    /// This node's externally-reachable IP address.
    pub ip: String,
    /// This node's P2P listening port.
    pub port: u16,
    /// This node's type.
    pub node_type: NodeType,
}
```

## Acceptance Criteria

- [ ] Registration connects to introducer via WebSocket with TLS
- [ ] Chia protocol handshake is performed before sending registration
- [ ] `RegisterPeer` message includes ip, port, and node_type
- [ ] `RegisterAck` is received and indicates success/failure
- [ ] Connection is closed after receiving ack
- [ ] Timeout is enforced on the entire registration operation
- [ ] Registration failures return `GossipError` (do not panic)
- [ ] `RegisterPeer` and `RegisterAck` are DIG-specific types (not from chia-protocol)
- [ ] Uses DIG message type IDs in the 200+ range for wire encoding

## Implementation Notes

- Primary file: `src/discovery/introducer_client.rs`
- Dependencies: `chia-protocol` (`NodeType`, `Handshake`), `chia-sdk-client` (`Peer`, TLS), `chia-traits` (`Streamable`)
- Edge cases:
  - Introducer may reject registration (success=false): return the ack, let caller decide
  - Introducer may not support registration (old version): timeout or error
  - IP address should be the externally-reachable address, not a local/private address
  - Registration should be retried periodically (handled by discovery loop, not this function)
  - Message type IDs for RegisterPeer/RegisterAck must be in the DIG extension range (200+) to avoid colliding with Chia's ProtocolMessageTypes
