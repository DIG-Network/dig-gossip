# DSC-006: Discovery Loop

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)

> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)

## Summary

The discovery loop is the main driver for peer discovery, ported from Chia's `node_discovery.py:256-293` with improvements. It orchestrates DNS seeding, introducer queries, and peer connection attempts in a continuous loop. When the address manager is empty, it seeds via DNS first (round-robin across configured DNS introducers), then falls back to the introducer with exponential backoff. After peers are received, it waits 5 seconds before the next cycle.

## Specification

### Discovery Loop Algorithm

```rust
/// Main discovery loop. Runs continuously until cancelled.
/// Ported from node_discovery.py:256-293 with improvements.
pub async fn run_discovery_loop(
    address_manager: Arc<AddressManager>,
    config: &DiscoveryConfig,
    cancel: CancellationToken,
) {
    let mut introducer_backoff = Duration::from_secs(1);
    let max_backoff = Duration::from_secs(300);

    loop {
        if cancel.is_cancelled() {
            break;
        }

        // Phase 1: Seed if address manager is empty
        if address_manager.size().await == 0 {
            // Try DNS first (round-robin across configured DNS introducers)
            let addrs = dns_seed(&config.network, config.dns_timeout, config.dns_batch_size).await;

            if !addrs.is_empty() {
                // Convert and add to address manager
                let timestamped = to_timestamped(&addrs);
                address_manager.add_to_new_table(&timestamped, &self_info, 0).await;
                introducer_backoff = Duration::from_secs(1); // reset backoff
            } else {
                // DNS failed or empty -- fall back to introducer with backoff
                match query_introducer(&config.introducer_addr, &config.handshake, config.introducer_timeout).await {
                    Ok(peers) if !peers.is_empty() => {
                        address_manager.add_to_new_table(&peers, &introducer_info, 0).await;
                        introducer_backoff = Duration::from_secs(1); // reset backoff
                    }
                    _ => {
                        // Exponential backoff: 1s -> 2s -> 4s -> ... -> 300s max
                        tokio::time::sleep(introducer_backoff).await;
                        introducer_backoff = (introducer_backoff * 2).min(max_backoff);
                        continue; // retry immediately after backoff
                    }
                }
            }
        }

        // Phase 2: Connect to peers (see DSC-009 for parallel batching)
        // Phase 3: Feeler connections (see DSC-008)
        // Phase 4: /16 and AS diversity checks (see DSC-010, DSC-011)

        // Wait before next cycle
        tokio::time::sleep(Duration::from_secs(5)).await;
    }
}
```

### DiscoveryConfig

```rust
/// Configuration for the discovery loop.
pub struct DiscoveryConfig {
    /// Network configuration for DNS seeding.
    pub network: Network,
    /// DNS resolution timeout.
    pub dns_timeout: Duration, // default: 30s
    /// DNS batch size.
    pub dns_batch_size: usize, // default: 2
    /// Introducer server address.
    pub introducer_addr: String,
    /// Introducer query timeout.
    pub introducer_timeout: Duration, // default: 15s
    /// Handshake configuration.
    pub handshake: HandshakeConfig,
    /// Target number of outbound connections.
    pub target_outbound: usize, // default: 8
}
```

### Backoff Schedule

- **Initial backoff:** 1 second
- **Backoff multiplier:** 2x (exponential)
- **Maximum backoff:** 300 seconds (5 minutes)
- **Reset:** Backoff resets to 1 second when peers are successfully received from any source
- **Applies to:** Introducer queries only (DNS is always attempted first)

### Seeding Priority

1. **DNS first:** Always attempt DNS seeding first when address manager is empty. DNS is fast, low-overhead, and does not require a persistent connection.
2. **Introducer fallback:** If DNS returns no results, query the introducer. Introducer requires a full WebSocket connection and handshake, so it is more expensive.
3. **Round-robin DNS:** When multiple DNS introducers are configured, they are queried in round-robin order across discovery cycles.

## Acceptance Criteria

- [ ] Discovery loop runs continuously until cancellation token is triggered
- [ ] DNS seeding is attempted first when address manager is empty
- [ ] Introducer is queried only when DNS returns no results
- [ ] Introducer backoff starts at 1 second
- [ ] Introducer backoff doubles on each failure (exponential)
- [ ] Introducer backoff caps at 300 seconds
- [ ] Backoff resets to 1 second when peers are successfully received
- [ ] 5-second wait between discovery cycles after peers are available
- [ ] DNS introducers are queried in round-robin order
- [ ] Discovery loop does not panic on any failure condition
- [ ] All errors are logged with appropriate severity

## Implementation Notes

- Primary file: `src/discovery/node_discovery.rs`
- Dependencies: `tokio` (timers, cancellation), AddressManager (DSC-001), DNS seeding (DSC-003), introducer query (DSC-004)
- Edge cases:
  - Both DNS and introducer fail simultaneously: backoff and retry
  - Address manager becomes empty after initial seeding (all peers failed): re-enter seeding phase
  - Cancellation during backoff sleep: check token after sleep returns
  - Multiple DNS introducers: round-robin index wraps around
  - Very fast cycles when address manager is not empty but target outbound not met: the 5s sleep prevents busy-looping
