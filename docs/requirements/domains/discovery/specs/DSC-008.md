# DSC-008: Feeler Connections

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 6.4](../../../resources/SPEC.md)

## Summary

Feeler connections are short-lived connections used to test whether addresses in the "new" table are reachable. They run on a Poisson schedule with a 240-second average interval. A random address is selected from the new table, a connection is attempted, and on success the address is promoted to the "tried" table via `mark_good()`. The connection is then immediately closed -- feelers are not maintained as persistent peers. Ported from `node_discovery.py:308-325`.

## Specification

### Feeler Connection Logic

```rust
/// Average interval between feeler connections (Poisson distribution).
pub const FEELER_INTERVAL_SECS: u64 = 240;

/// Run feeler connections on a Poisson schedule.
/// Ported from node_discovery.py:308-325.
pub async fn run_feeler_loop(
    address_manager: Arc<AddressManager>,
    handshake_config: &HandshakeConfig,
    cancel: CancellationToken,
) {
    loop {
        // Poisson-distributed wait: -ln(U) * FEELER_INTERVAL_SECS
        let wait = poisson_next_interval(FEELER_INTERVAL_SECS);
        tokio::select! {
            _ = tokio::time::sleep(wait) => {},
            _ = cancel.cancelled() => break,
        }

        // Select a random address from the new table
        let Some(candidate) = address_manager.select_peer(true).await else {
            continue; // no new-table addresses available
        };

        // Attempt connection
        match try_connect(&candidate.peer_info, handshake_config).await {
            Ok(peer) => {
                // Success: promote to tried table
                address_manager.mark_good(&candidate.peer_info).await;
                // Immediately close -- feelers are not maintained
                drop(peer);
                tracing::debug!("Feeler success: {}", candidate.peer_info);
            }
            Err(_) => {
                // Failure: record the failed attempt
                address_manager.attempt(&candidate.peer_info, true).await;
                tracing::debug!("Feeler failed: {}", candidate.peer_info);
            }
        }
    }
}
```

### Poisson Schedule

```rust
use rand::Rng;

/// Generate the next interval for a Poisson process.
/// Returns Duration based on exponential distribution with given average.
fn poisson_next_interval(average_secs: u64) -> Duration {
    let mut rng = rand::thread_rng();
    let u: f64 = rng.gen_range(0.0001..1.0); // avoid ln(0)
    let interval_secs = -(u.ln()) * average_secs as f64;
    Duration::from_secs_f64(interval_secs)
}
```

### Feeler Behavior

1. **Selection:** Always from the "new" table (`select_peer(new_only=true)`).
2. **On success:** Call `mark_good()` to promote to tried table, then close connection.
3. **On failure:** Call `attempt(count_failure=true)` to record the failure.
4. **Duration:** Feelers are immediately closed after the connection result is known. They do not participate in gossip or peer exchange.

## Acceptance Criteria

- [ ] Feeler connections run on a Poisson schedule with 240s average interval
- [ ] Candidates are selected from the new table only (`select_peer(true)`)
- [ ] On successful connection, `mark_good()` is called to promote to tried table
- [ ] On successful connection, the peer is immediately disconnected
- [ ] On failed connection, `attempt(count_failure=true)` is called
- [ ] Feeler connections do not send `RequestPeers` or participate in gossip
- [ ] Poisson timing uses exponential distribution: `-ln(U) * average`
- [ ] Feeler loop respects cancellation token
- [ ] Empty new table is handled gracefully (skip and wait for next interval)

## Implementation Notes

- Primary file: `src/discovery/node_discovery.rs`
- Dependencies: `rand` (Poisson timing), `tokio` (timers, cancellation), AddressManager (DSC-001), connection utilities
- Edge cases:
  - New table may be empty: `select_peer(true)` returns None, skip this cycle
  - Connection may timeout: treat as failure
  - Very short Poisson intervals (rare but possible): do not rate-limit feelers beyond the natural distribution
  - Address may have been promoted by another path (e.g., outbound connect) between selection and feeler attempt: `mark_good()` handles this gracefully
  - Feeler should not count toward outbound connection limits
