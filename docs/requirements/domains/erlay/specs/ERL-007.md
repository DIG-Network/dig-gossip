# ERL-007: Inbound Peer Exclusion from Flood Set

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 8.3](../../../resources/SPEC.md)

## Summary

Inbound peers (peers that connected to us) MUST be excluded from the flood set. In the ERLAY model, inbound peers initiate set reconciliation with the local node rather than receiving flooded `NewTransaction` messages. This design choice is taken directly from the ERLAY paper (Naumenko et al., 2019) and ensures optimal propagation latency: outbound peers are chosen by the local node and are more likely to be diverse and well-connected, while inbound peers are self-selected and could be adversarial or redundant.

## Specification

### Inbound/Outbound Classification

The `PeerConnection.is_outbound` field determines whether a peer is outbound (we initiated the connection) or inbound (they connected to us).

```rust
/// Extended peer connection state for the gossip layer.
pub struct PeerConnection {
    /// The underlying chia-sdk-client Peer connection.
    pub peer: Peer,
    /// Unique peer identifier.
    pub peer_id: PeerId,
    /// Whether we initiated this connection (outbound) or they connected to us (inbound).
    pub is_outbound: bool,
    // ... other fields
}
```

### Flood Set Filtering

```rust
impl ErlayState {
    /// Select the flood set, filtering to outbound peers only.
    /// Inbound peers are NEVER candidates for the flood set.
    pub fn select_flood_set(
        &mut self,
        outbound_peers: &[PeerId],
        rng: &mut impl rand::Rng,
    ) {
        // Caller MUST pass only outbound peers.
        // This is enforced at the call site:
        //
        // let outbound_peers: Vec<PeerId> = connections.values()
        //     .filter(|c| c.is_outbound)
        //     .map(|c| c.peer_id.clone())
        //     .collect();
        // erlay_state.select_flood_set(&outbound_peers, &mut rng);

        self.flood_set.clear();
        let mut candidates: Vec<PeerId> = outbound_peers.to_vec();
        candidates.shuffle(rng);
        let count = candidates.len().min(ERLAY_FLOOD_PEER_COUNT);
        self.flood_set = candidates[..count].iter().cloned().collect();
    }
}
```

### Inbound Peer Reconciliation Role

Inbound peers participate in ERLAY exclusively through set reconciliation:

```
Inbound peer behavior:
  - NEVER receives NewTransaction via flooding from the local node
  - DOES receive tx_ids via set reconciliation (ERL-004)
  - MAY initiate reconciliation with the local node (they send their sketch first)
  - The local node responds to reconciliation requests from inbound peers
```

### Invariants

- The flood set MUST contain zero inbound peers at all times.
- The `select_flood_set` function MUST only receive outbound peer IDs as candidates.
- Inbound peers MUST still participate in set reconciliation (they are not excluded from ERLAY entirely, only from flooding).
- If all connected peers are inbound, the flood set MUST be empty.
- When an inbound peer initiates reconciliation, the local node MUST respond with its own sketch and participate in difference resolution.

## Acceptance Criteria

- [ ] Flood set contains zero inbound peers after every selection and rotation
- [ ] Only `PeerConnection` entries with `is_outbound == true` are candidates for flood set
- [ ] Inbound peers still participate in set reconciliation
- [ ] `NewTransaction` is never flooded to inbound peers
- [ ] With 10 inbound peers and 0 outbound peers, flood set is empty
- [ ] With 5 inbound and 5 outbound peers, flood set contains at most 5 peers (all outbound)
- [ ] With 5 inbound and 20 outbound peers, flood set contains exactly 8 outbound peers
- [ ] Inbound peers can initiate reconciliation and the local node responds
- [ ] Inbound peers accumulate tx_ids in their reconciliation sets

## Implementation Notes

- Primary file: `src/gossip/erlay.rs`
- Dependencies: `PeerConnection` from `types::peer` (for `is_outbound` field)
- Edge cases:
  - A peer that was outbound may become classified differently if the connection is re-established in the opposite direction; the current connection's `is_outbound` flag is authoritative
  - Eclipse attack mitigation: excluding inbound peers from flooding prevents an attacker who opens many inbound connections from receiving all transaction announcements early, which could be used for front-running or transaction censorship detection
  - Mixed network: in a network where a node has many inbound and few outbound peers, transaction propagation via flooding is limited; reconciliation compensates for this
