# ERL-001: Flood Set Selection

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 8.3](../../../resources/SPEC.md)

## Summary

The flood set determines which connected peers receive immediate `NewTransaction` messages via low-fanout flooding. The set MUST consist of exactly `ERLAY_FLOOD_PEER_COUNT` (8) randomly selected connected outbound peers. If fewer than 8 outbound peers are connected, the flood set includes all available outbound peers. This is the entry point for ERLAY-style transaction relay: only flood-set peers get eager transaction announcements; all other peers receive transactions via periodic set reconciliation.

## Specification

### Flood Set Construction

On each flood set computation (at startup and every rotation interval), the node selects up to `ERLAY_FLOOD_PEER_COUNT` peers from the set of currently connected outbound peers using uniform random sampling without replacement.

```rust
use crate::constants::ERLAY_FLOOD_PEER_COUNT;
use crate::types::peer::PeerId;
use rand::seq::SliceRandom;
use std::collections::HashSet;

/// ERLAY flood set manager.
/// Tracks which peers receive immediate NewTransaction flooding.
pub struct ErlayState {
    /// Peers selected for low-fanout flooding.
    /// These receive NewTransaction immediately.
    pub flood_set: HashSet<PeerId>,
    /// Sketch of tx_ids announced since last reconciliation, per peer.
    /// Key: non-flood peer_id, Value: set of tx_ids to reconcile.
    pub reconciliation_sets: HashMap<PeerId, HashSet<Bytes32>>,
}

impl ErlayState {
    /// Select the flood set from connected outbound peers.
    /// Chooses min(ERLAY_FLOOD_PEER_COUNT, outbound_peers.len()) peers
    /// uniformly at random.
    pub fn select_flood_set(
        &mut self,
        outbound_peers: &[PeerId],
        rng: &mut impl rand::Rng,
    ) {
        self.flood_set.clear();
        let mut candidates: Vec<PeerId> = outbound_peers.to_vec();
        candidates.shuffle(rng);
        let count = candidates.len().min(ERLAY_FLOOD_PEER_COUNT);
        self.flood_set = candidates[..count].iter().cloned().collect();
    }
}
```

### Invariants

- The flood set MUST contain only outbound peers (see ERL-007).
- The flood set size MUST be `min(ERLAY_FLOOD_PEER_COUNT, connected_outbound_count)`.
- Selection MUST be uniformly random (no weighting by reputation, latency, or other factors).
- When a flood-set peer disconnects, it MUST be removed from the flood set immediately. The set is not refilled until the next rotation (ERL-006).

## Acceptance Criteria

- [ ] `ErlayState` struct exists with a `flood_set: HashSet<PeerId>` field
- [ ] `select_flood_set` populates `flood_set` with up to `ERLAY_FLOOD_PEER_COUNT` peers
- [ ] Only outbound peers are considered as candidates for the flood set
- [ ] Selection uses uniform random sampling without replacement
- [ ] When fewer than 8 outbound peers exist, all outbound peers are in the flood set
- [ ] When exactly 8 outbound peers exist, all are in the flood set
- [ ] When more than 8 outbound peers exist, exactly 8 are selected
- [ ] Disconnected peers are removed from the flood set immediately
- [ ] `ERLAY_FLOOD_PEER_COUNT` constant is defined as `8` in `constants.rs`

## Implementation Notes

- Primary file: `src/gossip/erlay.rs`
- Constants file: `src/constants.rs`
- Dependencies: `rand` for random peer selection, `PeerId` from `types::peer`
- Edge cases:
  - Zero connected outbound peers: flood set is empty; all transaction relay uses reconciliation only
  - Rapid peer churn: flood set may temporarily have fewer than 8 members between rotations
  - The flood set is a `HashSet` to ensure O(1) membership checks during `on_new_transaction`
