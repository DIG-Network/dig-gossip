# ERL-002: Low-Fanout Flooding via NewTransaction

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 8.3](../../../resources/SPEC.md)

## Summary

When a new transaction arrives (either from the local application layer or from a peer), the node sends `NewTransaction` only to peers in the flood set -- not to all connected peers. This is the "low-fanout" component of ERLAY. After flooding, the `tx_id` is added to the local reconciliation sketch so that non-flood peers will discover the transaction during the next set reconciliation round.

## Specification

### Low-Fanout Flood Algorithm

```rust
use chia_protocol::{Bytes32, NewTransaction};
use crate::constants::ERLAY_FLOOD_PEER_COUNT;

impl ErlayState {
    /// Handle a new transaction: flood to flood-set peers,
    /// then add tx_id to reconciliation sketches for non-flood peers.
    pub async fn on_new_transaction(
        &mut self,
        tx_id: Bytes32,
        new_tx: NewTransaction,
        connections: &HashMap<PeerId, PeerConnection>,
    ) -> Result<usize, GossipError> {
        let mut sent_count = 0;

        // Step 1: Send NewTransaction to flood-set peers only
        for peer_id in &self.flood_set {
            if let Some(conn) = connections.get(peer_id) {
                conn.peer.send(new_tx.clone()).await?;
                sent_count += 1;
            }
        }

        // Step 2: Add tx_id to local reconciliation sketch
        // for all non-flood connected peers
        for (peer_id, _conn) in connections {
            if !self.flood_set.contains(peer_id) {
                self.reconciliation_sets
                    .entry(peer_id.clone())
                    .or_default()
                    .insert(tx_id);
            }
        }

        Ok(sent_count)
    }
}
```

### Protocol Flow

```
on_new_transaction(tx_id, cost, fees):
  1. Select flood-set peers (already computed, see ERL-001)
  2. Send NewTransaction { tx_id, cost, fees } to flood-set peers only
  3. Add tx_id to local reconciliation sketch for each non-flood peer
```

### Invariants

- `NewTransaction` MUST NOT be sent to peers outside the flood set via this path.
- The `tx_id` MUST be added to reconciliation state for every connected non-flood peer.
- If the flood set is empty (no outbound peers), no flooding occurs; the transaction will propagate exclusively through reconciliation.
- Duplicate transactions (already seen) MUST NOT be flooded again (handled by the seen set / deduplication layer).

## Acceptance Criteria

- [ ] `on_new_transaction` sends `NewTransaction` only to peers in `flood_set`
- [ ] `on_new_transaction` does not send `NewTransaction` to peers outside `flood_set`
- [ ] After flooding, `tx_id` is added to reconciliation sets for all non-flood connected peers
- [ ] The method returns the count of peers the transaction was sent to
- [ ] If flood set is empty, zero peers receive `NewTransaction` and tx_id is added to all reconciliation sets
- [ ] Duplicate transactions are not re-flooded (deduplication check before flooding)
- [ ] If a flood-set peer is disconnected (not in connections), it is skipped without error

## Implementation Notes

- Primary file: `src/gossip/erlay.rs`
- Dependencies: `chia-protocol` for `NewTransaction`, `Bytes32`; `PeerConnection` for sending
- Edge cases:
  - A flood-set peer that disconnected between rotation and the flood call: skip it gracefully
  - High transaction volume: each call only touches `ERLAY_FLOOD_PEER_COUNT` peers, so cost is O(1) per transaction regardless of total connection count
  - The reconciliation set grows until the next reconciliation round (ERL-004) drains it
- Integration: works with the seen set (deduplication) and Plumtree (which handles block and attestation gossip separately)
