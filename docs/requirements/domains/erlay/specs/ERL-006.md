# ERL-006: Flood Set Rotation

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 8.3](../../../resources/SPEC.md)

## Summary

The flood set is re-randomized every `ERLAY_FLOOD_SET_ROTATION_SECS` (60) seconds. Rotation prevents long-lived topology patterns that could lead to transaction propagation blind spots or enable targeted censorship. Each rotation selects a fresh random set of up to `ERLAY_FLOOD_PEER_COUNT` outbound peers.

## Specification

### Rotation Timer

```rust
use crate::constants::{ERLAY_FLOOD_SET_ROTATION_SECS, ERLAY_FLOOD_PEER_COUNT};
use tokio::time::{interval, Duration};

impl ErlayState {
    /// Periodically re-randomize the flood set.
    /// This runs as a background task for the lifetime of the node.
    pub async fn flood_set_rotation_loop(
        &mut self,
        get_outbound_peers: impl Fn() -> Vec<PeerId>,
        mut rng: impl rand::Rng,
    ) {
        let mut timer = interval(
            Duration::from_secs(ERLAY_FLOOD_SET_ROTATION_SECS)
        );

        loop {
            timer.tick().await;

            let outbound_peers = get_outbound_peers();
            self.select_flood_set(&outbound_peers, &mut rng);

            tracing::debug!(
                flood_set_size = self.flood_set.len(),
                "Rotated ERLAY flood set"
            );
        }
    }
}
```

### Constants

```rust
/// Flood set re-randomization interval (seconds).
/// Default: 60.
pub const ERLAY_FLOOD_SET_ROTATION_SECS: u64 = 60;
```

### Rotation Behavior

```
Every ERLAY_FLOOD_SET_ROTATION_SECS (60 seconds):
  1. Get current list of connected outbound peers
  2. Call select_flood_set() with the current outbound peer list (see ERL-001)
  3. New flood set takes effect immediately for subsequent on_new_transaction() calls
  4. Non-flood peers that were previously in the flood set begin accumulating
     reconciliation entries; newly-added flood-set peers stop accumulating
```

### Invariants

- Rotation MUST occur every `ERLAY_FLOOD_SET_ROTATION_SECS` (60) seconds.
- The new flood set MUST be selected from the current connected outbound peers at rotation time (not a stale list).
- Rotation MUST use fresh randomness (not re-use a previous seed).
- Peers that move from flood set to non-flood MUST immediately begin accumulating tx_ids in their reconciliation set.
- Peers that move from non-flood to flood set MUST have their pending reconciliation set drained or cleared (any pending tx_ids were already handled by the previous reconciliation round or will be delivered via flooding going forward).
- Rotation MUST NOT disrupt in-flight reconciliation rounds; any in-progress round completes with the old classification.

## Acceptance Criteria

- [ ] Flood set is re-randomized every 60 seconds
- [ ] Rotation uses the current connected outbound peer list (not stale data)
- [ ] New flood set membership takes effect immediately for new transactions
- [ ] Fresh randomness is used on each rotation
- [ ] Peers transitioning from flood to non-flood begin accumulating reconciliation entries
- [ ] Peers transitioning from non-flood to flood have reconciliation state cleared
- [ ] In-flight reconciliation rounds are not disrupted by rotation
- [ ] `ERLAY_FLOOD_SET_ROTATION_SECS` constant is defined as `60` in `constants.rs`
- [ ] Rotation is logged at debug level with the new flood set size

## Implementation Notes

- Primary file: `src/gossip/erlay.rs`
- Constants file: `src/constants.rs`
- Dependencies: `tokio` for async timer, `rand` for randomization, `tracing` for logging
- Edge cases:
  - Peer count changes between rotations (peers connect/disconnect): rotation always uses the current list
  - All outbound peers disconnect: flood set becomes empty; all relay is via reconciliation with inbound peers
  - Rotation during high transaction volume: new flood set applies to the next `on_new_transaction` call; in-flight sends to the old flood set complete normally
  - Very rapid peer churn: flood set may contain peers that disconnect before the next rotation; these are skipped in `on_new_transaction` (ERL-002)
