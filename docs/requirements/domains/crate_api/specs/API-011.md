# API-011: ExtendedPeerInfo and VettedPeer

## Summary

ExtendedPeerInfo and VettedPeer are Rust ports of Chia Python types that have no Rust crate equivalent. ExtendedPeerInfo (from `address_manager.py:43`) stores extended metadata for peers in the address manager's tried/new tables, including timestamps, attempt counts, and bucket position. VettedPeer (from `introducer_peers.py:12-28`) tracks peers known to the introducer with their vetting status and attempt history.

## Specification

### ExtendedPeerInfo Struct

```rust
/// Extended peer info for the address manager.
/// Rust port of Chia's ExtendedPeerInfo (address_manager.py:43).
pub struct ExtendedPeerInfo {
    /// The peer's address info.
    pub peer_info: PeerInfo,
    /// Timestamp when this peer was last seen (Unix seconds).
    pub timestamp: u64,
    /// The peer that told us about this address (source).
    pub src: PeerInfo,
    /// Position in the random-order table (for O(1) random selection).
    pub random_pos: Option<usize>,
    /// Whether this peer is in the "tried" table (vs "new" table).
    pub is_tried: bool,
    /// Number of address manager entries referencing this peer.
    pub ref_count: u32,
    /// Timestamp of last successful connection (Unix seconds).
    pub last_success: u64,
    /// Timestamp of last connection attempt (Unix seconds).
    pub last_try: u64,
    /// Number of connection attempts.
    pub num_attempts: u32,
    /// Timestamp of last counted attempt (Unix seconds).
    pub last_count_attempt: u64,
}
```

### VettedPeer Struct

```rust
/// A peer tracked by the introducer with vetting status.
/// Rust port of Chia's VettedPeer (introducer_peers.py:12-28).
#[derive(Debug, Clone, PartialEq, Eq, Hash)]
pub struct VettedPeer {
    /// Peer hostname or IP address.
    pub host: String,
    /// Peer port.
    pub port: u16,
    /// Vetting status:
    /// 0 = not vetted,
    /// negative = consecutive failures,
    /// positive = consecutive successes.
    pub vetted: i32,
    /// Timestamp of last vetting check (Unix seconds).
    pub vetted_timestamp: u64,
    /// Timestamp of last connection attempt (Unix seconds).
    pub last_attempt: u64,
    /// Timestamp when this peer was first added (Unix seconds).
    pub time_added: u64,
}
```

### ExtendedPeerInfo Semantics

| Field | Semantics |
|-------|-----------|
| `peer_info` | The peer's resolved address (host + port) |
| `timestamp` | When this entry was last updated; used for staleness checks |
| `src` | The peer that advertised this address to us; used for source-group bucketing |
| `random_pos` | Index into the random-order array for O(1) random peer selection |
| `is_tried` | `true` if we have successfully connected to this peer at least once |
| `ref_count` | Number of "new" table bucket entries pointing to this peer; 0 when in tried table |
| `last_success` | Most recent successful connection timestamp; 0 if never connected |
| `last_try` | Most recent attempt timestamp; used for retry backoff |
| `num_attempts` | Total connection attempts; used with `MAX_RETRIES` for eviction |
| `last_count_attempt` | Rate-limits attempt counting to avoid rapid retry penalization |

### VettedPeer Semantics

| Field | Semantics |
|-------|-----------|
| `host` | Peer hostname or IP address string |
| `port` | Peer listening port |
| `vetted` | Consecutive success/failure counter; positive = good, negative = bad, 0 = unvetted |
| `vetted_timestamp` | When the last vetting check was performed |
| `last_attempt` | When the last connection attempt was made |
| `time_added` | When this peer was first added to the introducer's tracking |

## Implementation Notes

- Primary file: `src/types/peer.rs` (ExtendedPeerInfo alongside PeerInfo), `src/discovery/introducer_peers.rs` (VettedPeer)
- Dependencies: `PeerInfo` (this crate)
- Edge cases:
  - `random_pos = None` indicates the peer has not been assigned a random-order position yet
  - `ref_count = 0` for tried-table entries is expected; for new-table entries it means the entry should be garbage collected
  - `num_attempts` combined with `last_try` determines retry eligibility per `MAX_RETRIES` and `MIN_FAIL_DAYS` constants
  - `last_success = 0` means the peer has never been successfully connected to
  - `vetted` uses signed integer: positive values indicate the number of consecutive successful vetting connections, negative values indicate consecutive failures
  - `VettedPeer` implements `Hash` to allow use in `HashSet` for the introducer's peer tracking
