# API-010: IntroducerConfig and RelayConfig

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Sections 2.11, 2.12](../../../resources/SPEC.md)

## Summary

IntroducerConfig and RelayConfig are configuration structs for the optional introducer and relay subsystems. IntroducerConfig configures the introducer client used for bootstrap peer discovery and registration. RelayConfig configures the relay client used as a transparent fallback when direct P2P connections fail. Both derive Serialize/Deserialize for configuration file support.

## Specification

### IntroducerConfig Struct

```rust
/// Introducer client configuration.
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct IntroducerConfig {
    /// Introducer endpoint (e.g., "ws://introducer.example.com:9448").
    pub endpoint: String,
    /// Connection timeout in seconds. Default: 10.
    pub connection_timeout_secs: u64,
    /// Request timeout in seconds. Default: 10.
    pub request_timeout_secs: u64,
    /// Network ID string. Default: "DIG_MAINNET".
    pub network_id: String,
}
```

### RelayConfig Struct

```rust
/// Relay client configuration.
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct RelayConfig {
    /// Relay server endpoint (e.g., "wss://relay.example.com:9450").
    pub endpoint: String,
    /// Enable relay. Default: true (when endpoint set).
    pub enabled: bool,
    /// Connection timeout in seconds. Default: 10.
    pub connection_timeout_secs: u64,
    /// Reconnect delay in seconds. Default: 5.
    pub reconnect_delay_secs: u64,
    /// Max reconnect attempts. Default: 10.
    pub max_reconnect_attempts: u32,
    /// Ping interval in seconds. Default: 30.
    pub ping_interval_secs: u64,
    /// Prefer relay over direct. Default: false.
    pub prefer_relay: bool,
}
```

## Implementation Notes

- Primary file: `src/types/config.rs`
- Dependencies: `serde::{Serialize, Deserialize}`
- Edge cases:
  - `endpoint` has no meaningful default since it is deployment-specific; the `Default` impl may use an empty string as a sentinel, but code should validate it is non-empty before use
  - `connection_timeout_secs = 0` should be treated as "no timeout" or rejected during validation
  - `max_reconnect_attempts = 0` means no reconnection attempts
  - `prefer_relay = true` should route all traffic through relay even when direct P2P is available
  - `enabled = false` with a valid endpoint should prevent relay connection
