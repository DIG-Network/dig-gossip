## Summary

GossipService provides the entry point for the gossip layer. The `new()` constructor accepts a `GossipConfig`, initializes TLS via chia-ssl, and prepares internal state without starting any networking. The caller must explicitly invoke `start()` to begin peer discovery and connections. This two-phase initialization pattern separates configuration validation from runtime activation.

## Specification

### Constructor Signature

```rust
impl GossipService {
    /// Create a new gossip service with the given configuration.
    /// TLS is set up via chia-ssl (load_ssl_cert / ChiaCertificate::generate()).
    pub fn new(config: GossipConfig) -> Result<Self, GossipError>;
}
```

### Construction Behavior

1. **TLS initialization**: Load TLS certificate from `config.cert_path` and `config.key_path` using `chia-sdk-client::load_ssl_cert()`. If the files do not exist, generate a new certificate pair using `chia-ssl::ChiaCertificate::generate()`.
2. **Configuration validation**: Validate that required fields are populated (e.g., `network_id` is non-zero, `listen_addr` is parseable, `target_outbound_count <= max_connections`).
3. **Internal state preparation**: Initialize address manager, seen message set, peer connection map, and channel infrastructure. No tasks are spawned and no network I/O is performed.
4. **Error propagation**: Return `GossipError::IoError` for TLS file errors, `GossipError::ClientError` for TLS creation failures.

### Lifecycle Methods

```rust
impl GossipService {
    /// Start the gossip service: bind listener, start discovery, connect to
    /// bootstrap peers, start relay (if configured).
    pub async fn start(&self) -> Result<GossipHandle, GossipError>;

    /// Gracefully stop: disconnect all peers, stop discovery, close relay.
    pub async fn stop(&self) -> Result<(), GossipError>;
}
```

### Usage Example

```rust
use dig_gossip::{GossipService, GossipConfig, GossipError};

let config = GossipConfig {
    listen_addr: "0.0.0.0:9444".parse().unwrap(),
    peer_id: my_peer_id,
    network_id: dig_network_id,
    // ... remaining fields
    ..Default::default()
};

let service = GossipService::new(config)?;
let handle = service.start().await?;

// Use handle for messaging...

service.stop().await?;
```

## Acceptance Criteria

- [ ] `GossipService::new(config)` accepts a `GossipConfig` and returns `Result<Self, GossipError>`
- [ ] Constructor loads TLS cert via `load_ssl_cert()` when cert/key files exist
- [ ] Constructor generates TLS cert via `ChiaCertificate::generate()` when cert/key files are missing
- [ ] Constructor does NOT spawn any async tasks or perform network I/O
- [ ] Constructor validates configuration fields and returns appropriate errors
- [ ] `start()` returns a `GossipHandle` and begins networking
- [ ] `stop()` gracefully disconnects all peers and terminates background tasks
- [ ] Calling `start()` twice returns `GossipError` (already started)
- [ ] Calling methods on handle after `stop()` returns `GossipError::ServiceNotStarted`

## Implementation Notes

- Primary file: `src/service/gossip_service.rs`
- Dependencies: `chia-ssl::ChiaCertificate`, `chia-sdk-client::load_ssl_cert`, `GossipConfig`, `GossipError`, `GossipHandle`
- Edge cases:
  - TLS cert paths that point to unreadable files should produce `GossipError::IoError`
  - Generating a new cert should persist it to the configured paths for future reuse
  - The constructor is synchronous (`fn new`, not `async fn new`) since TLS cert loading is file I/O only
