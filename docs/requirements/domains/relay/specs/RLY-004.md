# RLY-004: Auto-Reconnect on Disconnect

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 7, Section 2.12](../../../resources/SPEC.md)

## Summary

The relay service must automatically reconnect to the relay server when the WebSocket connection drops unexpectedly. Reconnection uses a configurable delay between attempts and gives up after a maximum number of consecutive failures. This ensures relay-dependent peers maintain connectivity through transient network issues or relay server restarts, without requiring manual intervention or full service restart.

### Relay Service with Auto-Reconnect

```rust
/// RelayService manages the relay connection lifecycle including auto-reconnect.
pub struct RelayService {
    config: RelayConfig,
    peer_id: PeerId,
    network_id: Bytes32,
    protocol_version: u32,
    client: Option<RelayClient>,
    consecutive_failures: u32,
    stats: RelayStats,
}

impl RelayService {
    /// Run the relay service loop. Connects, processes messages, and
    /// reconnects on disconnection until max_reconnect_attempts is exceeded.
    pub async fn run(
        &mut self,
        inbound_tx: mpsc::Sender<(PeerId, Message)>,
        mut shutdown: tokio::sync::watch::Receiver<bool>,
    ) -> Result<(), GossipError> {
        loop {
            // Attempt connection and registration (RLY-001)
            match RelayClient::connect_and_register(
                &self.config,
                self.peer_id,
                self.network_id,
                self.protocol_version,
            ).await {
                Ok(client) => {
                    self.client = Some(client);
                    self.consecutive_failures = 0;
                    self.stats.connected = true;
                    self.stats.last_connected_at = Some(current_timestamp());
                    self.stats.reconnect_attempts = 0;

                    tracing::info!(
                        relay = %self.config.endpoint,
                        "relay connection established"
                    );

                    // Run message loop until disconnect or shutdown
                    let disconnect_reason = self.run_message_loop(
                        &inbound_tx,
                        &mut shutdown,
                    ).await;

                    self.client = None;
                    self.stats.connected = false;

                    match disconnect_reason {
                        DisconnectReason::Shutdown => {
                            tracing::info!("relay service shutting down");
                            return Ok(());
                        }
                        DisconnectReason::Error(e) => {
                            tracing::warn!("relay disconnected: {e}");
                        }
                        DisconnectReason::ServerClosed => {
                            tracing::info!("relay server closed connection");
                        }
                    }
                }
                Err(e) => {
                    tracing::warn!(
                        relay = %self.config.endpoint,
                        attempt = self.consecutive_failures + 1,
                        max = self.config.max_reconnect_attempts,
                        "relay connection failed: {e}"
                    );
                }
            }

            // Increment failure counter
            self.consecutive_failures += 1;
            self.stats.reconnect_attempts = self.consecutive_failures;

            // Check if we've exceeded max attempts
            if self.consecutive_failures >= self.config.max_reconnect_attempts {
                tracing::error!(
                    attempts = self.consecutive_failures,
                    "relay reconnection attempts exhausted, giving up"
                );
                return Err(GossipError::RelayError(
                    format!(
                        "max reconnect attempts ({}) exceeded",
                        self.config.max_reconnect_attempts
                    )
                ));
            }

            // Wait before retrying (check for shutdown during wait)
            tokio::select! {
                _ = tokio::time::sleep(
                    Duration::from_secs(self.config.reconnect_delay_secs)
                ) => {
                    tracing::debug!(
                        delay_secs = self.config.reconnect_delay_secs,
                        "reconnecting to relay"
                    );
                }
                _ = shutdown.changed() => {
                    tracing::info!("relay service shutting down during reconnect delay");
                    return Ok(());
                }
            }
        }
    }
}

enum DisconnectReason {
    Shutdown,
    Error(String),
    ServerClosed,
}
```

## Acceptance Criteria

- [ ] Relay service automatically attempts reconnection after unexpected disconnect
- [ ] Reconnect delay is `RelayConfig::reconnect_delay_secs` (default: 5 seconds)
- [ ] Reconnection stops after `RelayConfig::max_reconnect_attempts` (default: 10) consecutive failures
- [ ] `consecutive_failures` counter resets to 0 on successful reconnection
- [ ] `RelayStats::reconnect_attempts` reflects the current consecutive failure count
- [ ] Reconnect delay is interruptible by shutdown signal
- [ ] Graceful shutdown stops the reconnect loop immediately
- [ ] Each reconnection attempt performs the full Register/RegisterAck handshake (RLY-001)
- [ ] Connection errors are logged with attempt count and max attempts
- [ ] Exceeding max attempts returns `GossipError::RelayError`

## Implementation Notes

- Primary files: `src/relay/relay_service.rs`, `src/relay/relay_client.rs`
- Dependencies: `tokio::time::sleep`, `tokio::sync::watch` (shutdown signal)
- Edge cases:
  - Relay server may accept TCP but reject at WebSocket or Register level; all count as failures
  - If the relay server is restarted, all clients reconnect simultaneously; consider adding jitter to `reconnect_delay_secs` (e.g., `delay * (0.5 + rand(0..1))`)
  - A long-lived connection that drops after hours of uptime should reset `consecutive_failures` to 0 before incrementing (it was a transient failure, not a persistent one)
  - The shutdown channel should be checked both during the reconnect delay and during the message loop
