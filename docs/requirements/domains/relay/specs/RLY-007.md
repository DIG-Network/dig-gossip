# RLY-007: NAT Traversal Hole Punching

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 7.1](../../../resources/SPEC.md)

## Summary

The relay client must support NAT traversal via STUN-style hole punching coordinated through the relay server. When two peers are connected via relay, either peer can initiate a hole punch attempt to upgrade to a direct P2P connection. The relay server acts as a signaling channel: it forwards external address information between peers and coordinates simultaneous connection attempts. On success, message traffic migrates to the direct connection and the relay path is dropped. On failure, the relay path is kept and a retry is scheduled after `HOLE_PUNCH_RETRY_SECS` (default: 300 seconds). This extends the static relay connections from `l2_driver_state_channel` with an upgrade path that reduces relay load and latency.

## Specification

### NAT Traversal Message Variants (SPEC Section 7.1)

```rust
use std::net::SocketAddr;

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(tag = "type")]
pub enum RelayMessage {
    // ... Register, RegisterAck, RelayGossipMessage, Broadcast, GetPeers, Peers, Ping, Pong ...

    /// Request NAT traversal assistance.
    /// Sent by initiating peer to the relay server.
    HolePunchRequest {
        /// The requesting peer's ID.
        peer_id: PeerId,
        /// The target peer to attempt direct connection with.
        target_peer_id: PeerId,
        /// The requesting peer's observed external IP:port
        /// (determined via STUN or relay server feedback).
        external_addr: SocketAddr,
    },

    /// NAT traversal coordination from relay.
    /// Sent by the relay server to both peers to initiate simultaneous connect.
    HolePunchCoordinate {
        /// The peer ID of the other side of the hole punch.
        peer_id: PeerId,
        /// The other peer's observed external IP:port to connect to.
        external_addr: SocketAddr,
    },

    /// NAT traversal result.
    /// Sent by each peer back to the relay to report outcome.
    HolePunchResult {
        /// The peer reporting the result.
        peer_id: PeerId,
        /// Whether the direct connection was successfully established.
        success: bool,
    },
}

/// Retry interval for failed hole punch attempts (seconds).
pub const HOLE_PUNCH_RETRY_SECS: u64 = 300;
```

### Hole Punch Procedure (SPEC Section 7.1)

```rust
/// NAT traversal state machine for a single peer pair.
#[derive(Debug)]
pub enum HolePunchState {
    /// No hole punch attempted or last attempt failed.
    Idle,
    /// HolePunchRequest sent, waiting for HolePunchCoordinate.
    AwaitingCoordination,
    /// HolePunchCoordinate received, attempting simultaneous connect.
    Connecting {
        target_addr: SocketAddr,
        started_at: tokio::time::Instant,
    },
    /// Direct connection established, relay path dropped.
    Upgraded,
    /// Hole punch failed, retry scheduled.
    RetryScheduled {
        retry_at: tokio::time::Instant,
    },
}

impl RelayClient {
    /// Initiate NAT traversal with a target peer.
    ///
    /// Step 1-2 of SPEC Section 7.1:
    /// Both peers A and B are connected via relay.
    /// A sends HolePunchRequest to relay with its observed external IP:port.
    pub async fn initiate_hole_punch(
        &mut self,
        target_peer_id: PeerId,
        external_addr: SocketAddr,
    ) -> Result<(), GossipError> {
        let msg = RelayMessage::HolePunchRequest {
            peer_id: self.peer_id,
            target_peer_id,
            external_addr,
        };
        self.send(&msg).await?;

        self.hole_punch_states.insert(
            target_peer_id,
            HolePunchState::AwaitingCoordination,
        );

        tracing::info!(
            target = %target_peer_id,
            external_addr = %external_addr,
            "initiated hole punch request"
        );

        Ok(())
    }

    /// Handle HolePunchCoordinate received from relay.
    ///
    /// Step 3-5 of SPEC Section 7.1:
    /// Relay forwards coordination with the other peer's external address.
    /// Both peers attempt simultaneous connection.
    pub async fn handle_hole_punch_coordinate(
        &mut self,
        peer_id: PeerId,
        external_addr: SocketAddr,
    ) -> Result<(), GossipError> {
        tracing::info!(
            peer = %peer_id,
            addr = %external_addr,
            "received hole punch coordinate, attempting direct connection"
        );

        self.hole_punch_states.insert(
            peer_id,
            HolePunchState::Connecting {
                target_addr: external_addr,
                started_at: tokio::time::Instant::now(),
            },
        );

        // Attempt simultaneous direct connection
        // Both peers connect to each other's external_addr at the same time
        match self.attempt_direct_connection(peer_id, external_addr).await {
            Ok(direct_peer) => {
                // Step 6: Success - migrate traffic to direct connection
                tracing::info!(
                    peer = %peer_id,
                    addr = %external_addr,
                    "hole punch succeeded, migrating to direct connection"
                );

                self.hole_punch_states.insert(peer_id, HolePunchState::Upgraded);

                // Report success to relay
                let result = RelayMessage::HolePunchResult {
                    peer_id: self.peer_id,
                    success: true,
                };
                self.send(&result).await?;

                // Signal the gossip service to migrate this peer's traffic
                self.migration_tx.send((peer_id, direct_peer)).await
                    .map_err(|_| GossipError::ChannelClosed)?;
            }
            Err(e) => {
                // Step 7: Failure - keep relay path, schedule retry
                tracing::warn!(
                    peer = %peer_id,
                    addr = %external_addr,
                    error = %e,
                    retry_secs = HOLE_PUNCH_RETRY_SECS,
                    "hole punch failed, keeping relay path"
                );

                self.hole_punch_states.insert(
                    peer_id,
                    HolePunchState::RetryScheduled {
                        retry_at: tokio::time::Instant::now()
                            + Duration::from_secs(HOLE_PUNCH_RETRY_SECS),
                    },
                );

                // Report failure to relay
                let result = RelayMessage::HolePunchResult {
                    peer_id: self.peer_id,
                    success: false,
                };
                self.send(&result).await?;
            }
        }

        Ok(())
    }

    /// Attempt a direct TCP+TLS connection to the peer's external address.
    async fn attempt_direct_connection(
        &self,
        peer_id: PeerId,
        external_addr: SocketAddr,
    ) -> Result<Peer, GossipError> {
        // Use chia-sdk-client's connect_peer() for TLS handshake
        let timeout_duration = Duration::from_secs(10);
        let peer = tokio::time::timeout(
            timeout_duration,
            connect_peer(external_addr, self.tls_connector.clone()),
        )
        .await
        .map_err(|_| GossipError::RelayError("hole punch connect timeout".into()))?
        .map_err(|e| GossipError::RelayError(format!("hole punch connect failed: {e}")))?;

        Ok(peer)
    }
}
```

### Traffic Migration

```rust
/// When hole punch succeeds, migrate message traffic from relay to direct.
impl GossipService {
    async fn handle_hole_punch_migration(
        &mut self,
        peer_id: PeerId,
        direct_peer: Peer,
    ) {
        // 1. Register the direct connection
        self.add_peer_connection(peer_id, direct_peer).await;

        // 2. Remove the peer from relay-only routing
        if let Some(relay) = &mut self.relay_service {
            relay.remove_relay_route(peer_id);
        }

        // 3. Future messages to this peer use direct connection
        tracing::info!(
            peer = %peer_id,
            "migrated from relay to direct P2P connection"
        );
    }
}
```

## Acceptance Criteria

- [ ] `HolePunchRequest { peer_id, target_peer_id, external_addr }` is sent to the relay to initiate NAT traversal
- [ ] Relay forwards `HolePunchCoordinate { peer_id, external_addr }` to both peers
- [ ] Both peers attempt simultaneous direct TCP+TLS connection to each other's external address
- [ ] On success: `HolePunchResult { success: true }` is sent to the relay
- [ ] On success: message traffic migrates to the direct connection
- [ ] On success: relay path for this peer pair is dropped
- [ ] On failure: `HolePunchResult { success: false }` is sent to the relay
- [ ] On failure: relay path is kept (no change to message delivery)
- [ ] On failure: retry is scheduled after `HOLE_PUNCH_RETRY_SECS` (300 seconds)
- [ ] Hole punch state is tracked per peer pair (`HolePunchState` enum)
- [ ] Direct connection uses full TLS handshake via `chia-sdk-client::connect_peer()`
- [ ] Connection attempt has a timeout (does not block indefinitely)

## Implementation Notes

- Primary files: `src/relay/relay_client.rs`, `src/relay/relay_service.rs`, `src/relay/relay_types.rs`, `src/service/gossip_service.rs`
- Dependencies: `chia-sdk-client::connect_peer`, `tokio::time`, `tokio::net::TcpStream`
- Edge cases:
  - Both peers may initiate hole punch simultaneously for the same pair; the relay should deduplicate
  - The external address observed by the relay may differ from the peer's actual externally reachable address (e.g., behind symmetric NAT); hole punch will fail in this case
  - Symmetric NAT (different external port per destination) makes hole punching unreliable; the 300s retry avoids wasted effort
  - If the direct connection succeeds but the TLS handshake fails (wrong certificate), treat as failure
  - Migration must be atomic: no message loss during the switch from relay to direct
  - If the direct connection drops after migration, the peer should fall back to relay (not just reconnect direct)
  - Retry timer should be cancelled if the peer disconnects entirely from the relay
