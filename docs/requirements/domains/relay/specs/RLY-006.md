# RLY-006: Relay Keepalive

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 7, Section 2.12, Section 2.13](../../../resources/SPEC.md)

## Summary

The relay client must maintain the WebSocket connection by sending periodic `Ping` messages and expecting `Pong` responses. The ping interval is configured via `RelayConfig::ping_interval_secs` (default: 30). If no `Pong` is received within `PEER_TIMEOUT_SECS` (90 seconds), the connection is considered dead and the auto-reconnect mechanism (RLY-004) is triggered. Keepalive also provides latency measurements that are tracked in `RelayStats`.

### Keepalive Loop

```rust
use std::time::{SystemTime, UNIX_EPOCH};

/// Constants from SPEC Section 2.13.
pub const PING_INTERVAL_SECS: u64 = 30;
pub const PEER_TIMEOUT_SECS: u64 = 90;

impl RelayService {
    async fn run_message_loop(
        &mut self,
        inbound_tx: &mpsc::Sender<(PeerId, Message)>,
        shutdown: &mut tokio::sync::watch::Receiver<bool>,
    ) -> DisconnectReason {
        let ping_interval = Duration::from_secs(self.config.ping_interval_secs);
        let mut ping_timer = tokio::time::interval(ping_interval);
        let mut last_pong_at = tokio::time::Instant::now();

        loop {
            tokio::select! {
                // Handle incoming messages from relay
                msg = self.client.as_mut().unwrap().recv() => {
                    match msg {
                        Some(Ok(RelayMessage::Pong { timestamp })) => {
                            last_pong_at = tokio::time::Instant::now();

                            // Calculate RTT from echoed timestamp
                            let now_ms = SystemTime::now()
                                .duration_since(UNIX_EPOCH)
                                .unwrap()
                                .as_millis() as u64;
                            let rtt_ms = now_ms.saturating_sub(timestamp);
                            self.stats.latency_ms = Some(rtt_ms);

                            tracing::trace!(rtt_ms, "relay pong received");
                        }
                        // ... handle other message types ...
                        Some(Err(e)) => {
                            return DisconnectReason::Error(format!("{e}"));
                        }
                        None => {
                            return DisconnectReason::ServerClosed;
                        }
                    }
                }

                // Send periodic Ping
                _ = ping_timer.tick() => {
                    // Check if we've timed out waiting for a pong
                    if last_pong_at.elapsed() > Duration::from_secs(PEER_TIMEOUT_SECS) {
                        tracing::warn!(
                            elapsed_secs = last_pong_at.elapsed().as_secs(),
                            timeout_secs = PEER_TIMEOUT_SECS,
                            "relay connection timed out (no pong received)"
                        );
                        return DisconnectReason::Error(
                            "relay keepalive timeout".into()
                        );
                    }

                    let timestamp = SystemTime::now()
                        .duration_since(UNIX_EPOCH)
                        .unwrap()
                        .as_millis() as u64;

                    let ping = RelayMessage::Ping { timestamp };
                    if let Err(e) = self.client.as_mut().unwrap().send(&ping).await {
                        tracing::warn!("failed to send relay ping: {e}");
                        return DisconnectReason::Error(format!("ping send failed: {e}"));
                    }

                    tracing::trace!("relay ping sent");
                }

                // Handle shutdown
                _ = shutdown.changed() => {
                    return DisconnectReason::Shutdown;
                }
            }
        }
    }
}
```

### Latency Tracking

```rust
impl RelayClient {
    /// Send a Ping message with the current timestamp.
    pub async fn send_ping(&mut self) -> Result<(), GossipError> {
        let timestamp = SystemTime::now()
            .duration_since(UNIX_EPOCH)
            .unwrap()
            .as_millis() as u64;

        let msg = RelayMessage::Ping { timestamp };
        self.send(&msg).await
    }

    /// Process a Pong response and return the round-trip time in milliseconds.
    pub fn process_pong(&mut self, timestamp: u64) -> u64 {
        let now_ms = SystemTime::now()
            .duration_since(UNIX_EPOCH)
            .unwrap()
            .as_millis() as u64;
        now_ms.saturating_sub(timestamp)
    }
}
```

## Acceptance Criteria

- [ ] Client sends `Ping { timestamp }` at `RelayConfig::ping_interval_secs` intervals (default: 30s)
- [ ] `Ping` includes a millisecond-precision timestamp for RTT calculation
- [ ] Server responds with `Pong { timestamp }` echoing the ping timestamp
- [ ] RTT is calculated as `now - pong.timestamp` and stored in `RelayStats::latency_ms`
- [ ] If no `Pong` is received within `PEER_TIMEOUT_SECS` (90s), the connection is declared dead
- [ ] Dead connection triggers auto-reconnect (RLY-004)
- [ ] `last_pong_at` is initialized to connection time (avoids false timeout on connect)
- [ ] Failed `Ping` send triggers disconnect and reconnect
- [ ] Keepalive runs concurrently with message processing (non-blocking)
- [ ] Ping/Pong messages are serialized as JSON like all relay messages

## Implementation Notes

- Primary files: `src/relay/relay_service.rs`, `src/relay/relay_client.rs`, `src/relay/relay_types.rs`
- Dependencies: `tokio::time::interval`, `std::time::SystemTime`
- Edge cases:
  - Clock skew between client and server does not affect RTT calculation (timestamp is echoed, not compared to server time)
  - If the ping interval is longer than the timeout, the timeout will never fire on the first check; ensure `ping_interval_secs < PEER_TIMEOUT_SECS`
  - WebSocket protocol-level pings (opcode 0x9) are separate from application-level Ping messages; both may be active
  - Very high RTT (>1000ms) should be logged as a warning but not treated as a failure
  - `saturating_sub` prevents underflow if clock jumps backward
