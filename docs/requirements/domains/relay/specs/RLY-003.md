# RLY-003: Relay Broadcast

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 7, Section 8.1 step 7](../../../resources/SPEC.md)

## Summary

The relay client must support broadcasting a message to all peers connected to the relay server on the same network. The `Broadcast` message includes an exclude list to prevent sending the message back to the original source or to peers that have already received it via direct P2P. This integrates with the Plumtree gossip algorithm (SPEC Section 8.1, step 7) where relay broadcast supplements the eager/lazy push to reach peers only accessible via relay.

## Specification

### Broadcast Variant

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(tag = "type")]
pub enum RelayMessage {
    // ... Register, RegisterAck, RelayGossipMessage ...

    /// Broadcast a message to all relay-connected peers on the same network.
    Broadcast {
        /// Sender's PeerId.
        from: PeerId,
        /// Serialized message payload.
        payload: Vec<u8>,
        /// PeerIds to exclude from receiving this broadcast
        /// (e.g., the origin peer, peers already reached via direct P2P).
        exclude: Vec<PeerId>,
    },

    // ... other variants ...
}
```

### Sending a Broadcast

```rust
impl RelayClient {
    /// Broadcast a message to all relay-connected peers, excluding specified peers.
    ///
    /// Called by the Plumtree broadcast algorithm (step 7) when a relay
    /// connection is active. The exclude list prevents echo loops.
    ///
    /// # Errors
    /// Returns `GossipError::RelayError` if the relay connection is broken
    /// or serialization fails.
    pub async fn broadcast(
        &mut self,
        payload: Vec<u8>,
        exclude: Vec<PeerId>,
    ) -> Result<(), GossipError> {
        let msg = RelayMessage::Broadcast {
            from: self.peer_id,
            payload,
            exclude,
        };

        let json = serde_json::to_string(&msg)
            .map_err(|e| GossipError::RelayError(format!("serialize error: {e}")))?;

        self.write.send(TungsteniteMessage::Text(json)).await
            .map_err(|e| GossipError::RelayError(format!("send error: {e}")))?;

        Ok(())
    }
}
```

### Receiving a Broadcast

When the relay server delivers a broadcast message to this client, it arrives as a `RelayGossipMessage` (the server converts broadcast to per-peer delivery). The client processes it via the same path as RLY-002:

```rust
/// Integration with Plumtree broadcast (SPEC Section 8.1, step 7):
///
/// broadcast(message, origin):
///   ...
///   5. For each peer in eager_peers (excluding origin): peer.send_raw(message)
///   6. For each peer in lazy_peers (excluding origin): peer.send_raw(LazyAnnounce)
///   7. If relay connected: relay.broadcast(message, exclude_list)
///   ...
async fn plumtree_broadcast(
    &self,
    message: &Message,
    origin: Option<PeerId>,
    relay_client: Option<&mut RelayClient>,
) -> usize {
    let mut count = 0;

    // Steps 5-6: direct P2P eager/lazy push (handled elsewhere)

    // Step 7: relay broadcast
    if let Some(relay) = relay_client {
        let payload = message.to_bytes();
        // Exclude: the origin peer + all peers we already sent to via direct P2P
        let mut exclude = self.eager_peers.iter()
            .chain(self.lazy_peers.iter())
            .copied()
            .collect::<Vec<_>>();
        if let Some(origin) = origin {
            exclude.push(origin);
        }

        if let Err(e) = relay.broadcast(payload, exclude).await {
            tracing::warn!("relay broadcast failed: {e}");
        } else {
            count += 1;
        }
    }

    count
}
```

## Acceptance Criteria

- [ ] `Broadcast { from, payload, exclude }` is correctly serialized as JSON
- [ ] `from` field is set to the broadcasting peer's PeerId
- [ ] `payload` contains the serialized message bytes
- [ ] `exclude` list contains PeerIds that must not receive the broadcast
- [ ] The exclude list includes the origin peer (prevents echo)
- [ ] The exclude list includes peers already reached via direct P2P eager/lazy push
- [ ] Relay broadcast is called as step 7 of the Plumtree broadcast algorithm
- [ ] Broadcast failure does not abort the overall broadcast (logged as warning)
- [ ] Relay connection errors return `GossipError::RelayError`
- [ ] An empty exclude list is valid (broadcasts to all relay peers)

## Implementation Notes

- Primary files: `src/relay/relay_client.rs`, `src/relay/relay_types.rs`, `src/gossip/broadcaster.rs`
- Dependencies: `serde_json`, `tokio-tungstenite`
- Edge cases:
  - If the exclude list contains all relay-connected peers, the relay server sends to nobody (valid no-op)
  - Large payloads (full blocks) may hit WebSocket frame size limits; ensure frame size is configured
  - The relay server is responsible for fanout; the client sends one message regardless of peer count
  - Race condition: a peer may connect to relay between building the exclude list and the relay processing the broadcast; this is acceptable (duplicate detection handles it)
