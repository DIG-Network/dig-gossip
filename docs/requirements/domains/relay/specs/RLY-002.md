# RLY-002: Relay Message Forwarding

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 7](../../../resources/SPEC.md)

## Summary

The relay client must support sending messages to a specific peer through the relay server using `RelayGossipMessage`. This is the unicast path: when a direct P2P connection to a peer is unavailable (due to NAT or firewall), the relay forwards the message on behalf of the sender. Each message carries a monotonically increasing sequence number to enable ordering and duplicate detection at the receiver.

## Specification

pub enum RelayMessage {

### Sending a Relayed Message

```rust
impl RelayClient {
    /// Send a message to a specific peer through the relay server.
    ///
    /// The payload is the serialized form of a chia-protocol::Message.
    /// The sequence number is tracked per target peer and auto-incremented.
    ///
    /// # Errors
    /// Returns `GossipError::RelayError` if the relay connection is broken
    /// or serialization fails.
    pub async fn send_to_peer(
        &mut self,
        to: PeerId,
        payload: Vec<u8>,
    ) -> Result<(), GossipError> {
        let seq = self.next_seq(&to);

        let msg = RelayMessage::RelayGossipMessage {
            from: self.peer_id,
            to,
            payload,
            seq,
        };

        let json = serde_json::to_string(&msg)
            .map_err(|e| GossipError::RelayError(format!("serialize error: {e}")))?;

        self.write.send(TungsteniteMessage::Text(json)).await
            .map_err(|e| GossipError::RelayError(format!("send error: {e}")))?;

        Ok(())
    }

    /// Get the next sequence number for messages to a specific peer.
    fn next_seq(&mut self, to: &PeerId) -> u64 {
        let entry = self.seq_counters.entry(*to).or_insert(0);
        *entry += 1;
        *entry
    }
}
```

### Receiving a Relayed Message

```rust
impl RelayClient {
    /// Process an incoming RelayGossipMessage from the relay server.
    /// Verifies the message is addressed to this peer and checks for
    /// duplicate/out-of-order sequence numbers.
    async fn handle_relay_gossip_message(
        &mut self,
        from: PeerId,
        to: PeerId,
        payload: Vec<u8>,
        seq: u64,
    ) -> Result<Option<Vec<u8>>, GossipError> {
        // Verify this message is for us
        if to != self.peer_id {
            tracing::warn!(
                from = %from,
                to = %to,
                "received relay message addressed to another peer"
            );
            return Ok(None);
        }

        // Check sequence number for duplicate detection
        let last_seq = self.received_seqs.entry(from).or_insert(0);
        if seq <= *last_seq {
            tracing::debug!(from = %from, seq, "duplicate relay message, ignoring");
            return Ok(None);
        }
        *last_seq = seq;

        Ok(Some(payload))
    }
}
```

### Transparent Delivery

Inbound relay messages MUST be delivered to the same application-layer channel as direct P2P messages. The caller sees no difference in transport:

```rust
/// Relay messages are forwarded to the gossip service's inbound channel,
/// identical to how direct P2P messages arrive.
async fn relay_message_loop(
    relay_client: &mut RelayClient,
    inbound_tx: mpsc::Sender<(PeerId, Message)>,
) {
    while let Some(msg) = relay_client.recv().await {
        match msg {
            RelayMessage::RelayGossipMessage { from, to, payload, seq } => {
                if let Some(payload) = relay_client
                    .handle_relay_gossip_message(from, to, payload, seq)
                    .await?
                {
                    // Deserialize and deliver to the same channel as direct messages
                    let message = Message::from_bytes(&payload)?;
                    inbound_tx.send((from, message)).await?;
                }
            }
            // ... handle other message types ...
        }
    }
}
```

## Acceptance Criteria

- [ ] `RelayGossipMessage { from, to, payload, seq }` is correctly serialized as JSON
- [ ] `from` field is set to the sender's PeerId
- [ ] `to` field specifies the intended recipient's PeerId
- [ ] `payload` contains the serialized chia-protocol `Message` bytes
- [ ] `seq` is monotonically increasing per sender-receiver pair
- [ ] Sequence numbers are tracked independently per target peer
- [ ] Duplicate messages (same or lower `seq` from same `from`) are silently dropped
- [ ] Messages addressed to a different peer are ignored with a warning log
- [ ] Inbound relay messages are delivered to the same channel as direct P2P messages
- [ ] Relay connection errors return `GossipError::RelayError`

## Implementation Notes

- Primary files: `src/relay/relay_client.rs`, `src/relay/relay_types.rs`
- Dependencies: `serde_json`, `tokio-tungstenite`
- Edge cases:
  - Payload may be large (up to 2MB+ for full blocks); WebSocket frame size limits must be configured accordingly
  - Sequence number overflow at `u64::MAX` is practically unreachable but should wrap gracefully
  - If the relay server is restarted, sequence counters on both sides reset; receivers should handle seq resets from known peers
  - The `payload` field uses `Vec<u8>` which serde_json serializes as an array of integers; consider base64 encoding for efficiency
