# RLY-005: Relay Peer List

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 7, Section 2.9](../../../resources/SPEC.md)

## Summary

The relay client must support querying the list of peers currently connected to the relay server on a given network. The client sends `GetPeers { network_id }` and the server responds with `Peers { peers }` containing a vector of `RelayPeerInfo` entries. This peer list is used by the gossip layer to know which peers are reachable via relay, enabling targeted message forwarding (RLY-002) and informing transport selection decisions (RLY-008).

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct RelayPeerInfo {
    /// The peer's unique identifier.
    pub peer_id: PeerId,
    /// The network this peer is connected to.
    pub network_id: Bytes32,
    /// The peer's protocol version.
    pub protocol_version: u32,
    /// Unix timestamp (seconds) when the peer connected to the relay.
    pub connected_at: u64,
    /// Unix timestamp (seconds) of the peer's last activity.
    pub last_seen: u64,
}

### Querying Relay Peers

```rust
impl RelayClient {
    /// Request the list of peers connected to the relay on our network.
    ///
    /// Sends GetPeers and waits for the Peers response. Updates the
    /// internal relay peer list and stats.
    ///
    /// # Errors
    /// Returns `GossipError::RelayError` if the relay connection is broken,
    /// the response times out, or the response is malformed.
    pub async fn get_peers(&mut self) -> Result<Vec<RelayPeerInfo>, GossipError> {
        let msg = RelayMessage::GetPeers {
            network_id: self.network_id,
        };

        let json = serde_json::to_string(&msg)
            .map_err(|e| GossipError::RelayError(format!("serialize error: {e}")))?;

        self.write.send(TungsteniteMessage::Text(json)).await
            .map_err(|e| GossipError::RelayError(format!("send error: {e}")))?;

        // Wait for Peers response (with timeout)
        let response = tokio::time::timeout(
            Duration::from_secs(self.config.connection_timeout_secs),
            self.read_next_message(),
        )
        .await
        .map_err(|_| GossipError::RelayError("get_peers response timeout".into()))?;

        match response? {
            RelayMessage::Peers { peers } => {
                self.relay_peers = peers.clone();
                self.stats.relay_peer_count = peers.len();
                tracing::debug!(peer_count = peers.len(), "relay peer list updated");
                Ok(peers)
            }
            other => Err(GossipError::RelayError(
                format!("unexpected response to GetPeers: {other:?}")
            )),
        }
    }

    /// Get the cached relay peer list (from the last GetPeers call or
    /// from RegisterAck connected_peers updates).
    pub fn cached_relay_peers(&self) -> &[RelayPeerInfo] {
        &self.relay_peers
    }

    /// Check if a specific peer is reachable via the relay.
    pub fn is_peer_on_relay(&self, peer_id: &PeerId) -> bool {
        self.relay_peers.iter().any(|p| p.peer_id == *peer_id)
    }
}
```

### Periodic Peer List Refresh

```rust
/// The relay service periodically refreshes the peer list to stay
/// aware of new relay-connected peers and detect departed ones.
impl RelayService {
    async fn run_message_loop(
        &mut self,
        inbound_tx: &mpsc::Sender<(PeerId, Message)>,
        shutdown: &mut tokio::sync::watch::Receiver<bool>,
    ) -> DisconnectReason {
        let mut peer_refresh_interval = tokio::time::interval(
            Duration::from_secs(60) // Refresh every 60 seconds
        );

        loop {
            tokio::select! {
                // ... handle incoming messages ...

                _ = peer_refresh_interval.tick() => {
                    if let Some(client) = &mut self.client {
                        if let Err(e) = client.get_peers().await {
                            tracing::warn!("relay peer list refresh failed: {e}");
                        }
                    }
                }

                // ... handle shutdown ...
            }
        }
    }
}
```

## Acceptance Criteria

- [ ] `GetPeers { network_id }` is correctly serialized and sent to the relay server
- [ ] `Peers { peers }` response is correctly deserialized into `Vec<RelayPeerInfo>`
- [ ] `RelayPeerInfo` contains `peer_id`, `network_id`, `protocol_version`, `connected_at`, `last_seen`
- [ ] Relay peer list is cached locally after each `get_peers()` call
- [ ] `is_peer_on_relay()` checks the cached peer list
- [ ] `RelayStats::relay_peer_count` is updated after each peer list refresh
- [ ] Timeout on `Peers` response returns `GossipError::RelayError`
- [ ] Unexpected response to `GetPeers` returns `GossipError::RelayError`
- [ ] Peer list is periodically refreshed (every 60 seconds)
- [ ] The response only includes peers on the requested `network_id`

## Implementation Notes

- Primary files: `src/relay/relay_client.rs`, `src/relay/relay_types.rs`
- Dependencies: `serde_json`, `tokio::time`
- Edge cases:
  - The relay server may return an empty peer list if no other peers are connected
  - A peer may appear in the peer list but be unreachable (just disconnected); the gossip layer should handle send failures gracefully
  - The `Peers` response may interleave with other relay messages (e.g., incoming `RelayGossipMessage`); the client must buffer or route non-`Peers` messages appropriately
  - Very large peer lists (1000+ entries) should be handled without excessive memory allocation
