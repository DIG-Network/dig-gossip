# PRF-004: Parallel Bootstrap

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Sections 1.8, 4.4, 6.4](../../../resources/SPEC.md)

## Summary

Bootstrap connection establishment must use `PARALLEL_CONNECT_BATCH_SIZE` (8) concurrent connection attempts via `FuturesUnordered`. Unlike Chia's sequential one-at-a-time approach with `asyncio.sleep()` between attempts, all batch members must be in-flight concurrently. This reduces bootstrap time from approximately `N * connection_timeout` to approximately `1 * connection_timeout` for N connections, enabling a node to reach its target outbound count in seconds rather than minutes.

## Specification

### Parallel Connection Batch

The discovery loop selects up to `PARALLEL_CONNECT_BATCH_SIZE` candidate peers from the address manager and initiates all connections concurrently using `FuturesUnordered`. The system does not wait for one connection attempt to complete before starting the next.

### Implementation

```rust
use futures::stream::{FuturesUnordered, StreamExt};
use std::time::Duration;
use tokio::time::timeout;

/// Connect to multiple peers in parallel during bootstrap.
/// SPEC Section 6.4 item 2: "Select up to PARALLEL_CONNECT_BATCH_SIZE (8)
/// peers from the address manager and connect concurrently using
/// FuturesUnordered."
///
/// Chia connects one at a time with asyncio.sleep() between attempts.
/// This is N times faster for bootstrap.
pub async fn parallel_connect(
    address_manager: &AddressManager,
    connection_manager: &ConnectionManager,
    config: &GossipConfig,
    needed: usize,
) -> Vec<ConnectResult> {
    let batch_size = std::cmp::min(needed, PARALLEL_CONNECT_BATCH_SIZE);

    // 1. Select candidates from address manager (score-aware per PRF-002)
    let mut candidates = Vec::with_capacity(batch_size);
    for _ in 0..batch_size {
        if let Some(peer) = address_manager.select_peer(false).await {
            candidates.push(peer);
        }
    }

    if candidates.is_empty() {
        return Vec::new();
    }

    // 2. Launch all connection attempts concurrently via FuturesUnordered
    let mut futures = FuturesUnordered::new();
    for candidate in &candidates {
        let addr = candidate.peer_info.clone();
        let config = config.clone();
        futures.push(async move {
            let result = timeout(
                Duration::from_secs(config.connection_timeout_secs),
                connect_to_peer(&addr, &config),
            ).await;
            (addr, result)
        });
    }

    // 3. Collect results as they complete (not in submission order)
    let mut results = Vec::with_capacity(candidates.len());
    while let Some((addr, result)) = futures.next().await {
        match result {
            Ok(Ok(connection)) => {
                address_manager.mark_good(&addr).await;
                results.push(ConnectResult::Success(connection));
            }
            Ok(Err(e)) => {
                address_manager.attempt(&addr, true).await;
                results.push(ConnectResult::Failed(addr, e));
            }
            Err(_timeout) => {
                address_manager.attempt(&addr, true).await;
                results.push(ConnectResult::Timeout(addr));
            }
        }
    }

    results
}

/// Result of a single connection attempt within a parallel batch.
pub enum ConnectResult {
    /// Connection established successfully.
    Success(PeerConnection),
    /// Connection failed with an error.
    Failed(PeerInfo, GossipError),
    /// Connection attempt timed out.
    Timeout(PeerInfo),
}
```

### Discovery Loop Integration

```rust
/// Discovery loop main iteration (SPEC Section 6.4).
async fn discovery_loop_iteration(
    address_manager: &AddressManager,
    connection_manager: &ConnectionManager,
    config: &GossipConfig,
) {
    let current_outbound = connection_manager.outbound_count().await;
    let needed = config.target_outbound_count
        .saturating_sub(current_outbound);

    if needed == 0 {
        return;
    }

    // All connections attempted in parallel, not sequentially
    let results = parallel_connect(
        address_manager,
        connection_manager,
        config,
        needed,
    ).await;

    // Process results...
    for result in results {
        match result {
            ConnectResult::Success(conn) => {
                connection_manager.register(conn).await;
            }
            _ => {} // Failures already recorded in address manager
        }
    }
}
```

## Acceptance Criteria

- [ ] `PARALLEL_CONNECT_BATCH_SIZE` constant equals 8
- [ ] Connection attempts are initiated using `FuturesUnordered` (or equivalent concurrent executor)
- [ ] All batch members are in-flight concurrently; no sequential waiting between attempts
- [ ] Batch size is capped at `min(needed, PARALLEL_CONNECT_BATCH_SIZE)`
- [ ] If fewer candidates than batch size are available, all available candidates are used
- [ ] Successful connections call `address_manager.mark_good()`
- [ ] Failed connections call `address_manager.attempt(count_failure=true)`
- [ ] Timed-out connections call `address_manager.attempt(count_failure=true)`
- [ ] Results are collected as they complete (not blocked on slowest)
- [ ] Bootstrap with 8 needed connections creates 8 concurrent attempts, not 8 sequential attempts

## Implementation Notes

- Primary files: `src/discovery/discovery_loop.rs`, `src/connection/connection_manager.rs`
- Dependencies: `futures::stream::FuturesUnordered`, `tokio::time::timeout`, PRF-002 (score-aware `select_peer()`)
- Edge cases:
  - Address manager is empty: `select_peer()` returns `None` for all slots; `candidates` is empty; function returns empty `Vec`
  - All connections fail: all results are `Failed` or `Timeout`; address manager records failures; next iteration retries with different candidates
  - Some connections succeed, some fail: partial success is normal; discovery loop iterates again to fill remaining slots
  - Connection timeout: individual timeouts per connection attempt prevent a single slow peer from blocking the batch
  - Duplicate candidates: `select_peer()` may return the same peer twice if the address manager is small; deduplication should be applied before connecting
  - Rapid successive iterations: if the discovery loop runs again before the previous batch completes, the two batches overlap; this is acceptable as the connection manager rejects duplicate connections
