# PRF-005: Bandwidth Benchmarks

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 11.3](../../../resources/SPEC.md)

## Summary

Bandwidth benchmarks must validate the efficiency gains of the three major gossip optimizations over naive flooding. Using a simulated 50-node network, the benchmarks measure total bytes transferred and verify that: Plumtree uses less than 40% of naive flood bandwidth, compact block relay uses less than 10% of full block bandwidth, and ERLAY transaction relay uses less than 20% of flood relay bandwidth. These thresholds are derived from the theoretical improvement ranges stated in SPEC Section 1.8.

## Specification

### Benchmark 1: Plumtree vs Naive Flood

**Setup:** 50-node simulated network with realistic topology (each node connected to 8 outbound + variable inbound peers). 1000 messages broadcast from random origin nodes.

**Measurement:** Total bytes transferred across all links for all 1000 messages.

**Threshold:** Plumtree total bytes < 40% of naive flood total bytes.

```rust
/// Benchmark: Plumtree bandwidth vs naive flooding.
/// SPEC Section 11.3: "Plumtree < 40% of naive flood."
///
/// Naive flood: each node forwards to ALL connected peers.
/// Total bytes = messages * avg_degree * nodes * msg_size (minus origin)
///
/// Plumtree: each node forwards via eager (tree edges) + lazy (hash-only).
/// Total bytes = tree_edges * msg_size + non_tree_edges * hash_size
#[bench]
fn bench_plumtree_vs_flood_bandwidth() {
    let network = SimulatedNetwork::new(50, 8); // 50 nodes, 8 outbound each
    let message_count = 1000;
    let msg_size_bytes = 512; // typical message size

    // Measure naive flood bandwidth
    let flood_bytes = network.simulate_flood(message_count, msg_size_bytes);

    // Measure Plumtree bandwidth
    let plumtree_bytes = network.simulate_plumtree(message_count, msg_size_bytes);

    let ratio = plumtree_bytes as f64 / flood_bytes as f64;
    assert!(
        ratio < 0.40,
        "Plumtree bandwidth ratio {:.2}% exceeds 40% threshold",
        ratio * 100.0
    );
}
```

### Benchmark 2: Compact Block vs Full Block

**Setup:** 10-hop chain topology (block propagates through 10 relay hops). Block contains 500 transactions, receiver mempool has 95% overlap (475/500 transactions already known).

**Measurement:** Total bytes for block propagation across all 10 hops.

**Threshold:** Compact block total bytes < 10% of full block total bytes.

```rust
/// Benchmark: Compact block relay bandwidth vs full block relay.
/// SPEC Section 11.3: "compact block < 10% bandwidth of full block."
///
/// Full block: header + all serialized transactions (~2MB typical)
/// Compact block: header + 500 * 6-byte short IDs + ~25 prefilled txs
///                + round-trip for ~25 missing txs
#[bench]
fn bench_compact_block_vs_full_block_bandwidth() {
    let hops = 10;
    let tx_count = 500;
    let tx_size_bytes = 4000; // average transaction size
    let header_size_bytes = 200;
    let short_id_bytes = SHORT_TX_ID_BYTES; // 6
    let mempool_overlap = 0.95; // 95% of txs already in receiver's mempool
    let missing_txs = ((1.0 - mempool_overlap) * tx_count as f64) as usize;

    // Full block: header + all transactions, per hop
    let full_block_size = header_size_bytes + (tx_count * tx_size_bytes);
    let full_block_total = full_block_size * hops;

    // Compact block: header + short IDs + prefilled (missing) txs, per hop
    // Plus round-trip for missing tx requests
    let compact_block_size = header_size_bytes
        + (tx_count * short_id_bytes)       // short IDs for all txs
        + (missing_txs * tx_size_bytes)     // prefilled/requested missing txs
        + (missing_txs * 32);               // request overhead (32-byte tx hashes)
    let compact_block_total = compact_block_size * hops;

    let ratio = compact_block_total as f64 / full_block_total as f64;
    assert!(
        ratio < 0.10,
        "Compact block bandwidth ratio {:.2}% exceeds 10% threshold",
        ratio * 100.0
    );
}
```

### Benchmark 3: ERLAY vs Flood Transaction Relay

**Setup:** Single node with 50 connections. 1000 transactions announced. Flood set size = `ERLAY_FLOOD_PEER_COUNT` (8). Reconciliation with remaining 42 peers via minisketch.

**Measurement:** Total bytes per transaction across all 50 connections.

**Threshold:** ERLAY total bytes < 20% of flood total bytes.

```rust
/// Benchmark: ERLAY transaction relay bandwidth vs naive flooding.
/// SPEC Section 11.3: "ERLAY < 20% of flood."
///
/// Flood: send NewTransaction (32-byte txid + overhead) to all 50 peers.
/// ERLAY: send NewTransaction to 8 flood peers + periodic sketch
///        reconciliation with 42 remaining peers.
#[bench]
fn bench_erlay_vs_flood_tx_bandwidth() {
    let total_connections = 50;
    let flood_peer_count = ERLAY_FLOOD_PEER_COUNT; // 8
    let reconciliation_peers = total_connections - flood_peer_count; // 42
    let tx_count = 1000;
    let tx_announce_bytes = 36; // 32-byte txid + 4-byte overhead
    let sketch_capacity = ERLAY_SKETCH_CAPACITY; // 20

    // Flood: every tx announced to every peer
    let flood_total = tx_count * total_connections * tx_announce_bytes;

    // ERLAY: tx announced to flood set + sketch reconciliation with rest
    let erlay_flood_bytes = tx_count * flood_peer_count * tx_announce_bytes;

    // Sketch reconciliation: one sketch per reconciliation round per peer.
    // Sketch size = capacity * 8 bytes (minisketch encoding).
    // Number of rounds = ceil(tx_count / sketch_capacity) per peer.
    let sketch_size = sketch_capacity * 8;
    let rounds_per_peer = (tx_count + sketch_capacity - 1) / sketch_capacity;
    let erlay_recon_bytes = reconciliation_peers * rounds_per_peer * sketch_size * 2; // both sides

    let erlay_total = erlay_flood_bytes + erlay_recon_bytes;

    let ratio = erlay_total as f64 / flood_total as f64;
    assert!(
        ratio < 0.20,
        "ERLAY bandwidth ratio {:.2}% exceeds 20% threshold",
        ratio * 100.0
    );
}
```

## Acceptance Criteria

- [ ] Plumtree vs flood benchmark exists and runs on a simulated 50-node network with 1000 messages
- [ ] Plumtree total bytes is less than 40% of naive flood total bytes
- [ ] Compact block vs full block benchmark exists and runs over 10 hops
- [ ] Compact block total bytes is less than 10% of full block total bytes
- [ ] ERLAY vs flood benchmark exists and measures per-transaction bandwidth across 50 connections
- [ ] ERLAY total bytes is less than 20% of flood total bytes
- [ ] All benchmarks report the actual ratio for diagnostic purposes
- [ ] Benchmarks use realistic message sizes (not trivially small payloads)
- [ ] Benchmarks account for protocol overhead (request/response round-trips for compact blocks, sketch exchange for ERLAY)

## Implementation Notes

- Primary files: `benches/bandwidth_benchmarks.rs` or `tests/prf_005_tests.rs`
- Dependencies: Simulated network harness (may be shared with integration tests), `ERLAY_FLOOD_PEER_COUNT`, `ERLAY_SKETCH_CAPACITY`, `SHORT_TX_ID_BYTES` constants
- Edge cases:
  - Plumtree with poorly formed tree (all lazy, no eager): bandwidth degrades to near-flood levels; benchmark assumes converged tree
  - Compact block with 0% mempool overlap: falls back to full block request; bandwidth ratio approaches 100%; benchmark uses realistic 95% overlap
  - ERLAY with sketch decode failure (symmetric difference exceeds capacity): falls back to full announcement for that round; benchmark assumes normal operation
  - Network topology affects results: benchmarks should use a fixed seed for reproducible topology
  - Message size distribution: benchmarks use fixed sizes; real-world performance may vary with message size variance
