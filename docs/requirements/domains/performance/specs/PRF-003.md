# PRF-003: Plumtree Tree Optimization

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Sections 1.8, 2.5, 8.1](../../../resources/SPEC.md)

## Summary

The Plumtree spanning tree must prefer low-latency peers as eager push targets. When a lower-latency peer is discovered (either through a new RTT measurement or through a GRAFT promotion from lazy), it should replace a higher-latency eager peer. The replaced peer is demoted to lazy and sent a PRUNE message. This optimization reduces message propagation latency across the gossip tree by routing full messages through the fastest links.

## Specification

### Tree Optimization Trigger

Tree optimization is triggered in two scenarios:

1. **New RTT measurement**: After `PeerReputation.record_rtt()` updates a peer's `avg_rtt_ms`, if that peer is currently lazy and has lower latency than the highest-latency eager peer, a swap is performed.
2. **GRAFT promotion**: When a lazy peer is promoted to eager via GRAFT (because a message was missed), the optimization check runs to determine whether the newly eager peer should displace an existing eager peer.

### Optimization Algorithm

```rust
use std::collections::HashSet;

impl PlumtreeState {
    /// Optimize the eager/lazy peer sets based on latency.
    /// Called after RTT updates or GRAFT promotions.
    ///
    /// SPEC Section 1.8 #6: "The Plumtree spanning tree is also optimized
    /// to prefer low-latency links for eager push."
    pub fn optimize_tree(
        &mut self,
        peer_reputations: &HashMap<PeerId, PeerReputation>,
    ) -> Vec<TreeOptimizationAction> {
        let mut actions = Vec::new();

        // Find the eager peer with the highest (worst) avg_rtt_ms
        let worst_eager = self.eager_peers.iter()
            .filter_map(|id| {
                peer_reputations.get(id)
                    .and_then(|rep| rep.avg_rtt_ms.map(|rtt| (id.clone(), rtt)))
            })
            .max_by_key(|(_, rtt)| *rtt);

        let (worst_eager_id, worst_eager_rtt) = match worst_eager {
            Some(w) => w,
            None => return actions, // No eager peers with RTT data
        };

        // Find the lazy peer with the lowest (best) avg_rtt_ms
        let best_lazy = self.lazy_peers.iter()
            .filter_map(|id| {
                peer_reputations.get(id)
                    .and_then(|rep| rep.avg_rtt_ms.map(|rtt| (id.clone(), rtt)))
            })
            .min_by_key(|(_, rtt)| *rtt);

        let (best_lazy_id, best_lazy_rtt) = match best_lazy {
            Some(b) => b,
            None => return actions, // No lazy peers with RTT data
        };

        // Swap if the best lazy peer is strictly faster than the worst eager peer
        if best_lazy_rtt < worst_eager_rtt {
            // Demote worst eager to lazy
            self.eager_peers.remove(&worst_eager_id);
            self.lazy_peers.insert(worst_eager_id.clone());

            // Promote best lazy to eager
            self.lazy_peers.remove(&best_lazy_id);
            self.eager_peers.insert(best_lazy_id.clone());

            actions.push(TreeOptimizationAction::Prune {
                peer_id: worst_eager_id,
            });
            actions.push(TreeOptimizationAction::Graft {
                peer_id: best_lazy_id,
            });
        }

        actions
    }
}

/// Actions to execute after tree optimization.
pub enum TreeOptimizationAction {
    /// Send PRUNE to this peer (demoted from eager to lazy).
    Prune { peer_id: PeerId },
    /// Send GRAFT to this peer (promoted from lazy to eager).
    Graft { peer_id: PeerId },
}
```

### Integration with RTT Updates

```rust
/// Called after each Pong is processed and RTT is recorded.
async fn on_rtt_updated(
    plumtree: &mut PlumtreeState,
    peer_reputations: &HashMap<PeerId, PeerReputation>,
    connections: &ConnectionManager,
) {
    let actions = plumtree.optimize_tree(peer_reputations);
    for action in actions {
        match action {
            TreeOptimizationAction::Prune { peer_id } => {
                // Send PRUNE message to demoted peer
                connections.send_prune(&peer_id).await;
            }
            TreeOptimizationAction::Graft { peer_id } => {
                // Send GRAFT message to promoted peer
                connections.send_graft(&peer_id, None).await;
            }
        }
    }
}
```

## Acceptance Criteria

- [ ] When a lazy peer has lower `avg_rtt_ms` than the highest-RTT eager peer, a swap occurs
- [ ] The swapped-out eager peer is moved to `lazy_peers` and sent a PRUNE message
- [ ] The swapped-in lazy peer is moved to `eager_peers` and sent a GRAFT message
- [ ] No swap occurs when all lazy peers have higher or equal RTT than all eager peers
- [ ] No swap occurs when eager or lazy peers have no RTT measurements (`avg_rtt_ms == None`)
- [ ] Tree optimization runs after each `record_rtt()` update
- [ ] Tree optimization runs after a GRAFT promotion
- [ ] The eager peer set size remains unchanged after optimization (one in, one out)
- [ ] Multiple rapid RTT updates do not cause oscillation (swap only if strictly better)

## Implementation Notes

- Primary files: `src/gossip/plumtree.rs`, `src/util/latency.rs`
- Dependencies: PRF-001 (RTT data in PeerReputation), `PeerId` from `types::peer`
- Edge cases:
  - Only one eager peer: it can still be swapped out if a lazy peer is faster
  - Eager and lazy peer have identical RTT: no swap (strictly less-than comparison prevents oscillation)
  - Peer disconnects during optimization: the connection manager handles the send failure; the set updates are still applied locally
  - All peers are eager (no lazy peers): optimization is a no-op since there are no candidates to promote
  - New peer added directly to eager (initial Plumtree behavior): optimization runs on next RTT update, may demote if a lazy peer is faster
