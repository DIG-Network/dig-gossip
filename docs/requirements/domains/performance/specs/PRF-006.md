# PRF-006: Latency Benchmarks

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 11.3](../../../resources/SPEC.md)

## Summary

Latency benchmarks must validate that consensus-critical message delivery remains fast under load and that parallel bootstrap meets its time target. Specifically: NewPeak message delivery p99 latency must stay below 50ms during concurrent bulk sync (RespondBlocks transfer), and bootstrap of 8 outbound connections must complete in under 15 seconds using parallel establishment (PRF-004). These benchmarks verify that priority lanes and parallel connection establishment deliver their promised latency improvements.

## Specification

### Benchmark 1: NewPeak p99 During Bulk Sync

**Setup:** Node is actively transferring bulk data via `RespondBlocks` (simulating historical block sync). Concurrently, `NewPeak` messages are injected at the gossip layer representing new chain tip announcements.

**Measurement:** p99 latency of `NewPeak` message delivery (time from send to receive at the next hop) across 1000 NewPeak samples during active bulk sync.

**Threshold:** p99 < 50ms.

**Rationale:** Priority lanes (SPEC Section 1.8 #4) ensure consensus-critical messages (`NewPeak`) are sent ahead of bulk data (`RespondBlocks`). Without priority lanes, a multi-megabyte `RespondBlocks` response would block the outbound queue, delaying `NewPeak` by the full transfer time.

```rust
use std::time::{Duration, Instant};

/// Benchmark: NewPeak p99 latency during concurrent bulk sync.
/// SPEC Section 11.3: "NewPeak delivery latency during concurrent
/// RespondBlocks transfer. Target: < 50ms p99."
#[bench]
fn bench_newpeak_p99_during_bulk_sync() {
    let rt = tokio::runtime::Runtime::new().unwrap();
    rt.block_on(async {
        // Set up two connected peers with priority lanes enabled
        let (node_a, node_b) = create_connected_pair().await;

        // Start bulk sync: continuously send RespondBlocks (large payloads)
        let bulk_sync_handle = tokio::spawn({
            let sender = node_a.clone();
            async move {
                loop {
                    let bulk_payload = vec![0u8; 2_000_000]; // 2MB block
                    sender.send_bulk(RespondBlocks { blocks: bulk_payload }).await;
                    tokio::task::yield_now().await;
                }
            }
        });

        // Inject NewPeak messages and measure delivery latency
        let sample_count = 1000;
        let mut latencies = Vec::with_capacity(sample_count);

        for _ in 0..sample_count {
            let start = Instant::now();
            node_a.send_critical(NewPeak {
                header_hash: Bytes32::default(),
                height: 1,
                weight: 1,
                fork_point_with_previous_peak: 0,
                unfinished_reward_block_hash: Bytes32::default(),
            }).await;

            // Wait for node_b to receive the NewPeak
            node_b.recv_newpeak().await;
            latencies.push(start.elapsed());
        }

        bulk_sync_handle.abort();

        // Compute p99
        latencies.sort();
        let p99_index = (sample_count as f64 * 0.99) as usize;
        let p99 = latencies[p99_index];

        assert!(
            p99 < Duration::from_millis(50),
            "NewPeak p99 latency {:?} exceeds 50ms threshold during bulk sync",
            p99
        );
    });
}
```

### Benchmark 2: Bootstrap 8 Connections

**Setup:** Fresh node with empty peer state. Address manager seeded with 8+ reachable peer addresses (mock peers that accept connections). Parallel bootstrap enabled (PRF-004).

**Measurement:** Wall-clock time from bootstrap initiation to 8 established outbound connections.

**Threshold:** < 15 seconds.

**Rationale:** Chia's sequential bootstrap connects one peer at a time with multi-second gaps, taking approximately `N * (connection_time + sleep_interval)`. Parallel bootstrap (SPEC Section 1.8 #5) reduces this to approximately `max(connection_time)` for N peers. With `PARALLEL_CONNECT_BATCH_SIZE = 8`, all 8 connections are attempted concurrently.

```rust
/// Benchmark: Bootstrap wall-clock time for 8 outbound connections.
/// SPEC Section 11.3: "Bootstrap time: time to establish 8 outbound
/// connections (parallel vs sequential). Target: < 15 seconds."
#[bench]
fn bench_bootstrap_8_connections() {
    let rt = tokio::runtime::Runtime::new().unwrap();
    rt.block_on(async {
        // Start 8 mock peers that accept connections with realistic latency
        let mock_peers = start_mock_peers(8, Duration::from_millis(500)).await;

        // Create a fresh node with address manager seeded with mock peer addresses
        let config = GossipConfig {
            target_outbound_count: 8,
            ..Default::default()
        };
        let address_manager = AddressManager::create_in_memory().await;
        for peer in &mock_peers {
            address_manager.add_to_new_table(
                &[peer.timestamped_info()],
                &peer.source_info(),
                0,
            ).await;
        }

        // Measure bootstrap time
        let start = Instant::now();

        let connection_manager = ConnectionManager::new(config.clone());
        let results = parallel_connect(
            &address_manager,
            &connection_manager,
            &config,
            8,
        ).await;

        let elapsed = start.elapsed();

        // Verify all 8 connected
        let success_count = results.iter()
            .filter(|r| matches!(r, ConnectResult::Success(_)))
            .count();
        assert_eq!(success_count, 8, "Expected 8 successful connections, got {}", success_count);

        // Verify wall-clock time
        assert!(
            elapsed < Duration::from_secs(15),
            "Bootstrap took {:?}, exceeds 15s threshold",
            elapsed
        );

        // Cleanup
        for peer in mock_peers {
            peer.shutdown().await;
        }
    });
}
```

### Comparison Benchmark: Sequential vs Parallel

```rust
/// Comparison benchmark showing parallel speedup over sequential.
/// This is informational, not a pass/fail threshold.
#[bench]
fn bench_bootstrap_sequential_vs_parallel() {
    let rt = tokio::runtime::Runtime::new().unwrap();
    rt.block_on(async {
        let mock_peers = start_mock_peers(8, Duration::from_millis(500)).await;

        // Sequential: connect one at a time
        let seq_start = Instant::now();
        for peer in &mock_peers {
            connect_to_peer(&peer.addr(), &GossipConfig::default()).await.unwrap();
        }
        let seq_elapsed = seq_start.elapsed();

        // Parallel: connect all at once (PRF-004)
        let par_start = Instant::now();
        let _ = parallel_connect(/* ... */).await;
        let par_elapsed = par_start.elapsed();

        // Report speedup
        let speedup = seq_elapsed.as_millis() as f64 / par_elapsed.as_millis() as f64;
        println!(
            "Sequential: {:?}, Parallel: {:?}, Speedup: {:.1}x",
            seq_elapsed, par_elapsed, speedup
        );

        // Parallel should be at least 2x faster
        assert!(
            speedup > 2.0,
            "Parallel speedup {:.1}x is below 2x minimum",
            speedup
        );
    });
}
```

## Acceptance Criteria

- [ ] Bootstrap benchmark exists and measures wall-clock time for 8 connections

## Implementation Notes

- Primary files: `benches/latency_benchmarks.rs` or `tests/prf_006_tests.rs`
- Dependencies: Priority lanes implementation (for NewPeak prioritization), PRF-004 (parallel bootstrap), mock peer infrastructure
- Edge cases:
  - NewPeak benchmark with no bulk sync running: latency is trivially low; benchmark must verify bulk sync is actively transferring during measurement
  - Bootstrap with slow peers: individual connection timeouts prevent infinite waits; the 15s threshold accommodates realistic network delays
  - System under CPU load: p99 benchmark is sensitive to scheduling; benchmark should run on a quiescent system or use margin
  - Mock peer latency: set to 500ms to simulate realistic handshake time; too low makes the benchmark trivially pass
  - GC pauses or async runtime contention: may cause sporadic latency spikes; p99 (not max) is used for robustness
  - Network jitter simulation: benchmarks use mock peers with fixed latency; production latency will vary more
