# PRF-002: Peer Selection Preference

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Sections 1.8, 6.4](../../../resources/SPEC.md)

## Summary

Outbound peer selection must prefer peers with higher composite scores (as computed by PRF-001). The `AddressManager.select_peer()` method must incorporate the peer score when choosing among candidate peers that have passed group/AS diversity filters. When multiple candidate peers are available, the one with the highest score is preferred. This ensures the node builds outbound connections through lower-latency, higher-trust paths, improving block and attestation propagation times.

## Specification

### Score-Aware Peer Selection

The discovery loop calls `AddressManager.select_peer()` when it needs a new outbound connection target. The existing address manager selects peers based on bucket position and recency (ported from Chia's `address_manager.py`). This requirement extends that selection to incorporate the composite peer score.

### Selection Algorithm

```rust
impl AddressManager {
    /// Select a peer for outbound connection.
    /// Filters by group/AS diversity, then prefers higher-scored peers.
    ///
    /// SPEC Section 6.4 item 6: "When multiple candidate peers pass the
    /// group/AS filter, prefer the one with the lowest average RTT from
    /// the peer scorer."
    pub async fn select_peer(&self, new_only: bool) -> Option<ExtendedPeerInfo> {
        // 1. Collect candidate peers from tried/new tables
        //    (existing Chia address manager logic)
        let candidates = self.get_candidates(new_only).await;

        // 2. Filter by /16 group diversity (fast filter)
        let candidates = self.filter_group_diversity(candidates).await;

        // 3. Filter by AS-level diversity
        let candidates = self.filter_as_diversity(candidates).await;

        if candidates.is_empty() {
            return None;
        }

        // 4. Among remaining candidates, prefer highest score
        //    Score is PeerReputation.score (trust_score * (1/avg_rtt_ms))
        //    Peers with score == 0.0 (no RTT data) are treated as lowest priority
        //    but still selectable if no scored peers are available.
        let scored: Vec<_> = candidates.iter()
            .filter(|c| c.reputation_score > 0.0)
            .collect();

        if scored.is_empty() {
            // No scored candidates; fall back to random selection
            // (matches Chia's default behavior for new/unmeasured peers)
            let idx = rand::random::<usize>() % candidates.len();
            return Some(candidates[idx].clone());
        }

        // Select the candidate with the highest composite score
        let best = scored.iter()
            .max_by(|a, b| a.reputation_score
                .partial_cmp(&b.reputation_score)
                .unwrap_or(std::cmp::Ordering::Equal))
            .unwrap();

        Some((*best).clone())
    }
}
```

### Discovery Loop Integration

```rust
/// Discovery loop (SPEC Section 6.4) uses select_peer() with score preference.
async fn connect_to_peers(
    address_manager: &AddressManager,
    peer_scorer: &PeerScorer,
    target_outbound: usize,
    current_outbound: usize,
) {
    let needed = target_outbound.saturating_sub(current_outbound);
    if needed == 0 {
        return;
    }

    // Select up to PARALLEL_CONNECT_BATCH_SIZE candidates
    // select_peer() already incorporates score preference
    let mut candidates = Vec::new();
    for _ in 0..std::cmp::min(needed, PARALLEL_CONNECT_BATCH_SIZE) {
        if let Some(peer) = address_manager.select_peer(false).await {
            candidates.push(peer);
        }
    }

    // Connect in parallel (see PRF-004)
    // ...
}
```

## Acceptance Criteria

- [ ] `AddressManager.select_peer()` returns the highest-scored candidate when multiple pass diversity filters
- [ ] Peers with `score > 0.0` are preferred over peers with `score == 0.0`
- [ ] When all candidates have `score == 0.0` (no RTT data), random selection is used as fallback
- [ ] Group/AS diversity filtering is applied before score-based selection
- [ ] Score comparison uses `PeerReputation.score` field (composite score from PRF-001)
- [ ] A peer with lower RTT (higher score) is selected over a peer with higher RTT (lower score), given equal trust
- [ ] A peer with higher trust (higher score) is selected over a peer with lower trust (lower score), given equal RTT
- [ ] `select_peer()` still returns `None` when no candidates are available

## Implementation Notes

- Primary files: `src/discovery/address_manager.rs`, `src/discovery/discovery_loop.rs`
- Dependencies: PRF-001 (composite score must be populated in `PeerReputation`)
- Edge cases:
  - All candidates have the same score: any may be returned (tie-breaking is implementation-defined)
  - Score is `f64::NAN`: `partial_cmp` returns `None`; the `unwrap_or(Ordering::Equal)` handles this safely
  - Candidate pool has only one peer: that peer is returned regardless of score
  - A previously high-scored peer whose RTT degrades: score decreases naturally on next `record_rtt()`, subsequent `select_peer()` calls reflect the change
  - New peers (never connected): `score == 0.0`, selected only when no scored alternatives exist
