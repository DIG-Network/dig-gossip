# INT-014: Crate-Level Documentation with Usage Example

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Section 1

## Summary

The crate root (`lib.rs`) MUST have a comprehensive `//!` doc comment showing:
1. What the crate does (one paragraph)
2. The core lifecycle: GossipConfig → GossipService::new() → start() → GossipHandle
3. Input/output contract: Message payloads in, (PeerId, Message) out
4. Feature flags and what they enable
5. Code example showing the complete lifecycle

## Acceptance Criteria

- [ ] `//!` doc on lib.rs includes lifecycle overview
- [ ] Code example compiles (doc-test or `# ignore`)
- [ ] Inputs clearly documented: GossipConfig, Message payloads via broadcast()/send_to()
- [ ] Outputs clearly documented: (PeerId, Message) via inbound_receiver()
- [ ] Feature flag table in docs
- [ ] Hard boundary stated: "does NOT validate blocks, run CLVM, manage mempool"
