# PRV-008: Rotation Opt-Out

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 1.9.2](../../../resources/SPEC.md)

## Summary

Nodes that require a stable, long-lived identity (such as bootstrap nodes, well-known seed nodes, or introducer nodes) can disable PeerId rotation by setting `rotation_interval_secs = 0`. When rotation is disabled, the existing `chia-ssl` certificate is loaded via `load_ssl_cert()` and kept permanently. No rotation task is spawned, and the PeerId remains stable across restarts as long as the same certificate files are used.

## Specification

### Opt-Out Mechanism

Setting `rotation_interval_secs = 0` in `PeerIdRotationConfig` MUST disable rotation entirely:

```rust
impl PeerIdRotationConfig {
    /// Returns true if rotation is effectively disabled.
    ///
    /// Rotation is disabled when:
    /// - `enabled` is false, OR
    /// - `rotation_interval_secs` is 0
    pub fn is_rotation_disabled(&self) -> bool {
        !self.enabled || self.rotation_interval_secs == 0
    }
}
```

### Startup Behavior When Rotation Is Disabled

When rotation is disabled, the node MUST:

1. Load the existing TLS certificate via `chia-ssl`'s `load_ssl_cert()` (or `ChiaCertificate::generate()` on first run if no certificate file exists)
2. Use that certificate for the lifetime of the process
3. NOT spawn the rotation background task
4. NOT generate any new certificates during runtime

```rust
use chia_ssl::ChiaCertificate;
use chia_sdk_client::load_ssl_cert;

/// Initialize TLS identity based on rotation configuration.
///
/// If rotation is disabled, loads the existing certificate (or generates one
/// on first run) and does NOT spawn a rotation task.
/// If rotation is enabled, generates a fresh certificate and spawns the
/// rotation loop.
pub async fn initialize_identity(
    config: &PeerIdRotationConfig,
    cert_path: &Path,
    key_path: &Path,
) -> Result<ChiaCertificate, IdentityError> {
    if config.is_rotation_disabled() {
        // Load existing certificate or generate on first run
        match load_ssl_cert(cert_path, key_path) {
            Ok(cert) => {
                tracing::info!("PeerId rotation disabled; using persistent certificate");
                Ok(cert)
            }
            Err(_) => {
                tracing::info!("No existing certificate found; generating initial certificate");
                let cert = ChiaCertificate::generate()?;
                // Save to disk for persistence across restarts
                cert.save(cert_path, key_path)?;
                Ok(cert)
            }
        }
    } else {
        // Rotation enabled: generate fresh certificate
        let cert = ChiaCertificate::generate()?;
        tracing::info!("PeerId rotation enabled; generated fresh certificate");
        Ok(cert)
    }
}
```

### Use Cases for Opt-Out

| Node Type | Reason for Stable Identity |
|-----------|---------------------------|
| Bootstrap nodes | Must be discoverable at a known identity; rotation would confuse initial peer discovery |
| Well-known seed nodes | Operators publish PeerId alongside IP for out-of-band verification |
| Introducer nodes | Clients expect a stable identity when registering/querying the introducer |
| Monitoring nodes | Network health dashboards track nodes by PeerId over time |

### Certificate Persistence

When rotation is disabled, the certificate MUST be persisted to disk:
- Certificate and key files are saved on first generation
- On subsequent restarts, the same files are loaded, yielding the same PeerId
- The PeerId is stable across process restarts as long as the certificate files are unchanged

## Acceptance Criteria

- [ ] `rotation_interval_secs = 0` disables rotation entirely
- [ ] `is_rotation_disabled()` returns `true` when `rotation_interval_secs = 0`
- [ ] `is_rotation_disabled()` returns `true` when `enabled = false`
- [ ] No rotation background task is spawned when rotation is disabled
- [ ] Existing certificate is loaded via `load_ssl_cert()` when rotation is disabled
- [ ] Certificate is unchanged after extended runtime (hours) when rotation is disabled
- [ ] PeerId is stable across process restarts when using the same certificate files
- [ ] First-run generates and saves a certificate when no existing cert files are found
- [ ] Rotation-disabled nodes function normally for all other networking operations

## Implementation Notes

- Primary file: `src/privacy/peer_id_rotation.rs`
- Dependencies:
  - `chia-ssl` (`ChiaCertificate::generate()`, certificate save/load)
  - `chia-sdk-client` (`load_ssl_cert()`)
- Edge cases:
  - First run with no existing certificate: must generate and persist one, even with rotation disabled
  - Certificate file corruption: should fail loudly at startup rather than silently generating a new identity
  - Changing from rotation-enabled to rotation-disabled (config change across restarts): the last-generated certificate becomes the permanent one
  - `enabled = false` with `rotation_interval_secs > 0` should also disable rotation (both conditions checked)
