# PRV-010: Tor Transport Implementation

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 1.9.3](../../../resources/SPEC.md)

## Summary

This spec defines the Tor transport layer implementation for `dig-gossip`. Outbound connections are routed through a SOCKS5 proxy (the local Tor daemon) so destination peers see only the Tor exit node's IP. Inbound connections are received via a `.onion` hidden service address published to the introducer, allowing peers to reach the node without knowing its real IP. Hybrid mode allows both direct P2P and Tor connections simultaneously. Transport selection follows a priority order based on the `prefer_tor` configuration. Mutual TLS (mTLS) via `chia-ssl` certificates is still required over Tor tunnels, ensuring the same authentication guarantees as direct connections.

## Specification

### Outbound Connections via SOCKS5

Outbound Tor connections MUST route WebSocket traffic through the configured SOCKS5 proxy using `tokio-socks`:

```rust
#[cfg(feature = "tor")]
use tokio_socks::tcp::Socks5Stream;
use tokio_tungstenite::client_async_tls;

/// Establish an outbound connection through the Tor SOCKS5 proxy.
///
/// Routes the TCP connection through the Tor daemon, then upgrades to
/// WebSocket with mTLS. The destination peer sees the Tor exit node's IP,
/// not this node's real IP.
#[cfg(feature = "tor")]
pub async fn connect_via_tor(
    tor_config: &TorConfig,
    target_addr: &str,
    tls_connector: &TlsConnector,
) -> Result<TorPeerConnection, TransportError> {
    // Step 1: Establish SOCKS5 connection through Tor
    let socks_stream = Socks5Stream::connect(
        &tor_config.socks5_proxy as &str,
        target_addr,
    )
    .await
    .map_err(TransportError::Socks5)?;

    let tcp_stream = socks_stream.into_inner();

    // Step 2: Apply mTLS over the Tor tunnel
    // chia-ssl certificate is presented through the SOCKS5 tunnel
    let tls_stream = tls_connector
        .connect(target_addr, tcp_stream)
        .await
        .map_err(TransportError::Tls)?;

    // Step 3: Upgrade to WebSocket over TLS over Tor
    let (ws_stream, _response) = client_async_tls(
        format!("wss://{}/ws", target_addr),
        tls_stream,
    )
    .await
    .map_err(TransportError::WebSocket)?;

    Ok(TorPeerConnection::new(ws_stream))
}
```

The destination peer sees the Tor exit node's IP address, NOT the connecting node's real IP address.

### Inbound Connections via .onion Hidden Service

For inbound Tor connections:

1. The node MUST publish its `.onion` address to the introducer during registration
2. Peers connect to the `.onion` address through Tor, reaching the node without knowing its real IP
3. The node's Tor hidden service forwards incoming connections to the local listen address

```rust
#[cfg(feature = "tor")]
/// Register the node's .onion address with the introducer.
///
/// This allows peers to discover and connect to this node via Tor
/// without knowing the node's real IP address.
pub async fn register_onion_with_introducer(
    tor_config: &TorConfig,
    introducer_client: &IntroducerClient,
) -> Result<(), IntroducerError> {
    if let Some(ref onion_addr) = tor_config.onion_address {
        introducer_client
            .register_address(onion_addr)
            .await?;
        tracing::info!(
            onion_address = %onion_addr,
            "Registered .onion address with introducer"
        );
    }
    Ok(())
}
```

### Hybrid Mode

Hybrid mode allows simultaneous acceptance of both direct P2P and Tor connections:

- **Direct P2P**: mTLS WebSocket connections on the configured `listen_addr` (e.g., `0.0.0.0:8444`)
- **Tor inbound**: Connections arriving via the `.onion` hidden service, forwarded to the same local listener

Both connection types MUST go through the same mTLS handshake and message processing pipeline. The only difference is the transport layer beneath TLS.

```rust
#[cfg(feature = "tor")]
/// Determines whether a node is running in hybrid mode.
///
/// Hybrid mode means the node accepts both direct P2P connections
/// (on listen_addr) and Tor connections (via .onion hidden service).
pub fn is_hybrid_mode(tor_config: &TorConfig) -> bool {
    tor_config.enabled && tor_config.onion_address.is_some()
}
```

### Transport Selection Order

The transport layer MUST select the connection method based on the following priority:

```
┌─────────────────────────────────────────────────────┐
│              Transport Selection Logic               │
├─────────────────────────────────────────────────────┤
│                                                     │
│  Is target a .onion address?                        │
│  ├─ YES → Always use Tor (SOCKS5)                   │
│  └─ NO ↓                                            │
│                                                     │
│  Is prefer_tor = true?                              │
│  ├─ YES → Use Tor (SOCKS5) for all outbound         │
│  └─ NO ↓                                            │
│                                                     │
│  prefer_tor = false (default):                      │
│  1. Try direct P2P connection (mTLS WebSocket)      │
│  2. If direct fails → try relay connection           │
│  3. If relay fails → try Tor (SOCKS5)               │
│                                                     │
└─────────────────────────────────────────────────────┘
```

```rust
#[cfg(feature = "tor")]
/// Select the appropriate transport for connecting to a peer.
pub fn select_transport(
    peer_addr: &str,
    tor_config: &TorConfig,
) -> TransportMethod {
    // .onion addresses always go through Tor
    if peer_addr.ends_with(".onion") {
        return TransportMethod::Tor;
    }

    // If Tor is configured and preferred, use it for everything
    if tor_config.is_configured() && tor_config.prefer_tor {
        return TransportMethod::Tor;
    }

    // Default fallback order: Direct → Relay → Tor
    TransportMethod::DirectWithFallback
}

/// Transport method for peer connections.
#[cfg(feature = "tor")]
pub enum TransportMethod {
    /// Direct P2P mTLS WebSocket connection.
    Direct,
    /// Connection routed through a relay server.
    Relay,
    /// Connection routed through Tor SOCKS5 proxy.
    Tor,
    /// Try direct first, then relay, then Tor.
    DirectWithFallback,
}
```

### Latency Tradeoff

Tor adds significant latency that affects peer scoring and gossip tree placement:

| Metric | Direct P2P | Tor |
|--------|-----------|-----|
| Typical RTT | 10-100ms | 200-1000ms |
| Peer score impact | Normal RTT scoring (PRF-001) | Lower scores due to higher RTT |
| Plumtree role | Eligible for eager peer (PRF-003) | Unlikely to be eager peer; likely lazy peer |
| Bandwidth | Full line rate | Limited by Tor circuit bandwidth |

- Tor peers will naturally receive lower peer scores due to higher RTT in latency-aware scoring (PRF-001)
- Tor peers are unlikely to be selected as Plumtree eager peers (PRF-003) because eager peer selection favors low-latency peers
- This is a deliberate tradeoff: privacy comes at the cost of propagation speed

### mTLS Over Tor

Mutual TLS (chia-ssl certificates) MUST still be required for connections over Tor:

- The chia-ssl certificate is presented through the Tor tunnel during the TLS handshake
- The same certificate validation applies as for direct connections
- PeerId is still derived from the TLS certificate's public key, even over Tor
- If PeerId rotation (PRV-007) is enabled, rotated certificates are used for Tor connections as well

## Acceptance Criteria

- [ ] Outbound connections are routed through the SOCKS5 proxy when Tor is selected
- [ ] Destination peer sees Tor exit node IP, not the connecting node's real IP
- [ ] `.onion` address is registered with the introducer when configured
- [ ] Peers can connect to the node via its `.onion` address through Tor
- [ ] Hybrid mode accepts both direct P2P connections and Tor connections simultaneously
- [ ] `prefer_tor = true` forces all outbound connections through Tor
- [ ] `prefer_tor = false` follows fallback order: direct first, then relay, then Tor
- [ ] `.onion` target addresses always route through Tor regardless of `prefer_tor` setting
- [ ] mTLS (chia-ssl certificate) works correctly over Tor tunnel
- [ ] TLS handshake completes successfully through SOCKS5 proxy
- [ ] Transport selection correctly identifies `.onion` addresses
- [ ] Tor transport integrates with PeerId rotation (PRV-007) — rotated certs used over Tor
- [ ] All Tor transport code is gated behind `#[cfg(feature = "tor")]`

## Implementation Notes

- Primary file: `src/privacy/tor.rs`
- Dependencies:
  - `tokio-socks` (SOCKS5 proxy client, behind `tor` feature gate)
  - `tokio-tungstenite` (WebSocket upgrade over TLS over Tor)
  - `chia-ssl` (mTLS certificates presented through Tor tunnel)
  - `chia-sdk-client` (TLS connector creation)
  - `tracing` (structured logging)
- Edge cases:
  - Tor daemon not running: SOCKS5 connection fails with a clear error; fall back to direct/relay if `prefer_tor = false`
  - Tor daemon restarts: existing Tor connections may drop; reconnection logic should handle this gracefully
  - `.onion` address misconfiguration: invalid format should be caught during registration with introducer
  - SOCKS5 authentication: the default Tor SOCKS5 port typically does not require authentication, but the implementation should support it if needed
  - DNS resolution: when connecting via Tor, DNS resolution MUST happen through Tor (not locally) to prevent DNS leaks. `tokio-socks` handles this by passing the hostname to the SOCKS5 proxy.
  - Concurrent Tor and direct connections to the same peer: should be avoided; only one connection per peer address
  - Circuit reuse: Tor may reuse circuits; this is controlled by the Tor daemon, not by dig-gossip
