# PRV-006: PeerIdRotationConfig Struct

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 1.9.2](../../../resources/SPEC.md)

## Summary

The `PeerIdRotationConfig` struct defines the configuration for ephemeral PeerId rotation. Chia's PeerId is derived from a permanent TLS certificate, enabling long-term tracking of nodes across sessions. DIG gossip breaks this linkability by periodically rotating TLS certificates, and this struct controls the rotation behavior: whether it is enabled, how often rotation occurs, and whether peers are reconnected after rotation.

## Specification

### PeerIdRotationConfig Struct Definition

The `PeerIdRotationConfig` struct MUST be defined with the following fields:

```rust
use serde::{Deserialize, Serialize};

/// Ephemeral PeerId rotation configuration.
///
/// Controls periodic regeneration of the node's TLS certificate to prevent
/// long-term identity tracking. The PeerId (SHA-256 hash of the TLS public key)
/// changes each time a new certificate is generated.
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct PeerIdRotationConfig {
    /// Enable periodic PeerId rotation.
    /// Default: true.
    pub enabled: bool,

    /// Rotation interval in seconds.
    /// Default: 86400 (24 hours). Set to 0 to disable rotation entirely.
    pub rotation_interval_secs: u64,

    /// Whether to reconnect to all peers after rotation.
    /// Default: true. If false, only new connections use the new identity;
    /// existing connections retain the old PeerId until they naturally close.
    pub reconnect_on_rotation: bool,
}
```

### Default Implementation

The `Default` impl MUST produce values matching the SPEC defaults:

```rust
impl Default for PeerIdRotationConfig {
    fn default() -> Self {
        Self {
            enabled: true,
            rotation_interval_secs: 86400, // 24 hours
            reconnect_on_rotation: true,
        }
    }
}
```

### Rotation Disabled Semantics

When `rotation_interval_secs` is set to `0`, rotation MUST be treated as disabled regardless of the `enabled` field. This provides a direct opt-out mechanism for bootstrap nodes and well-known seed nodes that require a stable identity.

## Acceptance Criteria

- [ ] `PeerIdRotationConfig` struct is defined with fields `enabled`, `rotation_interval_secs`, `reconnect_on_rotation`
- [ ] `enabled` field is `bool` with default `true`
- [ ] `rotation_interval_secs` field is `u64` with default `86400`
- [ ] `reconnect_on_rotation` field is `bool` with default `true`
- [ ] `Default` impl produces `enabled: true, rotation_interval_secs: 86400, reconnect_on_rotation: true`
- [ ] Struct derives `Debug`, `Clone`, `Serialize`, `Deserialize`
- [ ] `rotation_interval_secs = 0` is treated as rotation disabled
- [ ] Struct can be instantiated with custom values
- [ ] Struct can be serialized to and deserialized from JSON/bincode

## Implementation Notes

- Primary file: `src/privacy/peer_id_rotation.rs`
- Dependencies: `serde` (for serialization)
- Edge cases:
  - `rotation_interval_secs = 0` disables rotation even if `enabled = true` (zero interval is nonsensical)
  - Very small `rotation_interval_secs` values (e.g., 1 second) are technically valid but impractical; no minimum enforcement is required at the config level
  - The struct is configuration only; it does not perform rotation itself (see PRV-007 for rotation logic)
