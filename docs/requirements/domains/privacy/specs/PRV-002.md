# PRV-002: Stem Phase Forwarding

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 1.9.1](../../../resources/SPEC.md)

## Summary

When a node originates a new transaction or receives a `StemTransaction` from another node during the stem phase, it MUST forward the transaction to exactly one peer — the current epoch's stem relay. The transaction MUST NOT be added to the local mempool during the stem phase, and the node MUST NOT respond to `RequestTransaction` queries for transactions it is only stemming. This ensures that nodes on the stem path are indistinguishable from non-participants, preserving transaction origin privacy.

## Specification

### StemTransaction Type

The `StemTransaction` message type wraps an inner transaction with a TTL counter used for stem timeout tracking:

```rust
#[cfg(feature = "dandelion")]
/// A transaction in the Dandelion++ stem phase.
/// Not added to mempool — forwarded privately to exactly one stem relay peer.
pub struct StemTransaction {
    /// The inner transaction payload.
    pub tx: Transaction,

    /// Time-to-live counter for stem timeout tracking (seconds).
    /// Decremented or used to compute expiry at each hop.
    pub ttl: u64,
}
```

### Stem Phase Forwarding Rules

1. **Origination**: When a node creates a new transaction and Dandelion++ is enabled, it MUST wrap the transaction in a `StemTransaction` and send it to the current epoch's stem relay peer (selected per PRV-005).

2. **Forwarding**: When a node receives a `StemTransaction` from a peer, it MUST forward the `StemTransaction` to its own stem relay peer (not back to the sender, not broadcast).

3. **Exactly one peer**: The `StemTransaction` MUST be sent to exactly one peer — the epoch's stem relay. It MUST NOT be broadcast, gossiped, or sent to multiple peers.

4. **No mempool insertion**: During stem phase, the transaction MUST NOT be added to the local mempool. The node holds a reference to the stemmed transaction in a separate `stem_buffer` data structure.

5. **No RequestTransaction response**: If a peer sends `RequestTransaction` for a transaction that exists only in the stem buffer (not in the mempool), the node MUST NOT respond with the transaction data. This prevents revealing the node's position on the stem path.

### Stem Buffer

The node MUST maintain a `stem_buffer` to track transactions currently in stem phase:

```rust
#[cfg(feature = "dandelion")]
/// Buffer for transactions currently in Dandelion++ stem phase.
/// These transactions are NOT in the mempool and MUST NOT be served
/// in response to RequestTransaction.
pub struct StemBuffer {
    /// Map from transaction ID to the stemmed transaction and its timeout instant.
    entries: HashMap<TransactionId, StemBufferEntry>,
}

#[cfg(feature = "dandelion")]
struct StemBufferEntry {
    tx: Transaction,
    received_at: Instant,
    timeout_deadline: Instant,
}
```

### Pseudocode: on_stem_receive() Handler

```rust
#[cfg(feature = "dandelion")]
fn on_stem_receive(&mut self, stem_tx: StemTransaction, from_peer: PeerId) {
    let tx_id = stem_tx.tx.id();

    // 1. Deduplicate: if already seen (in stem_buffer or mempool), ignore.
    if self.stem_buffer.contains(&tx_id) || self.mempool.contains(&tx_id) {
        return;
    }

    // 2. Do NOT add to mempool.
    // 3. Store in stem buffer with timeout deadline.
    let deadline = Instant::now() + Duration::from_secs(self.config.stem_timeout_secs);
    self.stem_buffer.insert(tx_id, StemBufferEntry {
        tx: stem_tx.tx.clone(),
        received_at: Instant::now(),
        timeout_deadline: deadline,
    });

    // 4. Decide: fluff or continue stem (see PRV-003 for coin flip logic).
    if self.should_fluff() {
        // Transition to fluff phase (PRV-003).
        self.fluff_transaction(stem_tx.tx);
    } else {
        // 5. Forward to this node's stem relay (exactly one peer).
        let relay = self.current_stem_relay(); // Selected per PRV-005
        self.send_to_peer(relay, StemTransaction {
            tx: stem_tx.tx,
            ttl: stem_tx.ttl,
        });

        // 6. Start stem timeout timer (PRV-004).
        self.start_stem_timeout(tx_id, deadline);
    }
}
```

### on_request_transaction() Guard

```rust
#[cfg(feature = "dandelion")]
fn on_request_transaction(&self, tx_id: &TransactionId) -> Option<Transaction> {
    // Only serve transactions from the mempool, NEVER from the stem buffer.
    self.mempool.get(tx_id)
    // stem_buffer is intentionally not checked here.
}
```

## Acceptance Criteria

- [ ] `StemTransaction` struct is defined with `tx` (Transaction) and `ttl` (u64) fields
- [ ] `StemTransaction` is gated behind `#[cfg(feature = "dandelion")]`
- [ ] When a node originates a tx with Dandelion++ enabled, it sends `StemTransaction` to the stem relay
- [ ] When a node receives a `StemTransaction`, it forwards to its own stem relay (exactly 1 peer)
- [ ] `StemTransaction` is NEVER broadcast to multiple peers during stem phase
- [ ] Stem-phase transactions are NOT added to the local mempool
- [ ] Stem-phase transactions are stored in the `stem_buffer`
- [ ] `RequestTransaction` for a stem-only transaction returns nothing
- [ ] `RequestTransaction` for a mempool transaction returns the transaction normally
- [ ] Duplicate `StemTransaction` messages (same tx_id) are ignored
- [ ] Stem relay peer is the epoch's consistent relay (per PRV-005)
- [ ] Stem timeout timer is started on stem forward (per PRV-004)

## Implementation Notes

- Primary file: `src/privacy/dandelion.rs`
- Dependencies: PRV-001 (DandelionConfig), PRV-005 (stem relay selection)
- Related: PRV-003 (fluff transition), PRV-004 (stem timeout)
- Edge cases:
  - If the stem relay is unreachable at forward time, the node SHOULD force-fluff rather than silently dropping the transaction
  - If Dandelion++ is disabled (`config.enabled = false`), transactions skip stem entirely and go directly to normal gossip
  - The stem buffer should have a maximum size to prevent memory exhaustion under flood conditions
  - Transactions received via normal gossip (fluff) that are already in the stem buffer should be removed from the stem buffer and their timeout cancelled
