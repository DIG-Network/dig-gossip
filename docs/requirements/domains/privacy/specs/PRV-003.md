# PRV-003: Fluff Transition Probability

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 1.9.1](../../../resources/SPEC.md)

## Summary

At each stem hop, the receiving node flips a weighted coin to decide whether to transition the transaction from stem phase to fluff phase (normal gossip). The probability of fluffing is controlled by `DANDELION_FLUFF_PROBABILITY` (default 10%). If fluff is chosen, the transaction is added to the local mempool and broadcast via Plumtree eager/lazy push (or ERLAY if enabled). If stem is chosen, the transaction is forwarded to the node's stem relay and a stem timeout timer is started. This probabilistic transition is the core mechanism that decouples the transaction originator from the point where the transaction enters public gossip.

## Specification

### Coin Flip Logic

Each time a node receives a `StemTransaction` (whether originated locally or forwarded from another stem node), it MUST perform a weighted coin flip to decide the phase transition:

```rust
#[cfg(feature = "dandelion")]
use rand::Rng;

impl DandelionState {
    /// Decide whether to transition a stem transaction to fluff phase.
    /// Returns `true` for fluff (broadcast), `false` for stem (forward).
    pub fn should_fluff(&self) -> bool {
        let mut rng = rand::thread_rng();
        let roll: f64 = rng.gen(); // Uniform [0.0, 1.0)
        roll < self.config.fluff_probability
    }
}
```

- With `fluff_probability = 0.10`: 10% chance of fluff, 90% chance of continuing stem.
- With `fluff_probability = 0.0`: always stem (never voluntarily fluff; only timeout-based fluff).
- With `fluff_probability = 1.0`: always fluff (stem phase is effectively disabled).

### Fluff Action

When fluff is chosen (`should_fluff()` returns `true`):

```rust
#[cfg(feature = "dandelion")]
impl DandelionState {
    /// Transition a stem transaction to fluff phase.
    /// Adds to mempool and broadcasts via normal gossip.
    pub fn fluff_transaction(&mut self, tx: Transaction) {
        let tx_id = tx.id();

        // 1. Remove from stem buffer if present.
        self.stem_buffer.remove(&tx_id);

        // 2. Add to local mempool.
        self.mempool.insert(tx.clone());

        // 3. Cancel any pending stem timeout for this tx.
        self.cancel_stem_timeout(&tx_id);

        // 4. Broadcast via normal gossip (Plumtree eager/lazy push or ERLAY).
        self.broadcast_transaction(tx);
    }
}
```

### Stem Action

When stem is chosen (`should_fluff()` returns `false`):

```rust
#[cfg(feature = "dandelion")]
impl DandelionState {
    /// Continue stem phase: forward to this node's stem relay.
    pub fn continue_stem(&mut self, stem_tx: StemTransaction) {
        let tx_id = stem_tx.tx.id();

        // 1. Ensure tx is in stem buffer (for timeout tracking).
        //    (Already inserted by on_stem_receive handler — see PRV-002)

        // 2. Forward to this node's stem relay peer.
        let relay = self.current_stem_relay();
        self.send_to_peer(relay, stem_tx);

        // 3. Start stem timeout timer (see PRV-004).
        self.start_stem_timeout(tx_id);
    }
}
```

### Configurable Probability

The fluff probability MUST be configurable via `DandelionConfig.fluff_probability` (PRV-001). The value MUST be in the range `[0.0, 1.0]`:

- `0.0` = never voluntarily fluff (valid but relies entirely on stem timeout for liveness)
- `1.0` = always fluff on first hop (valid but disables privacy benefit of stem phase)
- `0.10` = default, balancing privacy and propagation latency per the Dandelion++ paper

### Integration with Gossip Layer

When `fluff_transaction()` is called, the broadcast MUST use the same mechanism as a non-Dandelion transaction:

- If ERLAY is enabled: the transaction enters the ERLAY reconciliation pipeline.
- If ERLAY is disabled: the transaction is broadcast via Plumtree eager push to eager peers and lazy push (IHAVE) to lazy peers.
- The fluffed transaction is indistinguishable from a locally-originated non-Dandelion transaction to all receiving peers.

## Acceptance Criteria

- [ ] `should_fluff()` method exists and uses `DandelionConfig.fluff_probability` for the coin flip
- [ ] Coin flip uses uniform random `[0.0, 1.0)` and compares against `fluff_probability`
- [ ] With `fluff_probability = 0.0`, `should_fluff()` always returns `false`
- [ ] With `fluff_probability = 1.0`, `should_fluff()` always returns `true`
- [ ] With `fluff_probability = 0.10`, approximately 10% of calls return `true` over many trials
- [ ] On fluff: transaction is added to the local mempool
- [ ] On fluff: transaction is broadcast via Plumtree/ERLAY (normal gossip)
- [ ] On fluff: transaction is removed from the stem buffer
- [ ] On fluff: stem timeout for this transaction is cancelled
- [ ] On stem: transaction is forwarded to exactly one peer (the stem relay)
- [ ] On stem: transaction is NOT added to the mempool
- [ ] On stem: stem timeout timer is started
- [ ] Fluffed transaction is indistinguishable from a non-Dandelion transaction to recipients

## Implementation Notes

- Primary file: `src/privacy/dandelion.rs`
- Dependencies: PRV-001 (DandelionConfig for fluff_probability), PRV-002 (stem forwarding), PRV-004 (timeout)
- External dependency: `rand` crate for random number generation
- Edge cases:
  - The coin flip MUST be performed independently for each received `StemTransaction` — not batched or pre-determined
  - If the RNG is seeded deterministically (e.g., in tests), the flip results are reproducible
  - `fluff_probability` can be changed at runtime via config reload, but the change takes effect on the next coin flip (not retroactively)
  - Floating-point comparison `roll < fluff_probability` is well-defined for IEEE 754 doubles in `[0.0, 1.0]`
