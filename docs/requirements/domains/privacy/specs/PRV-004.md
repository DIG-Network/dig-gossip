# PRV-004: Stem Timeout and Force-Fluff

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 1.9.1](../../../resources/SPEC.md)

## Summary

If a stemmed transaction is not observed via normal fluff gossip within `DANDELION_STEM_TIMEOUT_SECS` (default 30 seconds), the node holding the transaction MUST force-fluff it — adding it to the local mempool and broadcasting it via normal gossip. This timeout mechanism ensures liveness: even if the stem path is broken (e.g., the downstream relay disconnected or dropped the transaction), the transaction will still reach the network. Force-fluff produces a result identical to voluntary fluff, so observers cannot distinguish timeout-triggered fluff from coin-flip-triggered fluff.

## Specification

### Stem Timeout Timer

When a node continues the stem phase for a transaction (i.e., `should_fluff()` returned `false` and the `StemTransaction` was forwarded to the relay), it MUST start a timer:

```rust
#[cfg(feature = "dandelion")]
impl DandelionState {
    /// Start a stem timeout timer for a transaction.
    /// If the timer expires, the transaction will be force-fluffed.
    pub fn start_stem_timeout(&mut self, tx_id: TransactionId, deadline: Instant) {
        // Schedule a timeout task that fires at `deadline`.
        // On expiry, call `force_fluff(tx_id)`.
        let timeout_handle = tokio::spawn({
            let tx_id = tx_id.clone();
            let notifier = self.timeout_notifier.clone();
            async move {
                tokio::time::sleep_until(deadline.into()).await;
                let _ = notifier.send(tx_id);
            }
        });

        // Store handle so it can be cancelled.
        self.stem_buffer.set_timeout_handle(&tx_id, timeout_handle);
    }
}
```

The timeout duration is `DandelionConfig.stem_timeout_secs` (default 30 seconds, defined in PRV-001).

### Timer Cancellation

If the transaction is observed via normal fluff gossip (i.e., the node receives the same transaction through Plumtree/ERLAY broadcast) before the timeout expires, the timer MUST be cancelled:

```rust
#[cfg(feature = "dandelion")]
impl DandelionState {
    /// Cancel the stem timeout for a transaction.
    /// Called when the transaction is seen via normal fluff gossip.
    pub fn cancel_stem_timeout(&mut self, tx_id: &TransactionId) {
        if let Some(entry) = self.stem_buffer.get_mut(tx_id) {
            // Abort the timeout task.
            if let Some(handle) = entry.timeout_handle.take() {
                handle.abort();
            }
        }
        // Remove from stem buffer — it's now in the mempool via fluff.
        self.stem_buffer.remove(tx_id);
    }
}
```

### on_fluff_receive() Integration

When a node receives a transaction via normal gossip (not `StemTransaction`), it MUST check whether that transaction is in the stem buffer:

```rust
#[cfg(feature = "dandelion")]
fn on_transaction_received_via_gossip(&mut self, tx: Transaction) {
    let tx_id = tx.id();

    // If this tx was in stem phase, the stem succeeded — cancel timeout.
    if self.stem_buffer.contains(&tx_id) {
        self.cancel_stem_timeout(&tx_id);
    }

    // Normal gossip processing: add to mempool, relay, etc.
    self.mempool.insert(tx);
}
```

### Force-Fluff

When the stem timeout expires, the node MUST force-fluff the transaction. Force-fluff is identical to voluntary fluff (PRV-003):

```rust
#[cfg(feature = "dandelion")]
impl DandelionState {
    /// Force-fluff a transaction after stem timeout.
    /// Identical to voluntary fluff — adds to mempool and broadcasts.
    pub fn force_fluff(&mut self, tx_id: &TransactionId) {
        // 1. Retrieve the transaction from stem buffer.
        let tx = match self.stem_buffer.remove(tx_id) {
            Some(entry) => entry.tx,
            None => return, // Already fluffed or removed; no-op.
        };

        // 2. Add to local mempool.
        self.mempool.insert(tx.clone());

        // 3. Broadcast via normal gossip (Plumtree/ERLAY).
        //    This is identical to the fluff path in PRV-003.
        self.broadcast_transaction(tx);
    }
}
```

### Liveness Guarantee

The stem timeout ensures that no transaction is permanently lost due to a broken stem path. Specifically:

1. If the stem relay is offline, dropped the message, or itself failed to forward: the originating/holding node's timeout fires and the transaction enters gossip.
2. If the stem relay forwarded but a downstream node failed: that downstream node's timeout also fires, providing redundant safety.
3. Multiple nodes along the stem path may independently force-fluff the same transaction — this is safe because mempool insertion and gossip broadcast are idempotent for duplicate transaction IDs.

### Configurable Timeout

The timeout duration is set via `DandelionConfig.stem_timeout_secs` (PRV-001). The default is 30 seconds. The value MUST be configurable to allow tuning the tradeoff between liveness latency and stem path length:

- Lower values (e.g., 10s): faster liveness recovery, but reduces stem path length in practice
- Higher values (e.g., 60s): allows longer stem paths, but transactions take longer to recover from broken paths
- Zero (0s): immediate force-fluff on receipt — effectively disables stem phase

## Acceptance Criteria

- [ ] Stem timeout timer starts when a transaction continues stem phase (forwarded to relay)
- [ ] Timer duration is `DandelionConfig.stem_timeout_secs` (default 30s)
- [ ] If timeout expires without fluff observation, the transaction is force-fluffed
- [ ] Force-fluff adds the transaction to the local mempool
- [ ] Force-fluff broadcasts the transaction via normal gossip (Plumtree/ERLAY)
- [ ] Force-fluff is identical in behavior to voluntary fluff (PRV-003)
- [ ] If the transaction is seen via normal fluff gossip before timeout, the timer is cancelled
- [ ] Cancelled timer does NOT produce a force-fluff (no double-fluff)
- [ ] Timeout duration is configurable via `DandelionConfig.stem_timeout_secs`
- [ ] Multiple force-fluffs of the same transaction are idempotent (no crash or duplicate broadcast)
- [ ] Transaction is removed from stem buffer on force-fluff
- [ ] Transaction is removed from stem buffer on fluff observation (cancellation)

## Implementation Notes

- Primary file: `src/privacy/dandelion.rs`
- Dependencies: PRV-001 (DandelionConfig for stem_timeout_secs), PRV-002 (stem buffer), PRV-003 (fluff_transaction)
- Async runtime: Uses `tokio::time::sleep_until` for timer scheduling
- Edge cases:
  - If the node receives the same transaction via both stem and fluff nearly simultaneously, the stem buffer check in `on_transaction_received_via_gossip` must be race-condition safe
  - If `force_fluff` is called but the transaction was already removed from the stem buffer (e.g., by a concurrent fluff observation), the function is a no-op
  - Timer handles must be properly dropped/aborted on node shutdown to prevent dangling tasks
  - Under high transaction volume, many concurrent timers may be active; consider using a timer wheel or `tokio::time::interval` batching for efficiency
