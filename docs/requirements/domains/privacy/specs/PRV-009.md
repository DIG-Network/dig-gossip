# PRV-009: TorConfig Struct

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 1.9.3](../../../resources/SPEC.md)

## Summary

The `TorConfig` struct defines the configuration for Tor/SOCKS5 proxy transport. For nodes requiring strong IP privacy, `dig-gossip` supports routing connections through the Tor network. This struct controls whether Tor is enabled, the SOCKS5 proxy address for the local Tor daemon, an optional `.onion` address for inbound connections, and whether Tor should be preferred over direct connections. All Tor functionality is feature-gated behind the `tor` feature flag, which depends on the `tokio-socks` crate.

## Specification

### TorConfig Struct Definition

The `TorConfig` struct MUST be defined with the following fields, gated behind the `tor` feature:

```rust
use serde::{Deserialize, Serialize};

/// Tor/SOCKS5 proxy configuration.
///
/// Controls routing of peer connections through the Tor network for IP privacy.
/// Requires a running Tor daemon accessible at the configured SOCKS5 proxy address.
///
/// This struct is only available when the `tor` feature is enabled.
#[cfg(feature = "tor")]
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct TorConfig {
    /// Enable Tor transport.
    /// Default: false.
    pub enabled: bool,

    /// SOCKS5 proxy address (Tor daemon).
    /// Default: "127.0.0.1:9050".
    pub socks5_proxy: String,

    /// Hidden service address (.onion) for inbound connections.
    /// If None, Tor is outbound-only (node can connect out via Tor
    /// but does not accept inbound Tor connections).
    pub onion_address: Option<String>,

    /// Prefer Tor over direct connections for all outbound.
    /// Default: false. If true, all outbound connections go through Tor.
    /// If false, direct P2P is preferred, then relay, then Tor.
    pub prefer_tor: bool,
}
```

### Default Implementation

The `Default` impl MUST produce values matching the SPEC defaults:

```rust
#[cfg(feature = "tor")]
impl Default for TorConfig {
    fn default() -> Self {
        Self {
            enabled: false,
            socks5_proxy: "127.0.0.1:9050".to_string(),
            onion_address: None,
            prefer_tor: false,
        }
    }
}
```

### Helper Methods

The struct MUST provide an `is_configured()` helper that checks whether Tor is enabled and the proxy address is non-empty:

```rust
#[cfg(feature = "tor")]
impl TorConfig {
    /// Returns true if Tor is enabled and minimally configured.
    ///
    /// Checks that `enabled` is true and `socks5_proxy` is not empty.
    pub fn is_configured(&self) -> bool {
        self.enabled && !self.socks5_proxy.is_empty()
    }

    /// Returns true if the node has a .onion address for inbound Tor connections.
    pub fn has_onion_address(&self) -> bool {
        self.onion_address.is_some()
    }

    /// Returns true if the node should be operating in hybrid mode
    /// (accepting both direct P2P and Tor connections).
    pub fn is_hybrid_mode(&self) -> bool {
        self.enabled && self.onion_address.is_some()
    }
}
```

### Feature Gate

The `tor` feature MUST be defined in `Cargo.toml`:

```toml
[features]
tor = ["tokio-socks"]

[dependencies]
tokio-socks = { version = "0.5", optional = true }
```

All Tor-related code MUST be gated behind `#[cfg(feature = "tor")]`. When the `tor` feature is not enabled:
- `TorConfig` struct MUST NOT be compiled
- No `tokio-socks` dependency is pulled in
- The crate compiles and functions normally without Tor support

## Acceptance Criteria

- [ ] `TorConfig` struct is defined with fields `enabled`, `socks5_proxy`, `onion_address`, `prefer_tor`
- [ ] `enabled` field is `bool` with default `false`
- [ ] `socks5_proxy` field is `String` with default `"127.0.0.1:9050"`
- [ ] `onion_address` field is `Option<String>` with default `None`
- [ ] `prefer_tor` field is `bool` with default `false`
- [ ] `Default` impl produces `enabled: false, socks5_proxy: "127.0.0.1:9050", onion_address: None, prefer_tor: false`
- [ ] Struct derives `Debug`, `Clone`, `Serialize`, `Deserialize`
- [ ] `is_configured()` returns `true` only when `enabled = true` and `socks5_proxy` is non-empty
- [ ] `is_configured()` returns `false` when `enabled = false`
- [ ] `has_onion_address()` returns `true` when `onion_address` is `Some`
- [ ] `is_hybrid_mode()` returns `true` when enabled with an onion address
- [ ] All Tor code is gated behind `#[cfg(feature = "tor")]`
- [ ] Crate compiles without `tor` feature (no Tor code included)
- [ ] Crate compiles with `tor` feature (tokio-socks dependency resolved)

## Implementation Notes

- Primary file: `src/privacy/tor.rs`
- Dependencies:
  - `serde` (for serialization)
  - `tokio-socks` (optional, behind `tor` feature gate)
- Edge cases:
  - `enabled = true` but `socks5_proxy` is empty: `is_configured()` returns false; transport layer should fall back to direct
  - `onion_address` with an invalid `.onion` format: validation is deferred to the transport layer (PRV-010), not enforced at config level
  - `prefer_tor = true` but `enabled = false`: Tor preference is ignored; direct connections are used. The `is_configured()` check gates all Tor behavior.
  - When the `tor` feature is disabled, any configuration file fields related to Tor should be silently ignored during deserialization (use `#[serde(default)]` on the parent config struct)
