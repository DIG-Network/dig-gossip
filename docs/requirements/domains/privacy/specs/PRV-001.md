# PRV-001: DandelionConfig Struct

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)

> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 1.9.1](../../../resources/SPEC.md)

## Summary

The `DandelionConfig` struct defines all tunable parameters for the Dandelion++ transaction origin privacy system. It controls whether stem-phase forwarding is enabled, the probability of transitioning from stem to fluff at each hop, the timeout before force-fluffing a stemmed transaction, and the epoch duration for stem relay rotation. The struct and its associated constants are feature-gated behind the `dandelion` feature flag and MUST NOT compile into the binary when the feature is disabled.

## Specification

### DandelionConfig Struct

The `DandelionConfig` struct MUST be defined as follows:

```rust
#[cfg(feature = "dandelion")]
/// Dandelion++ configuration.
pub struct DandelionConfig {
    /// Enable Dandelion++ stem phase for outgoing transactions.
    /// Default: true.
    pub enabled: bool,

    /// Probability of transitioning from stem to fluff at each hop.
    /// Default: 0.10 (10%). Higher values = shorter stems = less privacy.
    /// MUST be in range [0.0, 1.0].
    pub fluff_probability: f64,

    /// Timeout before a stem transaction is force-fluffed (seconds).
    /// Default: 30.
    pub stem_timeout_secs: u64,

    /// Duration of a stem relay epoch (seconds).
    /// The stem relay peer is re-randomized at each epoch boundary.
    /// Default: 600 (10 minutes).
    pub epoch_secs: u64,
}
```

### Default Implementation

The `Default` impl MUST produce values matching the SPEC constants:

```rust
#[cfg(feature = "dandelion")]
impl Default for DandelionConfig {
    fn default() -> Self {
        Self {
            enabled: true,
            fluff_probability: DANDELION_FLUFF_PROBABILITY,
            stem_timeout_secs: DANDELION_STEM_TIMEOUT_SECS,
            epoch_secs: DANDELION_EPOCH_SECS,
        }
    }
}
```

### Constants

The following constants MUST be defined in `src/constants.rs` and gated behind the `dandelion` feature:

```rust
#[cfg(feature = "dandelion")]
/// Probability of transitioning stem -> fluff at each hop.
pub const DANDELION_FLUFF_PROBABILITY: f64 = 0.10;

#[cfg(feature = "dandelion")]
/// Timeout before a stem transaction is force-fluffed (seconds).
pub const DANDELION_STEM_TIMEOUT_SECS: u64 = 30;

#[cfg(feature = "dandelion")]
/// Duration of a stem relay epoch (seconds). Relay re-randomized each epoch.
pub const DANDELION_EPOCH_SECS: u64 = 600;
```

### Feature Gate

The `dandelion` feature flag MUST be declared in `Cargo.toml`:

```toml
[features]
dandelion = []
```

When the `dandelion` feature is not enabled, `DandelionConfig`, its `Default` impl, and all associated constants MUST NOT be compiled. Code that references these items MUST use `#[cfg(feature = "dandelion")]` guards.

### Field Validation

The `fluff_probability` field MUST be validated to be within the range `[0.0, 1.0]` inclusive. A constructor or validation method SHOULD be provided:

```rust
#[cfg(feature = "dandelion")]
impl DandelionConfig {
    /// Validate that all fields contain legal values.
    pub fn validate(&self) -> Result<(), String> {
        if !(0.0..=1.0).contains(&self.fluff_probability) {
            return Err(format!(
                "fluff_probability must be in [0.0, 1.0], got {}",
                self.fluff_probability
            ));
        }
        Ok(())
    }
}
```

## Implementation Notes

- Primary file: `src/privacy/dandelion.rs`
- Constants file: `src/constants.rs`
- Feature gate file: `Cargo.toml`
- Dependencies: None (pure data struct with no external dependencies)
- Edge cases:
  - `fluff_probability = 0.0` means transactions never voluntarily fluff (only via timeout) — valid but extreme
  - `fluff_probability = 1.0` means every stem hop immediately fluffs — effectively disables Dandelion++
  - `stem_timeout_secs = 0` would cause immediate force-fluff on receipt — effectively disables stem phase
  - `epoch_secs = 0` would cause constant relay rotation — should be guarded or documented
