# PRV-005: Stem Relay Epoch Rotation

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 1.9.1](../../../resources/SPEC.md)

## Summary

Each node maintains a single stem relay peer for the Dandelion++ stem phase. The relay is re-randomized every `DANDELION_EPOCH_SECS` (default 600 seconds / 10 minutes). Within an epoch, all stem-phase transactions are forwarded to the same relay peer, creating a consistent routing topology that is harder to fingerprint than per-transaction random selection. The epoch timer is local to each node (not network-synchronized). If the current stem relay disconnects mid-epoch, a new relay is immediately selected to maintain liveness.

## Specification

### Epoch-Based Relay Selection

The stem relay selection is managed by an epoch system:

```rust
#[cfg(feature = "dandelion")]
/// Manages stem relay selection with epoch-based rotation.
pub struct StemRelayManager {
    /// The currently selected stem relay peer.
    current_relay: Option<PeerId>,

    /// Instant when the current epoch started.
    epoch_start: Instant,

    /// Duration of each epoch (from DandelionConfig.epoch_secs).
    epoch_duration: Duration,
}
```

### Relay Selection Logic

At each epoch boundary (and on startup), the node MUST select a new random outbound peer as the stem relay:

```rust
#[cfg(feature = "dandelion")]
impl StemRelayManager {
    /// Create a new StemRelayManager with the given epoch duration.
    pub fn new(epoch_secs: u64) -> Self {
        Self {
            current_relay: None,
            epoch_start: Instant::now(),
            epoch_duration: Duration::from_secs(epoch_secs),
        }
    }

    /// Get the current stem relay, selecting a new one if the epoch has expired.
    pub fn get_or_rotate_relay(&mut self, outbound_peers: &[PeerId]) -> Option<PeerId> {
        let now = Instant::now();

        // Check if epoch has expired.
        if now.duration_since(self.epoch_start) >= self.epoch_duration
            || self.current_relay.is_none()
        {
            self.rotate_relay(outbound_peers);
        }

        self.current_relay
    }

    /// Select a new random outbound peer as stem relay. Resets the epoch timer.
    fn rotate_relay(&mut self, outbound_peers: &[PeerId]) {
        use rand::seq::SliceRandom;

        self.current_relay = outbound_peers
            .choose(&mut rand::thread_rng())
            .copied();
        self.epoch_start = Instant::now();
    }
}
```

### Consistency Within Epoch

Within a single epoch, ALL stem-phase transactions MUST be forwarded to the same relay peer. The relay does NOT change per transaction:

```rust
#[cfg(feature = "dandelion")]
impl DandelionState {
    /// Get the current epoch's stem relay peer.
    pub fn current_stem_relay(&mut self) -> Option<PeerId> {
        self.relay_manager.get_or_rotate_relay(&self.outbound_peers())
    }
}
```

This consistency is critical for the Dandelion++ privacy guarantee. Per-transaction random selection would create a "tree" topology where the originator is the root; consistent relay creates a "line" topology where any node on the line is equally likely to be the originator.

### Epoch Timer Independence

The epoch timer is local to each node. Nodes do NOT synchronize their epoch boundaries:

- Node A may rotate its relay at `t=0, t=600, t=1200, ...`
- Node B may rotate its relay at `t=47, t=647, t=1247, ...`
- There is no epoch synchronization message or protocol.

This independence prevents an adversary from predicting when all nodes rotate simultaneously.

### Mid-Epoch Relay Disconnection

If the current stem relay disconnects mid-epoch, the node MUST immediately select a new relay from the remaining outbound peers:

```rust
#[cfg(feature = "dandelion")]
impl StemRelayManager {
    /// Handle disconnection of a peer. If it was the stem relay,
    /// immediately select a new one.
    pub fn on_peer_disconnected(
        &mut self,
        disconnected: PeerId,
        outbound_peers: &[PeerId],
    ) {
        if self.current_relay == Some(disconnected) {
            // Immediate re-selection, but do NOT reset epoch timer.
            // The epoch continues — only the relay changes.
            self.current_relay = {
                use rand::seq::SliceRandom;
                outbound_peers
                    .iter()
                    .filter(|p| **p != disconnected)
                    .copied()
                    .collect::<Vec<_>>()
                    .choose(&mut rand::thread_rng())
                    .copied()
            };
        }
    }
}
```

Note: mid-epoch re-selection does NOT reset the epoch timer. The next scheduled rotation still occurs at the original epoch boundary.

### New Peer Eligibility

When a new outbound peer connects during an epoch:

- If a relay is already selected and connected, the new peer does NOT replace it.
- If no relay is currently selected (e.g., all peers were disconnected), the new peer MAY be immediately selected as the relay.
- At the next epoch boundary, the new peer is included in the selection pool.

### Startup Behavior

On node startup:

1. The `StemRelayManager` is initialized with `current_relay = None`.
2. The first call to `get_or_rotate_relay()` selects a relay from available outbound peers.
3. If no outbound peers are available yet, `current_relay` remains `None` and stem transactions are force-fluffed (per PRV-004 timeout or immediate fallback).

## Acceptance Criteria

- [ ] Each node maintains exactly one `current_relay` peer for stem forwarding
- [ ] Relay is selected randomly from outbound peers
- [ ] Relay is consistent within an epoch — all stem txs in the same epoch go to the same peer
- [ ] Relay is re-randomized every `DANDELION_EPOCH_SECS` (default 600s)
- [ ] Epoch timer is local to each node (not network-synchronized)
- [ ] On epoch boundary: new random outbound peer is selected as relay
- [ ] Mid-epoch relay disconnection triggers immediate re-selection from remaining peers
- [ ] Mid-epoch re-selection does NOT reset the epoch timer
- [ ] Newly connected peers are eligible for relay selection at the next epoch boundary
- [ ] If no outbound peers are available, `current_relay` is `None`
- [ ] When `current_relay` is `None`, stem transactions fall back to force-fluff
- [ ] `StemRelayManager` is gated behind `#[cfg(feature = "dandelion")]`

## Implementation Notes

- Primary file: `src/privacy/dandelion.rs`
- Dependencies: PRV-001 (DandelionConfig for epoch_secs), `rand` crate for random selection
- Related: PRV-002 (uses current_stem_relay for forwarding), PRV-004 (fallback when no relay available)
- Edge cases:
  - If only one outbound peer exists, it is always the relay (no randomness needed)
  - If the selected relay is the same peer that sent the `StemTransaction`, a privacy-conscious implementation may skip it and choose another — but the SPEC does not require this
  - Epoch duration of 0 would cause rotation on every call — functionally equivalent to per-transaction random selection
  - Clock monotonicity: uses `Instant` (monotonic clock), not wall-clock time, to avoid issues with system clock adjustments
