# PRV-007: Certificate Rotation and Reconnection

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 1.9.2](../../../resources/SPEC.md)

## Summary

Every `rotation_interval_secs`, the node generates a fresh `ChiaCertificate` via `chia-ssl`'s `ChiaCertificate::generate()`, which gives the node a new PeerId (the SHA-256 hash of the new TLS public key). If `reconnect_on_rotation` is true, all connected peers are disconnected and reconnected using the new identity. Network identity (PeerId from TLS cert) is independent of consensus identity (validator BLS keys), so rotation does not affect staking or attestation signing. The address manager tracks peers by IP:port, not by PeerId, so rotation does not break peer discovery.

## Specification

### Certificate Rotation Loop

The rotation loop MUST run as a background task that sleeps for `rotation_interval_secs`, then performs the rotation sequence:

```rust
use chia_ssl::ChiaCertificate;
use std::time::Duration;
use tokio::time::sleep;

/// Runs the PeerId rotation loop.
///
/// This task sleeps for the configured interval, generates a new TLS certificate,
/// updates the node's TLS acceptor/connector, and optionally reconnects all peers.
pub async fn rotation_loop(
    config: PeerIdRotationConfig,
    tls_state: Arc<RwLock<TlsState>>,
    peer_manager: Arc<PeerManager>,
) {
    // Do not run if rotation is disabled
    if !config.enabled || config.rotation_interval_secs == 0 {
        tracing::info!("PeerId rotation disabled, rotation loop will not run");
        return;
    }

    loop {
        // Sleep for the rotation interval
        sleep(Duration::from_secs(config.rotation_interval_secs)).await;

        // Step 1: Generate a fresh ChiaCertificate
        let new_cert = match ChiaCertificate::generate() {
            Ok(cert) => cert,
            Err(e) => {
                tracing::error!("Failed to generate new ChiaCertificate: {}", e);
                continue;
            }
        };

        // Step 2: Update the TLS state (acceptor + connector) with the new cert
        // The new PeerId is the SHA-256 hash of the new TLS public key
        {
            let mut tls = tls_state.write().await;
            tls.update_certificate(new_cert).await;
        }

        tracing::info!("PeerId rotated to new identity");

        // Step 3: Optionally reconnect all peers with the new identity
        if config.reconnect_on_rotation {
            tracing::info!("Disconnecting all peers for identity rotation");
            peer_manager.disconnect_all().await;
            peer_manager.reconnect_all().await;
            tracing::info!("All peers reconnected with new PeerId");
        } else {
            tracing::info!(
                "reconnect_on_rotation=false; only new connections will use new identity"
            );
        }
    }
}
```

### PeerId Derivation

The PeerId MUST be derived from the TLS certificate's public key:

- PeerId = SHA-256(TLS public key bytes)
- Generating a new `ChiaCertificate` produces a new key pair, which yields a new PeerId
- The old PeerId is no longer valid once the TLS state is updated

### Reconnection Behavior

When `reconnect_on_rotation` is **true**:
- All currently connected peers MUST be disconnected
- The node MUST reconnect to the same set of peers using the new TLS certificate
- Peers will see a new PeerId on the incoming connection

When `reconnect_on_rotation` is **false**:
- Existing connections MUST remain open with the old TLS certificate/PeerId
- Only newly established connections (outbound or inbound) will use the new certificate
- This creates a transitional period where the node has two identities in the network

### Identity Separation

- **Network identity**: PeerId derived from TLS certificate. This is what rotates.
- **Consensus identity**: Validator BLS keys used for attestation signing, checkpoint participation, and staking. These MUST NOT be affected by PeerId rotation.
- Rotation MUST NOT invalidate or change any consensus-layer keys or state.

### Address Manager Independence

The address manager MUST track peers by `IP:port` (socket address), NOT by PeerId:
- When a peer rotates its PeerId, the address manager retains the peer's entry because the IP:port has not changed
- Rotation does not cause address manager churn or bucket redistribution
- Peer reputation and scoring tied to IP:port persist across rotations

## Acceptance Criteria

- [ ] After `rotation_interval_secs` elapses, a new `ChiaCertificate` is generated via `ChiaCertificate::generate()`
- [ ] The new certificate updates the node's TLS acceptor and connector
- [ ] After rotation, the node's PeerId (SHA-256 of TLS public key) has changed
- [ ] The old PeerId is no longer used for new connections
- [ ] When `reconnect_on_rotation = true`, all peers are disconnected and reconnected with the new identity
- [ ] When `reconnect_on_rotation = false`, existing connections retain old identity; only new connections use new identity
- [ ] Consensus identity (validator BLS keys) is unchanged after rotation
- [ ] Address manager entries are unaffected by PeerId rotation (tracked by IP:port)
- [ ] Rotation loop exits cleanly when `enabled = false` or `rotation_interval_secs = 0`
- [ ] Rotation loop handles certificate generation failures gracefully (logs error, retries on next interval)

## Implementation Notes

- Primary file: `src/privacy/peer_id_rotation.rs`
- Dependencies:
  - `chia-ssl` (`ChiaCertificate::generate()`)
  - `chia-sdk-client` (`create_native_tls_connector()` / `create_rustls_connector()` for rebuilding TLS with new cert)
  - `tokio` (async sleep, task spawning)
  - `tracing` (structured logging)
- Edge cases:
  - Certificate generation failure should not crash the node; log and retry at next interval
  - During reconnection, there is a brief window where the node has zero peers. The connection manager should handle re-bootstrapping if reconnection to known peers fails.
  - If the node is shutting down during a rotation sleep, the task should be cancellable via `tokio::select!` with a shutdown signal
  - Multiple rapid rotations (very small interval) should not cause connection storms; the reconnection logic should respect backoff
