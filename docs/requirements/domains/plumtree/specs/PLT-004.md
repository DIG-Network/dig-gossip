# PLT-004: Duplicate Detection and Pruning

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 8.1](../../../resources/SPEC.md)

## Summary

When a message is received via eager push from a peer, and the message hash is already in the seen set, the spanning tree has a redundant edge. The protocol resolves this by demoting the sender from `eager_peers` to `lazy_peers` and sending a PRUNE message to the sender. The sender, upon receiving PRUNE, also moves us from its eager set to its lazy set. This progressively optimizes the spanning tree to eliminate unnecessary full-message transmissions.

## Specification

### On Eager Receive (Duplicate Path)

```rust
impl PlumtreeState {
    /// Handle receipt of a message via eager push.
    ///
    /// If the message is a duplicate (hash already seen), demotes the sender
    /// to lazy and sends PRUNE. If new, processes normally.
    pub async fn on_eager_receive(
        &mut self,
        from: PeerId,
        message: &Message,
        seen_set: &mut SeenSet,
        message_cache: &mut MessageCache,
        peers: &PeerMap,
    ) -> EagerReceiveResult {
        let hash = compute_message_hash(message.msg_type, &message.data);

        if seen_set.contains(&hash) {
            // Duplicate: tree has a redundant eager link
            // Demote sender from eager to lazy
            self.eager_peers.remove(&from);
            self.lazy_peers.insert(from);

            // Tell sender to stop eager-pushing to us
            if let Some(peer) = peers.get(&from) {
                peer.send_raw(Prune.to_message()).await;
            }

            return EagerReceiveResult::Duplicate;
        }

        // New message: process via broadcast pipeline
        // (steps 2-7 of broadcast algorithm)
        seen_set.insert(hash);
        message_cache.insert(hash, message.clone());

        // Cancel any pending lazy timer for this hash
        self.lazy_queue.remove(&hash);

        EagerReceiveResult::New { hash }
    }
}
```

### On Receiving PRUNE

```rust
impl PlumtreeState {
    /// Handle receipt of PRUNE from a peer.
    ///
    /// The peer is telling us to stop eager-pushing to them.
    /// Move them from our eager set to our lazy set.
    pub fn on_prune(&mut self, from: &PeerId) {
        self.eager_peers.remove(from);
        self.lazy_peers.insert(*from);
    }
}
```

### State Transition Diagram

```
Peer receives duplicate from sender:
  +-----------+                          +-----------+
  |  sender   |  -- duplicate msg -->    | receiver  |
  |  (eager)  |                          |           |
  +-----------+                          +-----------+
                                              |
                                         detect duplicate
                                         (hash in seen_set)
                                              |
                                         demote sender:
                                         eager -> lazy
                                              |
  +-----------+                          +-----------+
  |  sender   |  <-- PRUNE ----------   | receiver  |
  |           |                          |           |
  +-----------+                          +-----------+
       |
  on_prune: move receiver
  from eager to lazy
       |
  +-----------+
  |  sender   |
  | (lazy)    |
  +-----------+
```

### Behavior Rules

- Duplicate detection MUST use the seen set (PLT-008), not the message cache.
- On duplicate, the sender MUST be moved from `eager_peers` to `lazy_peers` atomically (remove then insert).
- A PRUNE message MUST be sent to the sender after demotion.
- On receiving PRUNE, the peer that sent us PRUNE MUST be moved from `eager_peers` to `lazy_peers`.
- If a new (non-duplicate) message arrives eagerly, any pending lazy timer for that hash MUST be cancelled.

## Acceptance Criteria

- [ ] Receiving a duplicate message (hash in seen_set) demotes sender from `eager_peers` to `lazy_peers`
- [ ] A PRUNE message is sent to the sender on duplicate detection
- [ ] Sender is removed from `eager_peers` before being inserted into `lazy_peers`
- [ ] Non-duplicate messages are processed normally (inserted into seen_set and cache)
- [ ] Non-duplicate messages cancel pending lazy timers for the same hash
- [ ] `on_prune` moves the PRUNE sender from `eager_peers` to `lazy_peers`
- [ ] After duplicate detection, the disjoint invariant holds (peer not in both sets)
- [ ] If sender is already in `lazy_peers` when duplicate is received, state remains consistent

## Implementation Notes

- Primary file: `src/gossip/plumtree.rs`
- Dependencies: `SeenSet` (PLT-008), `MessageCache` (PLT-007), `compute_message_hash` (PLT-003)
- Edge cases:
  - A peer may send a duplicate even if already in `lazy_peers` (e.g., race condition where PRUNE hasn't arrived yet); in this case, `eager_peers.remove` is a no-op and `lazy_peers.insert` is idempotent
  - The PRUNE message must be defined in the DIG message type enum and serialized for transport
  - If the sender has already disconnected by the time we try to send PRUNE, the send fails silently
  - Multiple duplicates from the same peer in quick succession should be handled idempotently
