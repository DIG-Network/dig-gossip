# PLT-006: Tree Self-Healing on Disconnect

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 8.1](../../../resources/SPEC.md)

## Summary

When an eager peer disconnects, the spanning tree loses an edge and messages may stop flowing through that path. To heal the tree, the protocol checks the `lazy_queue` for peers that have announced hashes we have not yet received. These lazy peers are promoted to eager via GRAFT, restoring the tree. The entire healing process MUST complete within one `lazy_timeout_ms` cycle (500ms by default), ensuring minimal disruption to message propagation.

## Specification

### On Eager Peer Disconnect

```rust
impl PlumtreeState {
    /// Handle disconnection of a peer.
    ///
    /// If the peer was eager, attempt to heal the tree by promoting
    /// lazy peers that have announced pending hashes.
    pub async fn on_peer_disconnect(
        &mut self,
        disconnected: &PeerId,
        peers: &PeerMap,
    ) -> Vec<PeerId> {
        let was_eager = self.eager_peers.remove(disconnected);
        self.lazy_peers.remove(disconnected);

        // Clean up lazy_queue entries referencing the disconnected peer
        for entries in self.lazy_queue.values_mut() {
            entries.retain(|(p, _)| p != disconnected);
        }

        let mut promoted = Vec::new();

        if was_eager {
            // Tree edge lost - attempt to heal by promoting lazy peers
            // that have pending announcements (they have messages we need)
            promoted = self.promote_lazy_peers_with_pending_hashes(peers).await;
        }

        promoted
    }

    /// Promote lazy peers that have announced hashes still in lazy_queue.
    ///
    /// For each pending hash, find a lazy peer that announced it and
    /// promote them to eager via GRAFT.
    async fn promote_lazy_peers_with_pending_hashes(
        &mut self,
        peers: &PeerMap,
    ) -> Vec<PeerId> {
        let mut promoted = Vec::new();
        let mut to_promote: HashSet<PeerId> = HashSet::new();

        // Find lazy peers that have announced pending hashes
        for entries in self.lazy_queue.values() {
            for (peer_id, _) in entries {
                if self.lazy_peers.contains(peer_id) {
                    to_promote.insert(*peer_id);
                }
            }
        }

        // Promote each candidate
        for peer_id in to_promote {
            self.lazy_peers.remove(&peer_id);
            self.eager_peers.insert(peer_id);

            // Send GRAFT to establish eager link
            if let Some(peer) = peers.get(&peer_id) {
                // Send GRAFT for each pending hash this peer announced
                let hashes: Vec<Bytes32> = self.lazy_queue.iter()
                    .filter(|(_, entries)| entries.iter().any(|(p, _)| *p == peer_id))
                    .map(|(hash, _)| *hash)
                    .collect();

                for hash in &hashes {
                    peer.send_raw(Graft { hash: *hash }.to_message()).await;
                    peer.send_raw(
                        RequestByHash { hash: *hash }.to_message()
                    ).await;
                }
            }

            promoted.push(peer_id);
        }

        promoted
    }
}
```

### Healing Timeline

```
T=0       Eager peer disconnects
          |
          |  on_peer_disconnect() fires immediately
          |  - Remove from eager_peers and lazy_peers
          |  - Clean up lazy_queue references
          |  - Check lazy_queue for pending hashes
          |
T=0+      If lazy peers have pending announcements:
          |  - Promote them to eager immediately
          |  - Send GRAFT + RequestByHash
          |
T<=500ms  If no lazy peers with pending announcements yet:
          |  - Next LazyAnnounce arrives from remaining lazy peers
          |  - Normal lazy timeout (PLT-005) fires
          |  - Announcer promoted to eager via GRAFT
          |
T<=500ms  Tree reconverged
```

### Behavior Rules

- On eager peer disconnect, the peer MUST be removed from both `eager_peers` and `lazy_peers`.
- All `lazy_queue` entries referencing the disconnected peer MUST be cleaned up.
- If the disconnected peer was eager, the tree MUST attempt immediate healing.
- Healing MUST promote lazy peers that have announced hashes still in `lazy_queue`.
- GRAFT + `RequestByHash` MUST be sent for each pending hash the promoted peer announced.
- If no lazy peer has pending announcements, healing relies on the normal lazy timeout path (PLT-005).
- The tree MUST reconverge within one `lazy_timeout_ms` cycle.
- If a lazy peer disconnect occurs (not eager), no tree healing is needed.

## Acceptance Criteria

- [ ] Disconnected peer is removed from `eager_peers`
- [ ] Disconnected peer is removed from `lazy_peers`
- [ ] Disconnected peer's entries are cleaned from `lazy_queue`
- [ ] If disconnected peer was eager, lazy peers with pending hash announcements are promoted
- [ ] Promoted peers are moved from `lazy_peers` to `eager_peers`
- [ ] GRAFT is sent to each promoted peer for each relevant pending hash
- [ ] `RequestByHash` is sent alongside each GRAFT
- [ ] If disconnected peer was lazy, no tree healing is triggered
- [ ] Tree reconverges within one `lazy_timeout_ms` cycle
- [ ] Multiple eager peer disconnects in quick succession are handled correctly

## Implementation Notes

- Primary file: `src/gossip/plumtree.rs`
- Dependencies: `PeerMap`, GRAFT and RequestByHash message types, `lazy_queue`
- Edge cases:
  - If no lazy peers exist (all peers were eager), the tree cannot self-heal until a new peer connects
  - If the disconnected peer was the only path for certain messages, those messages may be lost until a lazy peer announces them
  - Multiple simultaneous disconnects should each trigger independent healing attempts
  - A lazy peer promoted during healing may itself disconnect before responding; this cascades to another healing round
  - The "within one lazy_timeout_ms" guarantee assumes at least one lazy peer has the missing messages; if no peer has them, the guarantee does not apply
  - `remove_peer` (PLT-001) and `on_peer_disconnect` may overlap; ensure they are coordinated or deduplicated
