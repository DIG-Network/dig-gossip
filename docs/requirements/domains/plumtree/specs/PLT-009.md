# PLT-009: PlumtreeMessage Wire Types

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 8.1](../../../resources/SPEC.md)

## Summary

The Plumtree protocol requires four wire message types for peer-to-peer communication: `LazyAnnounce` (hash-only announcement to lazy peers), `Prune` (demote sender to lazy), `Graft` (promote sender to eager and request message), and `RequestByHash` (request a full message by its hash). Each of these MUST have a corresponding `DigMessageType` ID for transmission over the DIG extension wire protocol.

## Specification

### PlumtreeMessage Enum

```rust
/// Plumtree protocol wire messages.
/// Each variant maps to a DigMessageType for wire encoding.
#[derive(Debug, Clone, PartialEq, Eq)]
pub enum PlumtreeMessage {
    /// Hash-only announcement sent to lazy peers.
    /// DigMessageType::PlumtreeLazyAnnounce (214).
    LazyAnnounce {
        /// SHA256(msg_type || data) of the announced message.
        hash: Bytes32,
        /// The original message type (ProtocolMessageTypes or DigMessageType discriminant).
        msg_type: u16,
    },
    /// Sent to a peer to demote them from eager to lazy.
    /// DigMessageType::PlumtreePrune (215).
    Prune,
    /// Sent to a peer to promote them from lazy to eager.
    /// Includes the hash of the message that triggered the promotion.
    /// DigMessageType::PlumtreeGraft (216).
    Graft {
        /// Hash of the message the sender wants to receive.
        hash: Bytes32,
    },
    /// Request a full message by its hash (sent alongside GRAFT).
    /// DigMessageType::PlumtreeRequestByHash (217).
    RequestByHash {
        /// Hash of the requested message.
        hash: Bytes32,
    },
}
```

### DigMessageType Mapping

| PlumtreeMessage Variant | DigMessageType | ID |
|------------------------|----------------|-----|
| `LazyAnnounce` | `PlumtreeLazyAnnounce` | 214 |
| `Prune` | `PlumtreePrune` | 215 |
| `Graft` | `PlumtreeGraft` | 216 |
| `RequestByHash` | `PlumtreeRequestByHash` | 217 |

### Wire Encoding

Each `PlumtreeMessage` variant is serialized to bytes and wrapped in a `chia-protocol::Message` with the corresponding `DigMessageType` discriminant as the `msg_type` field.

## Acceptance Criteria

- [ ] `PlumtreeMessage` enum exists with variants `LazyAnnounce`, `Prune`, `Graft`, `RequestByHash`
- [ ] `LazyAnnounce` contains `hash: Bytes32` and `msg_type: u16`
- [ ] `Prune` is a unit variant
- [ ] `Graft` contains `hash: Bytes32`
- [ ] `RequestByHash` contains `hash: Bytes32`
- [ ] `DigMessageType::PlumtreeLazyAnnounce` has discriminant 214
- [ ] `DigMessageType::PlumtreePrune` has discriminant 215
- [ ] `DigMessageType::PlumtreeGraft` has discriminant 216
- [ ] `DigMessageType::PlumtreeRequestByHash` has discriminant 217
- [ ] Each variant can be serialized and deserialized for wire transmission
- [ ] Wire encoding uses the DigMessageType discriminant as the msg_type in chia-protocol::Message

## Implementation Notes

- Primary file: `src/gossip/plumtree.rs`
- Dependencies: `Bytes32` from `chia-protocol`, `DigMessageType` from `types::dig_messages`
- Edge cases:
  - `Prune` is a unit variant with no payload; the message body is empty
  - `Graft` and `RequestByHash` are often sent together (GRAFT to promote, RequestByHash to fetch)
  - Hash computation must be consistent with PLT-003 and PLT-008
