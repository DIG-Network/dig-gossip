# PLT-003: Lazy Push

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 8.1](../../../resources/SPEC.md)

## Summary

Lazy push is the secondary dissemination mechanism in Plumtree. Instead of sending the full message, only a compact `LazyAnnounce` containing the message hash and type is sent to peers in the `lazy_peers` set. This provides a fallback path: if the eager tree fails to deliver a message, lazy peers can detect the gap and request the full message via GRAFT (PLT-005). The hash is computed as SHA256(msg_type || data).

## Specification

### LazyAnnounce Message

```rust
/// Compact announcement sent to lazy peers.
/// Contains only the hash and message type, not the full payload.
pub struct LazyAnnounce {
    /// SHA256(msg_type || data) - uniquely identifies the message.
    pub hash: Bytes32,
    /// The protocol message type, so the receiver knows what kind
    /// of message to expect when it requests the full payload.
    pub msg_type: ProtocolMessageType,
}
```

### Hash Computation

```rust
/// Compute the message hash used for deduplication and lazy announcements.
///
/// hash = SHA256(msg_type_u16_be || message_data)
pub fn compute_message_hash(msg_type: u16, data: &[u8]) -> Bytes32 {
    let mut hasher = Sha256::new();
    hasher.update(&msg_type.to_be_bytes());
    hasher.update(data);
    Bytes32::from(hasher.finalize())
}
```

### Lazy Push Algorithm

```rust
impl PlumtreeState {
    /// Send LazyAnnounce to all lazy peers, excluding the origin.
    ///
    /// Returns the number of peers the announcement was sent to.
    pub async fn lazy_push(
        &self,
        hash: Bytes32,
        msg_type: ProtocolMessageType,
        origin: Option<&PeerId>,
        peers: &PeerMap,
    ) -> usize {
        let announce = LazyAnnounce { hash, msg_type };
        let mut count = 0;
        for peer_id in &self.lazy_peers {
            if origin.map_or(false, |o| o == peer_id) {
                continue;
            }
            if let Some(peer) = peers.get(peer_id) {
                peer.send_raw(announce.to_message()).await;
                count += 1;
            }
        }
        count
    }
}
```

### Integration in Broadcast

```
broadcast(message: Message, origin: Option<PeerId>):
  1. hash = SHA256(message.msg_type || message.data)
  2. if seen_set.contains(hash) -> return 0
  3. seen_set.insert(hash)
  4. Deliver to local inbound channel
  5. eager_push(message, origin, peers)
  6. lazy_push(hash, msg_type, origin, peers)  // <-- THIS REQUIREMENT
  7. If relay connected: relay.broadcast(message, exclude_list)
  8. Return count sent
```

### Behavior Rules

- The origin peer MUST be excluded from lazy push to prevent announcement reflection.
- If `origin` is `None` (locally originated), all lazy peers receive the announcement.
- The `LazyAnnounce` MUST contain the hash and `msg_type` only, never the full message data.
- The hash MUST be computed as `SHA256(msg_type as u16 big-endian || data)`.
- Lazy push MUST NOT block on individual peer sends.

## Acceptance Criteria

- [ ] `lazy_push` sends `LazyAnnounce{hash, msg_type}` to each peer in `lazy_peers`
- [ ] `lazy_push` excludes the `origin` peer from the send list
- [ ] `lazy_push` sends to all lazy peers when `origin` is `None`
- [ ] `LazyAnnounce` contains only `hash` and `msg_type`, not the full message data
- [ ] Hash is computed as `SHA256(msg_type_u16_be || data)`
- [ ] `lazy_push` returns the count of peers the announcement was sent to
- [ ] Peers in `eager_peers` do not receive `LazyAnnounce` via lazy push
- [ ] A single peer send failure does not prevent sending to remaining peers

## Implementation Notes

- Primary file: `src/gossip/plumtree.rs`
- Dependencies: `Sha256` from a crypto library (e.g., `sha2` crate), `Bytes32`, `ProtocolMessageType` from chia-protocol
- Edge cases:
  - If `lazy_peers` is empty (all peers eager), no lazy announcements are sent; this is the expected initial state
  - The `msg_type` field in `LazyAnnounce` allows the receiver to know what kind of message to expect without having the full payload
  - The hash computation must use big-endian byte order for the `msg_type` u16 to ensure cross-platform consistency
  - `LazyAnnounce` itself must be serialized into a `Message` for transport via `send_raw`
