# PLT-001: PlumtreeState Structure

## Summary

The `PlumtreeState` struct holds the core state for the Plumtree gossip protocol. It classifies each connected peer as either eager (receives full messages) or lazy (receives hash-only announcements), and tracks pending lazy announcements awaiting timeout. All newly connected peers default to eager, forming the initial spanning tree. The lazy timeout determines how quickly the tree self-heals when eager paths fail.

## Specification

### Struct Definition

```rust
use std::collections::{HashMap, HashSet};

/// Plumtree peer classification for gossip routing.
///
/// Maintains the eager/lazy peer partition and pending lazy announcement
/// queue that together define the broadcast spanning tree.
pub struct PlumtreeState {
    /// Eager peers: receive full messages immediately (spanning tree neighbors).
    /// Default: all peers start as eager.
    pub eager_peers: HashSet<PeerId>,

    /// Lazy peers: receive hash-only announcements. Pull full message on demand.
    pub lazy_peers: HashSet<PeerId>,

    /// Pending lazy announcements: hash -> vec of (announcer_peer_id, timestamp_ms).
    /// Entries are added on LazyAnnounce receipt and removed when the message
    /// arrives eagerly or the timeout fires.
    pub lazy_queue: HashMap<Bytes32, Vec<(PeerId, u64)>>,

    /// Missing message timer: if a lazily-announced hash isn't received
    /// eagerly within this timeout (ms), pull from the lazy announcer.
    /// Default: 500.
    pub lazy_timeout_ms: u64,
}
```

### Default Construction

```rust
impl Default for PlumtreeState {
    fn default() -> Self {
        Self {
            eager_peers: HashSet::new(),
            lazy_peers: HashSet::new(),
            lazy_queue: HashMap::new(),
            lazy_timeout_ms: 500,
        }
    }
}
```

### Peer Addition

```rust
impl PlumtreeState {
    /// Add a newly connected peer. All peers start as eager.
    pub fn add_peer(&mut self, peer_id: PeerId) {
        self.eager_peers.insert(peer_id);
    }

    /// Remove a disconnected peer from both sets and any lazy_queue entries.
    pub fn remove_peer(&mut self, peer_id: &PeerId) {
        self.eager_peers.remove(peer_id);
        self.lazy_peers.remove(peer_id);
        for entries in self.lazy_queue.values_mut() {
            entries.retain(|(p, _)| p != peer_id);
        }
    }
}
```

### State Machine

```
                    +-----------+
  connect --------->|   EAGER   |
                    +-----------+
                      |       ^
        duplicate     |       |  GRAFT / lazy timeout
        received      |       |  / disconnect heal
        (send PRUNE)  v       |
                    +-----------+
                    |   LAZY    |
                    +-----------+
                      |
        disconnect -->| (removed from both sets)
                      v
                    +-----------+
                    | REMOVED   |
                    +-----------+
```

### Invariants

- A peer MUST NOT appear in both `eager_peers` and `lazy_peers` simultaneously.
- `eager_peers` and `lazy_peers` are disjoint: `eager_peers.intersection(&lazy_peers).count() == 0`.
- All connected peers MUST appear in exactly one of the two sets.
- `lazy_timeout_ms` MUST default to 500.

## Acceptance Criteria

- [ ] `PlumtreeState` struct has field `eager_peers: HashSet<PeerId>`
- [ ] `PlumtreeState` struct has field `lazy_peers: HashSet<PeerId>`
- [ ] `PlumtreeState` struct has field `lazy_queue: HashMap<Bytes32, Vec<(PeerId, u64)>>`
- [ ] `PlumtreeState` struct has field `lazy_timeout_ms: u64`
- [ ] `PlumtreeState::default()` sets `lazy_timeout_ms` to 500
- [ ] `PlumtreeState::default()` initializes all collections as empty
- [ ] `add_peer()` inserts peer into `eager_peers`
- [ ] `add_peer()` does not insert peer into `lazy_peers`
- [ ] `remove_peer()` removes peer from `eager_peers`, `lazy_peers`, and all `lazy_queue` entries
- [ ] No peer ever appears in both `eager_peers` and `lazy_peers` after any state transition

## Implementation Notes

- Primary file: `src/gossip/plumtree.rs`
- Dependencies: `PeerId` type alias from `src/types/peer.rs`, `Bytes32` from chia-protocol
- Edge cases:
  - Adding a peer that is already in `eager_peers` is a no-op (HashSet semantics)
  - Adding a peer that is in `lazy_peers` should move it to `eager_peers` (or callers should use `add_peer` only for new connections)
  - `remove_peer` must clean up `lazy_queue` entries referencing the disconnected peer to prevent stale announcements
  - `lazy_timeout_ms` can be overridden via `GossipConfig` but defaults to 500
