# CON-009: Mandatory Mutual TLS (mTLS) via chia-ssl on All Peer Connections

> **Authoritative requirement:** [CON-009](../NORMATIVE.md#CON-009)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Section 5.3 (Mandatory Mutual TLS)

---

## Summary

All peer-to-peer connections in dig-gossip MUST use **mutual TLS (mTLS)** where **both sides present certificates**. This is not server-only TLS — both the connecting peer and the listening peer authenticate each other by presenting `chia-ssl` certificates during the TLS handshake. Peer identity (`PeerId`) is derived from SHA256 of the **remote** peer's TLS certificate public key, extracted during mTLS negotiation. This makes certificate possession the foundation of the identity system.

This matches Chia's mTLS design where all peer connections use mutual certificate authentication:
- The server requires client certificates: `ssl_context.verify_mode = ssl.CERT_REQUIRED` ([`server.py:67`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/server/server.py#L67))
- The client presents its certificate: via `ssl_context.load_cert_chain()` ([`server.py:69`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/server/server.py#L69))
- Peer node ID is derived from the certificate: `peer_node_id` ([`ws_connection.py:95`](https://github.com/Chia-Network/chia-blockchain/blob/6e7a4954edccd8ab83fcacf938cfc42ddfcad7f2/chia/server/ws_connection.py#L95))

The `chia-ssl` crate provides `ChiaCertificate::generate()` for creating node certificates, and `chia-sdk-client` provides `load_ssl_cert()`, `create_native_tls_connector()`, and `create_rustls_connector()` which configure the TLS connector with the node's certificate as a **client certificate** for mutual auth.

## Specification

### Certificate Lifecycle

```rust
use chia_ssl::ChiaCertificate;
use chia_sdk_client::{load_ssl_cert, create_native_tls_connector};

// On first run: generate a new certificate
let cert = ChiaCertificate::generate()?;
std::fs::write(&config.cert_path, &cert.cert_pem)?;
std::fs::write(&config.key_path, &cert.key_pem)?;

// On subsequent runs: load existing certificate
let cert = load_ssl_cert(&config.cert_path, &config.key_path)?;

// Create TLS connector for outbound connections
let connector = create_native_tls_connector(&cert)?;
// OR for rustls:
// let connector = create_rustls_connector(&cert)?;
```

### Outbound Connection TLS

All outbound connections MUST use the TLS connector:

```rust
// connect_peer() internally uses Peer::connect() which requires a Connector
let (peer, receiver) = connect_peer(
    network_id,
    connector,   // TLS connector from chia-ssl certificate
    socket_addr,
    options,
).await?;
```

### Inbound Connection mTLS

The listener MUST wrap incoming TCP streams with mTLS before WebSocket upgrade. The TLS acceptor MUST be configured to **require client certificates** (`CERT_REQUIRED`):

```rust
use native_tls::TlsAcceptor as NativeTlsAcceptor;
use tokio_native_tls::TlsAcceptor;

// Build mTLS acceptor — requires client certificate presentation
let mut builder = NativeTlsAcceptor::builder(identity);
// Chia equivalent: ssl_context.verify_mode = ssl.CERT_REQUIRED (server.py:67)
// native-tls doesn't have a direct CERT_REQUIRED setter, but the
// chia-ssl CA cert is loaded to verify client certs.
let native_acceptor = builder.build()?;
let tls_acceptor = TlsAcceptor::from(native_acceptor);

// For each incoming connection:
let tcp_stream = listener.accept().await?;
let tls_stream = tls_acceptor.accept(tcp_stream).await?;  // mTLS handshake
// ↑ This fails if the connecting peer does not present a certificate

// Extract remote peer's certificate public key for PeerId derivation
let peer_cert = extract_peer_certificate(&tls_stream);
let peer_id: PeerId = sha256(&peer_cert.public_key_der());

let ws_stream = tokio_tungstenite::accept_async(tls_stream).await?;
let (peer, receiver) = Peer::from_websocket(ws_stream, options)?;
```

### Peer Identity from mTLS

```rust
/// PeerId is derived from SHA256 of the REMOTE peer's TLS certificate
/// public key, extracted during the mTLS handshake.
/// Both sides of the connection derive each other's PeerId this way.
/// This binds peer identity to cryptographic key material — 
/// impersonation requires possessing the private key.
/// Matches Chia's peer_node_id from ws_connection.py:95.
pub type PeerId = Bytes32;  // SHA256(remote_tls_public_key)
```

### Rejection of Non-mTLS Connections

- Outbound: `connect_peer()` requires a `Connector` parameter that includes the node's client certificate — there is no code path for plain WebSocket or server-only TLS.
- Inbound: The listener MUST NOT call `tokio_tungstenite::accept_async()` on a raw TCP stream. The mTLS acceptor MUST be applied first. If the peer does not present a certificate, the TLS handshake fails and the connection MUST be dropped.
- Server-only TLS (where only the listener has a cert but the client does not) MUST NOT be accepted for P2P connections. Both sides MUST present certificates.

## Acceptance Criteria

- [ ] All outbound P2P connections use a TLS connector that includes the node's chia-ssl certificate as a **client cert** (mutual TLS).
- [ ] All inbound P2P connections pass through a TLS acceptor configured to **require client certificates** (mTLS, not server-only TLS).
- [ ] Inbound connections where the peer does NOT present a certificate are rejected (TLS handshake failure).
- [ ] `ChiaCertificate::generate()` is used on first run when no certificate exists.
- [ ] `load_ssl_cert()` is used on subsequent runs to load existing certificate.
- [ ] Unencrypted WebSocket connections (`ws://` without TLS) are never accepted inbound.
- [ ] Unencrypted WebSocket connections are never initiated outbound.
- [ ] Server-only TLS (listener has cert, client does not) is NOT accepted for P2P.
- [ ] PeerId is derived from SHA256 of the **remote** peer's TLS certificate public key, extracted during mTLS handshake.
- [ ] Both sides of a connection derive each other's PeerId from the certificate exchange.
- [ ] TLS handshake failure on inbound connection results in connection drop (not fallback to plain).
- [ ] The `chia-ssl` crate is the only source of certificate management (no custom cert generation).
- [ ] Both `native-tls` and `rustls` feature flags work for mTLS connector/acceptor creation.
- [ ] Relay connections are exempt from mTLS (they use standard `wss://` server-only TLS).

## Implementation Notes

- **Primary files:** `src/connection/listener.rs` (inbound TLS), `src/service/gossip_service.rs` (cert loading)
- **Dependencies:** CON-001 (outbound uses connect_peer with connector), CON-002 (inbound uses TLS acceptor)
- **Edge cases:**
  - Certificate file missing on startup → generate new via `ChiaCertificate::generate()`
  - Certificate file corrupt → error on load, do not fall back to unencrypted
  - TLS handshake timeout → drop connection, log warning
  - Self-signed certificates are expected (Chia model) — `danger_accept_invalid_certs(true)` in native-tls connector
- **Relay connections are separate:** Relay uses its own WebSocket connection (typically `wss://`) and is not subject to the `chia-ssl` certificate requirement. Relay TLS is handled by standard WebSocket TLS, not mutual chia-ssl auth.
