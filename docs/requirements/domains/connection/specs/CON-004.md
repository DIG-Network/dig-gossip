# CON-004: Keepalive via Ping/Pong

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 2.13 (Constants)](../../../resources/SPEC.md)

## Summary

Each peer connection must maintain keepalive by sending `Ping` messages at 30-second intervals (`PING_INTERVAL_SECS`). If no `Pong` response is received within 90 seconds (`PEER_TIMEOUT_SECS`), the connection must be considered dead and disconnected. The Ping/Pong round-trip time is also used for latency-aware peer scoring (see `PeerReputation.avg_rtt_ms`).

## Specification

### Constants

```rust
/// Ping interval in seconds. Default: 30.
pub const PING_INTERVAL_SECS: u64 = 30;

/// Peer timeout in seconds (no Pong received). Default: 90.
pub const PEER_TIMEOUT_SECS: u64 = 90;
```

### Keepalive Loop

A keepalive task MUST be spawned for each connection. It sends `Ping` at the configured interval and tracks the last `Pong` received:

```rust
use tokio::time::{interval, Duration, Instant};

async fn keepalive_loop(
    peer: &Peer,
    peer_connection: &PeerConnection,
) -> Result<(), GossipError> {
    let mut ping_interval = interval(Duration::from_secs(PING_INTERVAL_SECS));
    let mut last_pong = Instant::now();

    loop {
        ping_interval.tick().await;

        // Check if peer has timed out
        if last_pong.elapsed() > Duration::from_secs(PEER_TIMEOUT_SECS) {
            // No Pong received within timeout — disconnect
            tracing::warn!(
                peer_id = %peer_connection.peer_id,
                "Peer timed out (no Pong in {}s), disconnecting",
                PEER_TIMEOUT_SECS
            );
            return Err(GossipError::RequestTimeout);
        }

        // Send Ping
        let ping_sent_at = Instant::now();
        peer.send(Ping::new()).await?;

        // Pong is received asynchronously via the message loop
        // and updates last_pong and RTT tracking
    }
}
```

### Pong Handling

When a `Pong` is received in the per-connection message loop, it MUST update the last-pong timestamp and RTT tracking:

```rust
fn handle_pong(
    peer_connection: &mut PeerConnection,
    ping_sent_at: Instant,
) {
    let rtt_ms = ping_sent_at.elapsed().as_millis() as u64;

    // Update last message time
    peer_connection.last_message_time = current_unix_timestamp();

    // Update RTT history for latency-aware scoring
    let reputation = &mut peer_connection.reputation;
    if reputation.rtt_history.len() >= RTT_WINDOW_SIZE {
        reputation.rtt_history.pop_front();
    }
    reputation.rtt_history.push_back(rtt_ms);

    // Recalculate average RTT
    let sum: u64 = reputation.rtt_history.iter().sum();
    reputation.avg_rtt_ms = Some(sum / reputation.rtt_history.len() as u64);
}
```

### Disconnect on Timeout

When the keepalive loop detects a timeout (no `Pong` within `PEER_TIMEOUT_SECS`), it MUST:

1. Log the timeout event with the peer ID
2. Close the WebSocket connection
3. Remove the peer from the active connections
4. Apply a `ConnectionIssue` penalty to the peer's reputation

## Acceptance Criteria

- [ ] `Ping` messages are sent every `PING_INTERVAL_SECS` (30 seconds)
- [ ] `Pong` responses update the last-pong timestamp
- [ ] Connection is disconnected if no `Pong` is received within `PEER_TIMEOUT_SECS` (90 seconds)
- [ ] RTT is measured from Ping send time to Pong receive time
- [ ] RTT history is maintained in `PeerReputation.rtt_history` (up to `RTT_WINDOW_SIZE` entries)
- [ ] `avg_rtt_ms` is updated on each Pong
- [ ] Keepalive task is spawned per connection (both inbound and outbound)
- [ ] Timeout disconnection logs a warning with the peer ID
- [ ] Timeout disconnection applies a `ConnectionIssue` penalty

## Implementation Notes

- Primary files: `src/connection/mod.rs` (keepalive loop), `src/types/reputation.rs` (RTT tracking)
- Dependencies: `tokio` (interval, Duration, Instant), `chia-protocol` (Ping/Pong messages via Peer)
- Edge cases:
  - Ping/Pong is bidirectional: we must also respond to incoming Pings with Pongs
  - If the WebSocket send fails during Ping, treat as connection failure
  - The 90s timeout allows for up to 3 missed Pong responses (30s interval x 3 = 90s)
  - Peer may send unsolicited Pong (without a Ping); still update last-pong timestamp
  - RTT calculation should handle the case where `rtt_history` is empty (division by zero)
