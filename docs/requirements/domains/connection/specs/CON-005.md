# CON-005: Per-Connection Rate Limiting

> **Authoritative requirement:** [NORMATIVE.md](../NORMATIVE.md)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md - Section 5.3](../../../resources/SPEC.md)

## Summary

Rate limiting uses `chia-sdk-client::RateLimiter` with `V2_RATE_LIMITS`. Outbound rate limiting is built into `Peer::send_raw()` (it loops with 1-second sleep until the rate limit clears). Each inbound connection must have its own separate `RateLimiter` instance to enforce per-connection message frequency and size limits. DIG extension message types (200+ range) must be added to the rate limit configuration.

## Specification

### Outbound Rate Limiting (Built-in)

Outbound rate limiting is handled internally by `chia-sdk-client::Peer::send_raw()`. No additional implementation is needed for outbound:

```rust
// Peer::send_raw() internally enforces rate limits.
// It loops with a 1-second sleep until the rate limit window clears.
// This is provided by chia-sdk-client — we do NOT reimplement.
```

### Inbound Rate Limiting (Per-Connection)

Each inbound connection MUST have its own `RateLimiter` instance:

```rust
use chia_sdk_client::{RateLimiter, V2_RATE_LIMITS};

// Create a separate RateLimiter for each inbound connection
let inbound_limiter = RateLimiter::new(
    true,   // incoming = true (this is an inbound connection)
    60,     // reset_seconds: rate limit window
    config.peer_options.rate_limit_factor,  // scaling factor
    V2_RATE_LIMITS.clone(),
);
```

### DIG Extension Rate Limits

DIG-specific message types (200+ range) MUST be added to the rate limits configuration. The base `V2_RATE_LIMITS` covers standard Chia protocol messages; DIG extensions need additional entries:

```rust
use chia_sdk_client::{RateLimits, RateLimit};

/// Extend V2_RATE_LIMITS with DIG-specific message type rate limits.
fn create_dig_rate_limits() -> RateLimits {
    let mut limits = V2_RATE_LIMITS.clone();

    // DIG extension message types (DigMessageType enum, 200+ range)
    // NewAttestation = 200
    limits.insert(200, RateLimit {
        frequency: 100,     // max per minute
        max_size: 4096,     // max message size in bytes
    });
    // NewCheckpointProposal = 201
    limits.insert(201, RateLimit {
        frequency: 10,
        max_size: 8192,
    });
    // NewCheckpointSignature = 202
    limits.insert(202, RateLimit {
        frequency: 100,
        max_size: 4096,
    });
    // RequestCheckpointSignatures = 203
    limits.insert(203, RateLimit {
        frequency: 10,
        max_size: 1024,
    });
    // RespondCheckpointSignatures = 204
    limits.insert(204, RateLimit {
        frequency: 10,
        max_size: 65536,
    });
    // RequestStatus = 205
    limits.insert(205, RateLimit {
        frequency: 10,
        max_size: 1024,
    });
    // RespondStatus = 206
    limits.insert(206, RateLimit {
        frequency: 10,
        max_size: 8192,
    });
    // NewCheckpointSubmission = 207
    limits.insert(207, RateLimit {
        frequency: 10,
        max_size: 65536,
    });
    // ValidatorAnnounce = 208
    limits.insert(208, RateLimit {
        frequency: 10,
        max_size: 4096,
    });

    limits
}
```

### Inbound Message Checking

Every inbound message MUST be checked against the per-connection rate limiter before processing:

```rust
async fn handle_inbound_message(
    message: &Message,
    limiter: &mut RateLimiter,
    peer_connection: &mut PeerConnection,
) -> Result<(), GossipError> {
    // Check rate limit for this message type
    if !limiter.check(message.msg_type as u8, message.data.len()) {
        // Rate limit exceeded — apply penalty
        peer_connection.reputation.penalty_points += PenaltyReason::RateLimitExceeded.points();
        return Err(GossipError::ClientError(
            ClientError::RateLimitExceeded
        ));
    }

    // Process message...
    Ok(())
}
```

## Acceptance Criteria

- [ ] Each inbound connection has its own `RateLimiter` instance
- [ ] `RateLimiter` is initialized with `incoming = true` for inbound connections
- [ ] `V2_RATE_LIMITS` is used as the base rate limit configuration
- [ ] DIG extension message types (200-208) are added with appropriate frequency and size limits
- [ ] Every inbound message is checked against the rate limiter before processing
- [ ] Rate limit violations apply a `RateLimitExceeded` penalty to the peer's reputation
- [ ] Outbound rate limiting is delegated to `Peer::send_raw()` (no custom implementation)
- [ ] Rate limiter uses the configured `rate_limit_factor` from `PeerOptions`
- [ ] Rate limit window is 60 seconds (reset_seconds)

## Implementation Notes

- Primary files: `src/connection/mod.rs` (rate limiter creation and checking)
- Dependencies: `chia-sdk-client` (RateLimiter, RateLimits, RateLimit, V2_RATE_LIMITS)
- Edge cases:
  - `rate_limit_factor` of 0 effectively disables rate limiting (useful for testing)
  - DIG message types must not collide with existing Chia `ProtocolMessageTypes` (200+ range is safe)
  - Rate limiter state is per-connection and not shared between connections
  - If a peer repeatedly exceeds rate limits, the accumulated `RateLimitExceeded` penalties will trigger a ban (see CON-007)
  - The `RateLimiter::check()` method returns false if the limit is exceeded; it does not block
